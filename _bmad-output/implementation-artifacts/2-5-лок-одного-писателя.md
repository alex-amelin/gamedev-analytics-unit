# Story 2.5: Лок одного писателя

Status: done

## Story

As a оператор юнита,
I want эксклюзивный лок писателя на уровне хранилища,
so that второй конкурентный писатель падал сразу, а не портил БД.

**Контекст эпика.** Пятая история Epic 2 «Приём данных и безопасное обновление хранилища». Фундамент стоит: 2.1 (`done`) — `paths.py` (в т.ч. **`get_writer_lock_path()` → `{root}/.writer.lock`**, уже готов) + `database_manager.py` (контекст-менеджер соединения write/read-only с `finally`-close); 2.2 (`done`) — атомарная запись Parquet-партиции; 2.3 (`done`) — жёсткая сверка строк; 2.4 (`ready-for-dev`) — мета-таблица `load_state` + реконсиляция. Теперь 2.5 кладёт **последний страж целостности** перед оркестратором: модуль `scripts/utils/writer_lock.py` — **дисциплину одного писателя**. Покрывает **FR-15** (process-level эксклюзивный `.writer.lock`; второй конкурентный писатель → fail-fast без ожидания/записи; чтение лока не требует); опорная часть **NFR-1** «не сломать базу». В цепочке протокола идемпотентного дня лок **оборачивает весь шаг записи**: p81 (2.7) берёт `.writer.lock` → `write_partition (2.2)` → сверка (2.3) → `load_state (2.4)` → освобождает лок. Лок — НЕ часть per-day цикла, а **внешняя рамка** вокруг всей записи обновления.

**Почему лок вообще нужен (корень требования).** DuckDB — **single-writer движок** (architecture.md:78): два процесса, открывших `gdau.duckdb` на запись, портят БД. Плюс наш сырьевой слой пишется через temp→`os.replace` (2.2): два писателя, наперегонки заменяющие одну партицию, дают неопределённый результат. Process-level лок — единственная дешёвая гарантия «один писатель в каждый момент» без серверных процессов (принцип переносимости: ноль демонов, только файлы + встроенный DuckDB).

**Кто это потребляет (проектируй API под них).**
- **Оркестратор p81 (2.7)** — единственный прямой потребитель: оборачивает **весь** прогон записи (`with writer_lock(): … вся запись диапазона …`). AC #2 эпика 2.7 (строка 334): «любая запись p81 — под `.writer.lock` (2.5)». Лок берётся **один раз** на прогон обновления (storage-level, не per-day): «один писатель на хранилище», не «один писатель на день».
- **`gdau-logs update` (2.9)** — высокоуровневая команда, заходит в запись через p81 → лок берётся внутри p81, не в CLI.
- **init `gdau-init` (4.3)** — создаёт DuckDB + view'ы при разворачивании; если делает это под тем же примитивом — тоже через `with writer_lock()` (init = тоже писатель). Спроектируй примитив переиспользуемым (путь инъектируется, дефолт — `get_writer_lock_path()`).
- **MCP-чтение (3.1)** и **view'ы (2.6)** — **НЕ** потребители: читатели лок НЕ берут (AC #3; architecture.md:515 «чтение — read-only, без лока»). Это граница, а не код: read-канал просто не зовёт `writer_lock`.

Проектируй как **контекст-менеджер** (`with writer_lock(): …`): захват на входе, **гарантированное освобождение в `finally`** на выходе при любом исходе — успех, исключение, прерывание (AC #4; epics.md:368 «`.writer.lock` освобождается в `finally`»). Путь лок-файла — **инъектируется** параметром (дефолт `get_writer_lock_path()`), чтобы тесты гоняли на `tmp_path` без сети и без рабочей БД (приём шва 2.1).

**Это НЕ вендоринг — новый модуль.** `writer_lock.py` помечен в дереве архитектуры как **`[новое]`** (строка 453). У directaiq дисциплина одного писателя решалась иначе (disk-guard/queue/`BaseScript`) — вся эта инфра **осознанно вырезана** ([[directaiq-reference]], NFR-6, project-context «не тащить инфру directaiq: queue/worker/disk-guard/cron»). Прямого аналога нет — пиши с нуля по контракту ниже. Дерево тестов архитектуры называет `test_writer_lock.py` (строка 485) — `writer_lock.py` его естественное зеркало.

### Главные риски / решения (читать до кода)

> ✅ **РЕШЕНИЕ ШЕФА (2026-05-24): вариант A — OS advisory-lock.** Эпик AC #5 буквально диктовал «stale **по живости PID**», но при разборе этот механизм оказался хрупким (Windows-footgun + переиспользование PID + TOCTOU, см. риски №2/№3). Шеф утвердил **вариант A (OS-level advisory-lock `fcntl`/`msvcrt`)**: тот же исход (умерший писатель не блокирует навсегда) достигается **надёжнее и проще**, без проверки PID. **Реализуй вариант A.** Вариант B ниже оставлен как разобранная отклонённая альтернатива (трассируемость решения). Контракт (контекст-менеджер, типы исключений, тесты, границы) от механизма не зависит.

1. **Механизм: OS advisory-lock (вариант A — УТВЕРЖДЁН Шефом) vs PID-файл + живость (вариант B — отклонён).**

   **Вариант A — OS-level advisory-lock (РЕАЛИЗОВАТЬ ЭТОТ).** Открыть лок-файл, взять на нём **неблокирующий эксклюзивный лок ядра**: POSIX — `fcntl.flock(fd, LOCK_EX | LOCK_NB)`; Windows — `msvcrt.locking(fd, LK_NBLCK, 1)`. Лок занят живым писателем → системный вызов сразу бросает `OSError`/`BlockingIOError` → **fail-fast** (AC #2, без ожидания). Ядро **само освобождает лок при смерти процесса** (крэш/`kill -9`/SIGKILL — даже без `finally`) → «залипший лок от умершего писателя» **физически невозможен**: новый писатель просто берёт лок успешно (AC #5 по исходу — «не блокирует навсегда» гарантируется ядром надёжнее любой проверки PID). Освобождение на штатном выходе — закрытие `fd` в `finally` (AC #4). **Никакой проверки живости PID, никакого чтения лок-файла, никакого ручного reclaim, никаких TOCTOU-гонок.** Это [[simplicity-first]] и максимальная устойчивость к edge-cases одновременно.

   **Вариант B — PID-файл + проверка живости (буква AC #5).** Лок-файл хранит PID; атомарное создание `os.open(O_CREAT | O_EXCL)`; занят → прочитать PID, проверить живость; жив → fail-fast; мёртв → распознать stale, удалить, переотобрать. Даёт PID в диагностике, но: **(а)** см. риск №2 — `os.kill(pid, 0)` на Windows **убивает процесс**; безопасная проверка требует `ctypes`-обвязки `OpenProcess`; **(б)** см. риск №3 — переиспользование PID даёт ложное «жив» → stale-лок не переотбирается никогда (нарушает AC #5); надёжный фикс требует сверки времени старта процесса (платформенно, тяжело); **(в)** ручной reclaim тащит TOCTOU между «прочитал PID → проверил → удалил → создал». Сложнее и **менее** надёжно, чем A.

   **Вывод:** A достигает исхода всех 5 AC проще и надёжнее; B буквально называет «PID», но платит сложностью и новыми дырами. Реализуй A до иного решения Шефа.

2. **Windows-footgun: `os.kill(pid, 0)` НЕ проверяет живость — он УБИВАЕТ процесс (критично, если выбран вариант B).** На POSIX `os.kill(pid, 0)` — идиома проверки живости (сигнал 0 не доставляется, только проверяется существование). **На Windows Python реализует `os.kill` для любого сигнала кроме `CTRL_C_EVENT`/`CTRL_BREAK_EVENT` через `TerminateProcess(handle, sig)`** — т.е. `os.kill(pid, 0)` **принудительно завершит** процесс с PID `pid` (exit-код 0). Это тихо убьёт чужой процесс. **Никогда не использовать `os.kill` для проверки живости кросс-платформенно.** Безопасная проверка на Windows — только через `ctypes`: `OpenProcess(PROCESS_QUERY_LIMITED_INFORMATION=0x1000, …)` + `GetExitCodeProcess` (`STILL_ACTIVE=259`). В варианте A этот код не нужен вовсе (ядро решает) — ещё один довод за A.

3. **Переиспользование PID → ложное «жив» (критично для варианта B).** ОС переиспользует PID-ы. Если писатель крэшнул и его PID занят посторонним процессом, наивная проверка живости вернёт «жив» → лок-файл считается удержанным навсегда → новый писатель падает fail-fast на **мёртвом** локе (прямое нарушение AC #5 «не блокирует навсегда»). Надёжное различение требует сверки **времени старта** процесса (POSIX — `/proc/<pid>/stat`, только Linux; Windows — `GetProcessTimes` через `ctypes`) — платформенно и тяжело против [[simplicity-first]]. В варианте A проблемы нет: ядро привязывает лок к **живому открытому дескриптору**, а не к числу PID — переиспользование невозможно по построению.

4. **Лок освобождается в `finally` при ЛЮБОМ исходе (AC #4, критично).** Контекст-менеджер обязан отпускать лок на: (а) штатном выходе, (б) исключении в теле, (в) прерывании (`KeyboardInterrupt`/SIGINT/SIGTERM — epics.md:368). `@contextlib.contextmanager` с `try: yield … finally: _release()` это даёт. **Дополнительная страховка варианта A:** на жёстком `kill -9`/SIGKILL `finally` не исполняется, но ядро всё равно освободит advisory-lock при смерти процесса — двойная гарантия. (В варианте B жёсткий kill оставит залипший PID-файл → нужен ручной stale-reclaim — снова довод за A.)

5. **Захват — НЕблокирующий (fail-fast, не ожидание; AC #2).** Второй писатель **немедленно** падает с понятной ошибкой, **НЕ** ждёт освобождения и **НЕ** пишет. Никаких `LOCK_EX` без `LOCK_NB`, никаких таймаут-циклов, никакого `time.sleep`-ретрая. Один оператор-агент: конкурентный писатель = аномалия (двойной запуск), его надо ловить громко и сразу, а не сериализовать молча. Сообщение fail-loud: что хранилище занято другим писателем + путь лок-файла + (если просто) совет «дождись завершения / проверь, не висит ли процесс».

6. **Лок-файл живёт ТОЛЬКО под корнем хранилища, НЕ в dev-репо (граница, инвариант).** Путь — строго `get_writer_lock_path()` (2.1) = `{GDAU_DATA_ROOT}/.writer.lock`. Резолвер fail-loud-ит при незаданном/несуществующем корне **до** любого `os.open` (наследуется из `get_storage_root`) — мусорный лок-файл в dev-репо не создаётся. **Никакого `mkdir`**: корень уже существует и провалидирован (его создаёт init 4.3); файл-лок лежит прямо под корнем, родителя досоздавать не нужно. `.writer.lock` уже в `.gitignore` шаблона хранилища (architecture.md:469) — не коммитится.

7. **`writer_lock` — независимый примитив: НЕ открывает `gdau.duckdb`, НЕ пишет данные, НЕ зависит от 2.2/2.4 (граница, тестируемость).** Модуль знает только про файл-лок: зависит **только** от `paths.get_writer_lock_path` (+ stdlib). **НЕ** импортирует `database_manager`/`parquet_store`/`load_state`/`duckdb` — он оркеструется снаружи (p81 берёт лок, *затем* открывает БД и пишет). Так примитив чисто тестируется на `tmp_path` без сети, без DuckDB, без партиций. Сериализацию записи в DuckDB и атомарность Parquet обеспечивают 2.1/2.2; 2.5 лишь гарантирует «писатель ровно один».

8. **Кросс-платформенный импорт под `sys.platform`, НЕ `os.name` (mypy strict, критично для зелёного CI).** `fcntl` есть только на POSIX, `msvcrt` — только на Windows; CI гоняет **обе** ОС с `mypy --strict`. mypy специально сужает по `sys.platform` (и `sys.version_info`), но **НЕ** по `os.name`. Поэтому условный импорт строй на `if sys.platform == "win32": import msvcrt` / `else: import fcntl`, и обращения к платформенному модулю держи **под тем же `sys.platform`-гардом** — тогда mypy на Linux считает win32-ветку недостижимой и не ругается на `msvcrt`, а на Windows — наоборот. С `os.name` mypy сужение не делает → strict упадёт на «module not found» на чужой ОС.

## Acceptance Criteria

1. **Given** операция записи, **When** она стартует, **Then** берётся эксклюзивный `.writer.lock` на уровне хранилища.
2. **Given** лок удержан живым писателем, **When** стартует второй писатель, **Then** немедленный fail-fast с понятной ошибкой — без ожидания/записи.
3. **Given** чтение через MCP, **When** оно выполняется, **Then** лок не берётся и чтение им не блокируется.
4. **Given** запись завершилась (успех или fail), **When** операция закончилась, **Then** лок освобождается.
5. **Given** залипший лок от умершего писателя (крэш/kill), **When** новый писатель проверяет лок, **Then** он распознаётся как stale (по живости PID) и переотбирается — не блокирует запись навсегда. _[edge-case: stale lock]_

> **Примечание к AC #5 (механизм — РЕШЕНО).** Шеф утвердил вариант A (2026-05-24). Исход AC #5 («залипший лок умершего писателя не блокирует навсегда; новый писатель переотбирает») обеспечивается **автоосвобождением advisory-lock ядром при смерти процесса** — надёжнее буквальной проверки живости PID (см. риски №1–№3). AC #1–#4 от механизма не зависят.

### Рекомендуемый контракт `writer_lock` (вариант A; финализируй под p81 2.7)

| Имя | Сигнатура | Смысл |
|---|---|---|
| `WriterLockError` | `class(RuntimeError)` | база: инцидент окружения/целостности лока (обёртка сырого `OSError`) |
| `WriterLockHeldError` | `class(WriterLockError)` | лок удержан живым писателем → **fail-fast** (AC #2); точечный `except` для p81 |
| `writer_lock` | `(*, lock_path: Path \| None = None) -> Iterator[None]` (контекст-менеджер) | захват на входе (неблокирующий), освобождение в `finally`; `None` → `get_writer_lock_path()` |

**Использование (p81, 2.7):**
```python
with writer_lock():                  # storage-level, один раз на прогон обновления
    for date in dates_to_load:       # вся запись диапазона — под одним локом
        write_partition(...)         # 2.2
        verify_row_count(...)        # 2.3
        mark_loaded(...)             # 2.4
# лок отпущен в finally при любом исходе (AC #4)
```

### Механизм захвата/освобождения (вариант A — образец, риски №1/№4/№8)

```python
@contextlib.contextmanager
def writer_lock(*, lock_path: Path | None = None) -> Iterator[None]:
    path = lock_path if lock_path is not None else get_writer_lock_path()  # fail-loud при битом корне ДО os.open
    fd = os.open(str(path), os.O_RDWR | os.O_CREAT, 0o644)
    try:
        _acquire_nonblocking(fd, path)   # flock/msvcrt LOCK_NB → занят: WriterLockHeldError
    except BaseException:
        os.close(fd)                     # не утечь дескриптором, если захват не удался
        raise
    try:
        yield
    finally:
        _release(fd)                     # unlock + close под suppress(OSError) — AC #4

def _acquire_nonblocking(fd: int, path: Path) -> None:
    if sys.platform == "win32":          # mypy сужает по sys.platform (НЕ os.name) — риск №8
        try:
            msvcrt.locking(fd, msvcrt.LK_NBLCK, 1)
        except OSError as exc:
            raise WriterLockHeldError(f"Хранилище занято другим писателем (лок {path})") from exc
    else:
        try:
            fcntl.flock(fd, fcntl.LOCK_EX | fcntl.LOCK_NB)
        except OSError as exc:
            raise WriterLockHeldError(f"Хранилище занято другим писателем (лок {path})") from exc
```

- Неблокирующий лок (`LOCK_NB`/`LK_NBLCK`) → fail-fast (AC #2), без ожидания.
- Ядро освобождает лок при смерти процесса → stale умершего писателя невозможен (AC #5 по исходу).
- `_release` симметричен: POSIX `flock(fd, LOCK_UN)` + `os.close(fd)`; Windows `os.lseek(fd,0,0)` + `msvcrt.locking(fd, LK_UNLCK, 1)` + `os.close(fd)`; всё под `contextlib.suppress(OSError)`.

### Latest Tech Information

- **`fcntl.flock(fd, LOCK_EX | LOCK_NB)` (POSIX):** эксклюзивный advisory-лок; `LOCK_NB` делает захват неблокирующим — занят → `BlockingIOError`/`OSError` (`EWOULDBLOCK`/`EACCES`). Лок привязан к **открытому описанию файла** (open file description): два отдельных `os.open` одного файла дают конфликтующие описания **даже внутри одного процесса** (отсюда in-process тестируемость AC #2). Освобождается `LOCK_UN` или автоматически при закрытии `fd`/смерти процесса (база AC #5). Локальная ФС поддерживает (хранилище локально; NFS не наш случай).
- **`msvcrt.locking(fd, mode, nbytes)` (Windows):** `LK_NBLCK` — неблокирующий эксклюзивный лок `nbytes` байт от текущей позиции → занят: `OSError` (`ERROR_LOCK_VIOLATION`). `LK_UNLCK` снимает (с той же позиции/длины). Лок per-handle, **освобождается при завершении процесса** ОС (двойная гарантия AC #5). Лочим 1 байт от позиции 0 (после `os.open` позиция = 0); перед `UNLCK` вернуть позицию в 0.
- **⚠️ `os.kill(pid, 0)` НЕ кросс-платформенен для проверки живости:** на POSIX — проверка существования, **на Windows** Python шлёт это через `TerminateProcess(handle, 0)` → **убивает** процесс с exit 0. Кросс-платформенно живость проверять **только** через `ctypes` `OpenProcess(PROCESS_QUERY_LIMITED_INFORMATION)` + `GetExitCodeProcess` (`STILL_ACTIVE=259`). В рекомендованном варианте A проверка живости **не нужна вовсе** (ядро ведёт лок) — пункт здесь как обоснование выбора A и страховка, если Шеф выберет B.
- **mypy + платформенные модули:** mypy сужает по `sys.platform`/`sys.version_info`, **не** по `os.name`. Условный импорт `if sys.platform == "win32": import msvcrt else: import fcntl` + обращения под тем же гардом → strict зелёный на обеих ОС (mypy считает чужую ветку недостижимой). С `os.name` сужения нет → strict упадёт «module not found». (риск №8)
- **Web-ресёрч не требуется:** stdlib стабилен и зафиксирован локом; внешнего сетевого контракта в истории нет (live-smoke неприменим, как 2.1/2.2/2.3/2.4).
