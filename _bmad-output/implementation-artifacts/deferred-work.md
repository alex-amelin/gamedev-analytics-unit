# Deferred Work

## Deferred from: code review of 1-1-uv-каркас-раскладка-проекта-и-ci (2026-05-23)

- **CI исполняет только editable-путь, не сборку wheel hatchling.** Шаги CI (`uv sync --frozen` + mypy + pytest) ставят пакет editable через uv-механизм (repo-root на `sys.path`), поэтому `import scripts.utils` зелёный из-за корня репо на пути, а не из-за hatchling `packages=["scripts"]`. Сборка wheel (hatchling) и установка wheel в CI не исполняются → регресс hatchling-конфига (например, поломка упаковки цифрового подкаталога `8x_metrica_logs_api`) или namespace-загрязнение (в editable импортируемы все топ-левел каталоги: `templates`, `tests`, `_bmad`…) CI не поймает. Wheel-сборка проверена вручную при ревью (ok). Рекомендация: добавить в CI smoke-шаг `uv build` (+ опц. установку wheel в чистый venv и `python -c "import scripts.utils"`). Источник: Edge Case Hunter.
- **`scripts.8x_metrica_logs_api` импортируем через `importlib.import_module`.** Цифровой префикс ломает `import scripts.8x_metrica_logs_api` (SyntaxError, как и задумано), НО `importlib.import_module("scripts.8x_metrica_logs_api")` отрабатывает и возвращает пустой namespace-модуль (в каталоге только `.gitkeep`). Будущий код, обратившийся к нему через `import_module`, молча получит пустой модуль вместо ошибки. Относится к механизму загрузки оркестратора p81 — история 2.7 (там планируется `spec_from_file_location`). Источник: Edge Case Hunter.
- **`tests/` без опорной pytest-конфигурации.** Нет `tests/__init__.py`, нет `conftest.py`, нет `[tool.pytest.ini_options]`. Сейчас `2 passed` зелёные, потому что editable-`.pth` кладёт корень репо на путь. Запуск pytest из подкаталога или будущий дублирующийся basename теста без `__init__.py` (default import-mode) могут дать коллизию имён модулей. Рекомендация: зафиксировать `[tool.pytest.ini_options] testpaths = ["tests"]` (и при росте набора — import-mode/`__init__.py`). Источник: Edge Case Hunter. _(NB: 1.3 завела `[tool.pytest.ini_options]` с `markers`+`addopts`, но `testpaths` так и не добавлен.)_

## Deferred from: code review of 1-3-вендоринг-metricaclient (2026-05-24)

- **`_check_response_errors` — «голый» трейсбек на нештатной форме `errors`.** `scripts/utils/metrica_client.py:97-99`. Списочное включение `[err.get("text", ...) for err in data["errors"]]` падает AttributeError/TypeError, если `errors` не список или содержит не-dict элементы (строки) → наружу уходит «голый» трейсбек вместо чистого RuntimeError (против духа AC #5). Наследие вендора; реальная вероятность низкая. Источник: Edge Case Hunter (подтверждено: тестом покрыта только well-formed list-of-dicts).
- **POST/download-путь не ловит не-JSON тело.** `scripts/utils/metrica_client.py:262, 336`. `create_log_request`/`clean_log_request` ловят только `RequestException`; при 2xx с не-JSON телом `response.json()` бросит `ValueError`/JSONDecodeError мимо обработки → непойманный трейсбек. GET-путь (`make_request`) этот случай ловит — асимметрия. Плюс POST-путь ошибок (`_handle_request_error`) вообще не покрыт offline-тестом. Наследие вендора. Источник: Edge Case Hunter + Blind Hunter.
- **`get_log_requests` может вернуть `None` вместо `list`.** `scripts/utils/metrica_client.py:277`. `response.get("requests", response.get("allow_log_requests", []))` — если ключ `requests` присутствует со значением `null`, вернётся `None` (нарушение аннотации `list[dict]`), и итерация ниже по стеку упадёт. Двойной fallback по именам ключей также маскирует отсутствие обоих (тихий `[]`). Наследие вендора. Источник: Edge Case Hunter + Blind Hunter.
- **`get_counters` падает на `site2=null` и молча отдаёт `[]`.** `scripts/utils/metrica_client.py:204-217`. `counter.get("site2", {}).get("site","")` ломается (`None.get`), если `site2` явно `null`; при переименовании ключей API метод тихо вернёт `[]` с INFO-логом «Found 0», скрыв дрейф контракта. Info-метод вне критического Logs-пути; live-тест дёргает только `get_counter_info`, эту форму не валидирует. Наследие вендора. Источник: Blind Hunter + Edge Case Hunter.
- **Расход дневной квоты при ретраях не покрыт тестом.** `tests/test_metrica_client.py:274`. Retry-тесты мокают `_rate_limit`, поэтому связка «ретрай ↔ `requests_count_today`» не проверяется. В проде каждый ретрай инкрементит суточный счётчик; шторм ретраев способен упереться в `MAX_REQUESTS_PER_DAY` «посреди» цикла и подменить транзиентную 503 на confusing quota-ошибку. Усиление тестов. Источник: Blind Hunter.
- **`test_token_not_stored_in_attributes` тавтологичен по `repr`.** `tests/test_metrica_client.py:102-107`. Assert `"tok..." not in repr(c.__dict__)` структурно всегда зелёный для поля `session` (`Session.__repr__` не дампит заголовки) → реально проверяется лишь `not hasattr(c,"token")`. NFR-5 недо-покрыт: утечка токена через лог или кастомный `__repr__` этим тестом не ловится. Усиление тестов. Источник: Blind Hunter.

## Deferred from: code review of 1-5-загрузчик-каталога-схемы-и-маппинг-типов (2026-05-24)

- **`UInt8→BOOLEAN` потенциально теряет небулевы UInt8.** `scripts/utils/catalog.py:60`. UInt8 в ClickHouse — целое 0–255, не только флаг 0/1; маппинг в BOOLEAN усечёт небулевы значения при `TRY_CAST` во view. Однако маппинг прописан в project-context (`UInt8 (флаг 0/1) → BOOLEAN`), на текущий каталог не влияет (типы уже сидированы явно; `duckdb_type_for` используется только для будущего сидинга). Пересмотреть на уровне архитектуры, если появятся небулевы UInt8-поля. Источник: Blind Hunter.
- **Рассинхрон маппинга и валидатора массивов.** `scripts/utils/catalog.py:96,116`. `duckdb_type_for("Array(Array(T)))")` рекурсивно даёт `T[][]`, но `_is_valid_duckdb_type` валидирует только один уровень `[]` → функции расходятся в решётке типов. Латентно: каталог вложенных массивов не использует; патологический/whitespace-ввод (`Array( UInt64 )`, `Array()`) всё равно fail-loud (ValueError), просто сообщение менее точное. Нет тестов на эти формы. Источник: Blind Hunter + Edge Case Hunter.
- **`open()` не оборачивает OSError/PermissionError в ValueError.** `scripts/utils/catalog.py:197`. После `is_file()`-проверки `resolved.open(...)` может бросить PermissionError/OSError (права, TOCTOU-гонка) мимо контракта «fail-loud с ValueError + путь». Raw OSError тоже громкий (non-zero), но тип исключения иной. Минорная устойчивость. Источник: Edge Case Hunter.
- **Битый симлинк (AC#8) не покрыт отдельным тестом.** `tests/test_catalog.py:292`. AC#8/DoD заявляют «битый симлинк → fail-loud». Код корректен (`is_file()` == False для dangling-симлинка), и эта же ветка `not is_file()` покрыта кейсом missing-file, но конкретный симлинк-сценарий не тестируется. Кросс-OS создание симлинков на Windows требует Dev Mode (риск флака). Источник: Acceptance Auditor.
