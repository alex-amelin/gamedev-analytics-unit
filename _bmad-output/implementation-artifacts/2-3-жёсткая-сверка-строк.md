# Story 2.3: Жёсткая сверка строк

Status: ready-for-dev

## Story

As a оператор юнита,
I want жёсткую сверку числа строк источник↔хранилище,
so that расхождение немедленно валило загрузку, а не проходило молча.

**Контекст эпика.** Третья история Epic 2 «Приём данных и безопасное обновление хранилища». 2.1 (`done`) положила фундамент (`paths.py` + `database_manager.py`); 2.2 (`review`) дала нижний слой записи — `parquet_store.write_partition(...) -> int`, которая атомарно кладёт день в Parquet и **возвращает число записанных строк**. Теперь 2.3 кладёт **страж целостности**: модуль `scripts/utils/row_check.py`, который сверяет число строк-данных источника (сырой TSV Logs API) с числом строк в записанной партиции и **валит загрузку жёстким исключением при любом расхождении** — не `warning`. Покрывает **FR-13** (сверка строк — жёсткий fail); опорная часть **NFR-1** «не сломать базу». В цепочке протокола идемпотентного дня 2.3 стоит между «запись (2.2)» и «отметка `load_state` (2.4)»: `download parts → собрать день → write_partition (2.2) → **сверка (2.3)** → load_state (2.4)`.

**Кто это потребляет (проектируй API под них).** Единственный прямой потребитель — оркестратор **p81 (2.7)**: он скачивает TSV-части (`MetricaClient.download_log_request_part` → `bytes`), парсит их в строки, зовёт `write_partition(...)` (получает `actual`), затем зовёт `row_check`, чтобы получить `expected` из сырого TSV и сверить с `actual`. Косвенно: запись `load_state` (2.4) и точка «день загружен» (2.7) **гейтятся** успехом этой сверки — день не помечается загруженным, если сверка бросила (AC #2). Проектируй API **чисто и без сети/файлов**: два числа на вход в сверщик, сырой TSV (или его строки) на вход в счётчик строк источника — чтобы тесты не требовали ни сети, ни DuckDB, ни партиций.

**Это НЕ вендоринг — новый модуль.** У directaiq сверка строк была `warning`-ом (мягкой); наш осознанный разворот — **жёсткий fail** (project-context, анти-паттерн «сверка как warning»). Прямого аналога нет, пиши с нуля по контракту ниже. Модуль — выделенный (`row_check.py`), не «довесок» к `parquet_store`/`load_state`: один модуль = одна ответственность (project-context). Дерево тестов архитектуры (`test_row_check.py`, строка 483) подтверждает выделенный модуль; имя модуля `row_check.py` — естественное зеркало.

### Главные риски / решения (читать до кода)

1. **2.3 — чистый примитив, НЕ импортирует `parquet_store` (зависимость от 2.2 = нулевая по коду).** `verify_row_count` берёт два `int` (`expected`, `actual`); `actual` = возврат `write_partition` (число записанных строк — уже зафиксированный контракт 2.2, см. `parquet_store.py:163`). `row_check` **не** импортирует `parquet_store`, **не** читает партицию, **не** открывает `gdau.duckdb`, **не** пишет `load_state`, **не** берёт `.writer.lock`. Поэтому история создаётся и реализуется параллельно ревью 2.2 без блокировки. (Если ревью 2.2 изменит семантику возврата — крайне маловероятно — поправится только проводка `expected↔actual` в p81 2.7, не код 2.3.)

2. **Источник `expected` = сырой TSV (число строк-данных), а НЕ распарсенный список p81 (это суть FR-13, критично).** Смысл сверки — поймать **молчаливую потерю строк** между источником и БД. Поэтому `expected` считается **независимо от парсинга** — прямым подсчётом строк-данных сырого TSV. Сравнение с `actual` (что реально записано через `write_partition`) ловит потерю при парсинге/сборке/записи. Если бы сверяли «сколько p81 распарсил» vs «сколько записал», баг парсинга (выброшенная строка) замаскировался бы — обе величины упали бы одинаково. Считаем источник «от первоисточника».

3. **Заголовок в КАЖДОЙ части → off-by-one превращается в off-by-P (AC #3, критично).** Logs API отдаёт каждую part отдельной выгрузкой со **своим TSV-заголовком** (`download_log_request_part` → `bytes`; формат заголовка — `tests/fixtures/logs_visits_sample.tsv:1`). Число строк-данных = по **каждой непустой части** `(непустых строк − 1 заголовок)`, затем **сумма по частям**. Наивное «всего строк во всех частях − 1» при `P > 1` частях даёт off-by-`(P−1)`. Поэтому счётчик работает **на уровне части** и суммирует. Допущение «заголовок есть в каждой части» — крест AC #3; подтвердить live-smoke оркестратора (2.7), здесь спроектировать защитно (вычитание заголовка на часть).

4. **Жёсткий fail, НЕ warning (AC #1; анти-паттерн project-context).** Расхождение `expected != actual` → **исключение** (`RowCountMismatchError`), которое выходит наружу как non-zero, а не `logger.warning(...)`. Это осознанное отличие от directaiq (architecture.md:375–377). Сообщение fail-loud: источник, дата, `expected`, `actual`.

5. **`0 == 0` для пустого дня = успех; дедуп НЕ делается (AC #4).** Сырьё хранится дословно (FR-6) → дубли визитов в источнике **считаются, а не схлопываются**; сверка — чисто по числу строк. Легитимно пустой день (API честно вернул 0 визитов; см. AC #7 истории 2.2) даёт `expected == 0` и `actual == 0` → `0 == 0` проходит как успех, а не «нет данных»/ошибка.

6. **2.3 НЕ пишет `load_state` — только ГЕЙТ для него (AC #2, границы).** Гарантия «день НЕ помечается загруженным при несошедшейся сверке» обеспечивается тем, что сверка **бросает исключение ДО** шага записи `load_state` в протоколе (`…→ write_partition → сверка(2.3) → load_state(2.4)`). Сам `load_state` и проверка «не помечен» — это история 2.4; интеграционный тест «при mismatch день остался незагруженным» принадлежит 2.4/2.7. В 2.3 тестируем, что сверка **бросает** (а успех — не бросает), и фиксируем порядок протокола, чтобы ревью не счёл AC #2 непокрытым.

## Acceptance Criteria

1. **Given** число строк дня от Logs API и в записанной партиции, **When** они сверяются, **Then** при расхождении — исключение/ненулевой код (НЕ warning).
2. **Given** сверка не сошлась, **When** обрабатывается день, **Then** день НЕ помечается загруженным в `load_state`.
3. **Given** TSV Метрики с заголовочной строкой, **When** считается число строк источника, **Then** заголовок исключается из счёта (нет off-by-one). _[edge-case: header в счёте]_
4. **Given** raw-слой хранит строки дословно (FR-6), **When** в дне есть дубли визитов, **Then** дедуп НЕ делается (сырьё verbatim); сверка — по числу строк, и совпадение `0==0` для пустого дня считается успехом. _[edge-case: политика дублей/пустого дня]_

## Dev Notes

### Рекомендуемый контракт `row_check` (финализируй под p81 2.7)

| Имя | Сигнатура | Смысл |
|---|---|---|
| `RowCountMismatchError` | `class(RuntimeError)` | несошедшаяся сверка — точечный `except` для p81 (AC #2); наружу = non-zero |
| `count_part_rows` | `(part: bytes \| str) -> int` | строки-данные одной TSV-части = `max(0, непустых_строк − 1)` (минус один заголовок, AC #3) |
| `count_source_rows` | `(parts: Iterable[bytes \| str]) -> int` | сумма `count_part_rows` по частям; дедупа нет (AC #4); `0` частей → `0` |
| `verify_row_count` | `(expected: int, actual: int, *, source: str, date: str) -> None` | `expected != actual` → `raise RowCountMismatchError`; `==` (вкл. `0==0`) → INFO + `None` (AC #1, #4) |

**Почему `expected` — от сырого TSV, а `actual` — от `write_partition`.** `actual` = ровно то, что записано в партицию (`write_partition` возвращает `len(rows_list)` — `parquet_store.py:163`). `expected` — независимый подсчёт первоисточника (строки TSV минус заголовки), чтобы сверка ловила потерю на участке источник→запись, а не сравнивала две производные одного парсинга (риск №2). Реконсиляция «мета × факт партиции» через `count()` по факту — отдельная забота 2.4 (источник истины там — факт партиции на старте), не дублируется здесь.

### Механизм счёта строк части (риск №3 — образец)

```
def count_part_rows(part):
    text = part.decode("utf-8") if isinstance(part, bytes) else part
    lines = [ln for ln in text.splitlines() if ln != ""]   # хвостовой \n не инфлейтит
    return max(0, len(lines) - 1)                            # минус ОДИН заголовок части
```

- Заголовок Logs API повторяется в **каждой** части (`download_log_request_part` → `bytes` на часть) → вычитание на часть, сумма по частям (off-by-P-гард).
- `splitlines()` устойчив к `\n`/`\r\n`; фильтр пустых убирает хвостовую пустую строку и случайные пустые разделители.
- Чистый stdlib: ни DuckDB, ни pandas, ни сети — единицы–десятки МБ TSV считаются построчно дёшево.

### Протокол идемпотентного дня (где 2.3 в цепочке; источник: architecture.md 379–381, 536–538)

`download parts → собрать день (p81) → write_partition (2.2) → [2.3] сверка expected↔actual → load_state (2.4) → clean`.
2.3 владеет **только** сверкой: считает `expected` из сырого TSV, берёт `actual` из возврата `write_partition`, и **бросает при расхождении ДО** записи `load_state`. Точка «день загружен» — запись `load_state` после rename+сверки (2.4/2.7), не выход `verify_row_count`. Сверка строк → исключение/non-zero, **никогда** `warning` (architecture.md:375–377; project-context анти-паттерн).

### Latest Tech Information

- **`str.splitlines()` vs `split("\n")`:** `splitlines()` корректно режет `\n`, `\r\n`, `\r` и НЕ создаёт хвостовой пустой элемент при финальном переводе строки → надёжнее для счёта строк (`"a\nb\n".splitlines()` → `["a","b"]`). Дополнительный фильтр `ln != ""` убирает случайные пустые строки между записями. Stdlib, без зависимостей.
- **TSV частей Logs API:** `download_log_request_part` возвращает `bytes` на часть; декодировать `utf-8`. По контракту Logs API каждая part несёт собственную строку-заголовок (имена `ym:s:*`/`ym:pv:*`) — отсюда вычитание заголовка **на часть** (риск №3). Окончательно подтверждается live-smoke оркестратора (2.7); в 2.3 спроектировано защитно.
- **Web-ресёрч не требуется:** stdlib стабилен и зафиксирован локом; внешнего сетевого контракта в истории нет (live-smoke неприменим, как в 2.2).
