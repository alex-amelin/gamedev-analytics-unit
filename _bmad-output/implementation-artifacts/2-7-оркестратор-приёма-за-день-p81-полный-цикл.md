## Story

As a оператор юнита,
I want оркестратор полного цикла Logs API для одного дня/источника,
so that день доезжал до сырья атомарно, под локом, со сверкой строк и чекпойнтом — а реальный контракт API проверялся end-to-end.

**Контекст эпика.** Седьмая история Epic 2 «Приём данных и безопасное обновление хранилища» — **точка сборки**: все примитивы 1.x/2.1–2.6 уже стоят, p81 склеивает их в один связный путь записи дня. Готовое (потребляется, не переписывается): `metrica_client` (1.3 — цикл Logs API, retry/rate-limit), `dates` (1.4 — clamp/формат/`moscow_yesterday`), `catalog` (1.5 — список полей выгрузки), `env_reader` (1.2 — креды), `parquet_store.write_partition` (2.2 — атомарная запись дня), `row_check` (2.3 — жёсткая сверка строк), `load_state` (2.4 — мета + реконсиляция), `writer_lock` (2.5 — лок одного писателя), `database_manager` (2.1 — соединение), `views.create_views` (2.6 — типизированные view; `ready-for-dev`, зависимость). Покрывает **FR-1** (оркестрация полного асинхронного цикла create→poll→download→clean; fail на `processing_failed`/`canceled`) и **FR-3** (оба источника visits+hits независимо; связь `watch_ids`↔`watch_id`). Место в упорядоченной цепочке: 2.7 = **один день один источник**; 2.8 = инкремент/hot-window (решение «какие дни» + диапазон); 2.9 = команда `gdau-logs update` (поверхность). После 2.7 один день реально доезжает от API до сырья под локом со сверкой и чекпойнтом.

**Почему p81 — это «склейка», а не новая логика (корень требования).** Каждый примитив 2.1–2.6 написан как **независимый, тестируемый в одиночку** кусок (conn инъектируется, сети нет, БД сам не открывает). Ни один не знает про асинхронный цикл Logs API и про порядок операций «download → собрать день → temp→rename → сверка → мета → clean». p81 — единственное место, где этот **порядок** материализуется (architecture.md:379–381 «протокол идемпотентного дня»). Вся «тяжёлая» механика (retry/rate-limit, атомарность, сверка-fail, реконсиляция, лок) уже реализована — p81 её **вызывает в правильной последовательности**, а не реализует заново (project-context anti-pattern «реимплементация retry/rate-limit вне клиента»).

**2.7 — ПЕРВАЯ end-to-end история против реального Logs API после 1.6.** Истории 2.1–2.6 в сеть не ходили (live-smoke неприменим — записано в каждой). p81 **и есть** реальный цикл Logs API: create→poll→download. Значит правило project-context «тесты внешнего API — не только моки: обязателен opt-in live-smoke» **снова в силе** (как 1.3/1.6). Именно здесь подтверждаются неподтверждённые предпосылки соседей: реальный формат TSV (разделитель, заголовок-на-часть), формат массивов и их экранирование (опровергнутая предпосылка AC #8 в 2.6), статусы асинхронного цикла. **Live-smoke обязателен; фикстуры offline освежаются из реального ответа** ([[realapi-smoke-tests]], defer 2.3 `deferred-work.md:35`, defer 2.6 примечание к AC #8).

**Это НЕ вендоринг — свой оркестратор.** В дереве архитектуры `p81_load_logs.py` — **своя** оркестрация (architecture.md:424 «**своя** оркестрация (Parquet, не DuckDB-CAST)»). У directaiq `p81_load_logs.py` наследует `BaseScript`, грузит CSV→DuckDB с `CAST` в самой таблице, держит `ym:s:*`-имена, тащит `config_manager`/goal-плейсхолдеры/`_force_drop_tables(DROP TABLE)`, параллельную очередь и «сверку как warning» (`directaiq .../p81_load_logs.py:783 "Расхождение: …" — warning!`). Наша модель — Parquet-партиции + snake_case + жёсткий fail + ноль инфры — **осознанное отличие**. Форму (`_create_parser`, поэтапный цикл create→wait→download) можно держать узнаваемой, но `BaseScript`/`config_manager`/`DROP TABLE`/параллель/warning-сверку **не тащить** (NFR-6). Прямого построчного аналога нет — пиши по контракту ниже. Дерево тестов архитектуры называет `test_p81_orchestrator.py` (architecture.md:484).

### Главные риски / решения (читать до кода)

> ✅ **РЕШЕНИЕ ШЕФА (2026-05-25): вариант A — УТВЕРЖДЁН.** Контракт ниже написан под **вариант A** (`load_day(conn, client, …)` инъектируемое ядро + run-level `ingest_day(source, date)` с одним захватом лока). **Реализуй вариант A.** Разобранная отклонённая альтернатива (вариант B — 2.7 даёт только чистое `load_day`, весь lock/run-level отложен в 2.8/2.9) оставлена в риске №1 для трассируемости решения. Всё остальное (порядок цикла, сплиттер, live-smoke, границы) от выбора A/B **не зависело**.

1. **Декомпозиция 2.7 и scope `.writer.lock`: ядро `load_day(conn, client, …)` (инъекция) + run-level `ingest_day(source, date)` (лок один раз). ✅ ВАРИАНТ A УТВЕРЖДЁН ШЕФОМ (2026-05-25).**

   **Почему вообще вопрос.** `writer_lock` (2.5) **не реентерабелен** (`writer_lock.py:82` docstring: «повторный захват того же пути тем же процессом конфликтует сам с собой → `WriterLockHeldError`»). Архитектура и docstring 2.5 фиксируют: лок берётся **один раз на весь прогон обновления — p81 (2.7)** (`writer_lock.py:19`, architecture.md:384). Значит лок **не** может браться внутри per-day-цикла, иначе 2.8 (диапазон дней под одним локом) словит сам-с-собой конфликт. Поэтому держатель лока — **тот, кто зацикливает дни**, а не сам день.

   **Вариант A (РЕАЛИЗОВАТЬ при подтверждении Шефа).** Два уровня:
   - **`load_day(conn, client, source, date, *, catalog=None, poll-параметры) -> int`** — **ядро цикла дня на инъектируемых `conn` + `client`**. НЕ берёт лок, НЕ открывает БД, НЕ строит клиент, НЕ ходит в сеть retry-логикой (это клиент). Делает: `mark_loading` → `create_log_request` → poll до `processed` → download всех частей → парсинг TSV → `write_partition` (2.2) → `verify_row_count` (2.3) → `mark_loaded` (**точка коммита**) → `clean`. Возвращает число строк. **Это главный тестируемый шов** (как `load_state`/`views`: conn инъектируется → in-memory DuckDB + поддельный клиент, без сети и без лока).
   - **`ingest_day(source, date, *, …) -> int`** — **run-level единичного дня**: `with writer_lock(): with DatabaseManager.connection() as conn:` строит клиент из `env_reader`, `ensure_load_state_table(conn)` + `create_views(conn)` (один раз), затем `load_day(conn, client, source, date)`. Это и есть AC #3 (запись под локом) + AC #1 (запускаемый p81 для одного дня). Ad-hoc единичный прогон.

   **2.8 НЕ зовёт `ingest_day` в цикле** (реентрантность!) — 2.8 берёт лок/conn/клиент **один раз** и зовёт **`load_day`** напрямую по вычисленному набору дней (reconcile→skip+hot-window). Контракт ниже это фиксирует в docstring `load_day`/`ingest_day`. Так лок-scope = «зацикливающий вход» (для 2.7 это `ingest_day`-один-день; для 2.8 — её диапазон-вход).

   **Вариант B (ОТКЛОНЁН Шефом 2026-05-25).** 2.7 отдаёт **только** `load_day(conn, client, …)`, а run-level (лок + conn + клиент + ensure_table + create_views) **целиком** в 2.8/2.9. Плюс: 2.7 — чистая библиотечная функция, минимум поверхности. Минус: AC #1 «когда **запускается** p81» и AC #3 «p81 **пишет** под локом» в 2.7 не имеют прод-входа — демонстрируются лишь тестом (load_day под локом в тесте). Эффективно переносит интеграцию в 2.8. **Отклонён** — Шеф выбрал A (AC #1/#3 демонстрируются end-to-end уже в 2.7 + есть ad-hoc единичный прогон). Контракт `load_day` идентичен в обоих, отличие — наличие `ingest_day` в 2.7.

2. **КРИТИЧНО (cross-component, defer 2.3): сплиттер записей p81 ОБЯЗАН совпасть с `count_part_rows` (2.3) — единый сплиттер.** `row_check.count_source_rows` считает `expected` = строки-данные сырого TSV (`split("\n")`, `rstrip("\r")`, отбрасывая пустые, минус **один заголовок на часть** — `row_check.py:76`). `actual` = возврат `write_partition` = число **распарсенных** p81 строк. Если сплиттер p81 разойдётся с `count_part_rows` (например, `csv`-модуль, склеивающий встроенный `\n` в значении; или `splitlines()`, режущий `\v`/`\f`/`\x85`/LS/PS), сверка даст **misfire**: ложный жёсткий fail на валидном дне **или** (если оба ошибаются одинаково) маскировку реальной потери. **Решение (defer 2.3 `deferred-work.md:35`): единый сплиттер.** Вынеси в `row_check` публичный хелпер `split_tsv_rows(part: bytes | str) -> list[str]` (декод UTF-8; `split("\n")`; `rstrip("\r")`; отбрасывание пустых — РОВНО логика `count_part_rows.py:71–76`), перепиши `count_part_rows` через него (`max(0, len(split_tsv_rows(part)) - 1)` — поведение неизменно, регресс-тесты 2.3 зелёные), и p81 использует **тот же** хелпер: `lines = split_tsv_rows(part)`, `header = lines[0]`, `data = lines[1:]`, ячейки = `line.split("\t")`. Контракт-тест: часть со встроенным `\r`/CRLF/хвостовым переводом строки/пустой строкой → число распарсенных p81 строк-данных **==** `count_part_rows(part)`. Так `expected` и парсинг согласованы по границам строк, а сверка ловит реальную потерю на участке **download→сборка дня→запись** (потеря целой части, дроп строки до `write_partition`). **Экранирование встроенных `\t`/`\n` в значениях `yandex-docs` не подтверждает → фиксируем простой `\t`/`\n`-сплит сейчас, подтверждаем live-smoke 2.7** (примечание AC #1, defer 2.3).

3. **Механизм загрузки модуля: digit-префикс `8x_metrica_logs_api` (defer 1.1 `deferred-work.md:6`).** Каталог `scripts/8x_metrica_logs_api/` начинается с цифры → `import scripts.8x_metrica_logs_api.p81_load_logs` как **statement** = `SyntaxError`. Но `importlib.import_module("scripts.8x_metrica_logs_api.p81_load_logs")` (строкой) **работает** — идентификатор валидной точечной нотации не требуется. В каталоге **нет** `__init__.py` (только `.gitkeep`) → неявный namespace-пакет, `import_module` отрабатывает. **Следствия:** (а) тесты импортируют p81 через `importlib.import_module(...)`, не через `from scripts.8x_… import …`; (б) будущий CLI `update` (2.9) импортирует p81 так же; (в) **прямого entry-point у p81 НЕТ** (`pyproject [project.scripts]` = `gdau-logs`→`logs_api_cli`, `gdau-init`→`init_project`; p81 дёргает 2.9, не консоль). Сам модуль — обычный: `from __future__ import annotations` первой строкой, абсолютные импорты `from scripts.utils.… import …`. Спека `deferred-work.md:6` упоминала `spec_from_file_location` как опцию — **не нужно**: `import_module` проще и достаточно (`[[simplicity-first]]`).

4. **LIVE-SMOKE ОБЯЗАТЕЛЕН (project-context, [[realapi-smoke-tests]]) + освежение фикстур.** 2.7 — реальный цикл Logs API → `test_p81_orchestrator_live.py` с `@pytest.mark.live` (как `test_metrica_client_live.py`/`test_logs_api_cli_live.py`; `addopts="-m 'not live'"` → в CI отсеян, явный `uv run pytest -m live`). Минимально: **узкое окно в 1 день** ≤ «вчера по МСК», источник `visits` (опц. `hits`), креды из `.env` хранилища → **`pytest.skip`** с понятной причиной если кредов/`GDAU_DATA_ROOT` нет (не ложный красный). Гоняет **полный реальный цикл** `ingest_day`/`load_day` против реального API (create→poll→download→write→verify→load_state→clean), уважая rate-limit (≤5000/day) и асинхронный poll (~30s). **Проверяет реальный контракт:** имена/типы полей ответа совпадают с каталогом; формат TSV (разделитель, заголовок-на-часть); статусы цикла. **Освежить offline-фикстуры из реального ответа:** `tests/fixtures/logs_visits_sample.tsv` уже есть (3-колоночный мини); **добавить `logs_hits_sample.tsv`**; при возможности обновить из реального прогона (формат массивов, экранирование строковых массивов — неподтверждённое в 2.6). Закрывает defer 2.3 и подтверждает решение 2.6 вариант A.

5. **Порядок операций и точка коммита дня (AC #1, #7) — протокол идемпотентного дня (architecture.md:379–381).** В `load_day`, СТРОГО: `mark_loading(conn,source,date)` → `create_log_request` → poll → **download ВСЕХ частей в память** → парсинг TSV → `write_partition(...)` (temp→`os.replace`, атомарно) → `verify_row_count(expected, actual, source=, date=)` → `mark_loaded(conn,source,date,row_count)` **= ТОЧКА КОММИТА** → `clean_log_request`. День «загружен» ТОЛЬКО после rename + сверка + `mark_loaded` (architecture.md:380). **`clean` ПОСЛЕ коммита (AC #7):** если `clean` упал (`RuntimeError` клиента) уже **после** `mark_loaded` → это **WARNING** (остаток квоты), день **остаётся загруженным**, без отката и без fail. **`clean` — не часть коммита.**

6. **Частичные части → день НЕ персистится (AC #6): download всё-или-ничего в память.** Сначала скачать **ВСЕ** части (`download_log_request_part` → `bytes`) в список **в памяти**; только если **все** успешны → парсинг + `write_partition`. Любая часть упала (`RuntimeError`/`OSError`) → исключение **ДО** `write_partition` → ничего не записано (не «собрали что есть», AC #6). Части держим **байтами в памяти** (не россыпью `.tsv`-файлов, как ad-hoc `download` CLI 1.6) → нет мусора частичных файлов (defer 1.6 `deferred-work.md:27`). Объём — части одного дня; для single-day приемлемо.

7. **Дисциплина poll (AC #5): ограниченный цикл, без вечного ожидания.** Poll `get_log_request(req_id)` циклом: интервал ~30s, **верхняя граница** ожидания, **лимит подряд-ошибок** → fail с диагностикой. Значений `config_manager` у нас НЕТ (не тащим, NFR-6) → **модульные константы** `POLL_INTERVAL_S=30.0`, `POLL_TIMEOUT_S=3600.0` (60 мин, как directaiq `_wait_for_request`), `MAX_CONSECUTIVE_POLL_ERRORS=5` + **переопределяемые keyword-параметры** (тесты дают крошечные значения). Статусы: `processed`→дальше; `canceled`/`processing_failed`→fail с диагностикой (AC #2); прочие (`created`/`processing`/…)→продолжать poll до таймаута. Ошибка `get_log_request` (клиент уже отретраил транзиент — поднятое наружу терминально) → счётчик подряд-ошибок, бэкофф, fail на лимите. **rate-limit/retry заново НЕ реализуем** (клиент, NFR-3). **Сон — через инъектируемый шов** (`sleep`-параметр со `default=time.sleep` ИЛИ poll-интервал `0.0` в тестах) — реальные 30s в offline-тестах недопустимы.

8. **Пустой день без частей → колонки из каталога (AC косвенно, граница 2.2 «валидный пустой день»).** API вернул `processed` с **0 частей** (честно пустой день) ИЛИ части только с заголовком → `expected = count_source_rows(parts) = 0`. Чтобы записать **пустую типизированную партицию** (2.2 AC #7), `write_partition` нужны `columns`: при наличии частей — **заголовок TSV** (родные имена); при **0 частей** — `catalog.metrica_fields(source)` (тот же список, что заказан). `rows=[]` → `write_partition` пишет пустую партицию со схемой, возвращает `0`; `verify_row_count(0,0)` → успех; `mark_loaded(...,0)`. День помечен загруженным, **не** «отсутствует»/ошибка. (Совпадает с `parquet_store` AC #7 и `row_check` «0==0 успех».)

9. **Источник колонок = заголовок TSV (родные имена), не переупорядочивать (граница 2.2).** `write_partition(columns=...)` ждёт **родные имена Метрики** в порядке TSV-заголовка (`parquet_store.py:60` — «задают и схему, и порядок»; он сам мапит→snake_case по каталогу, неизвестная колонка→`ValueError`). Бери заголовок **как пришёл** из части (он авторитетен для выравнивания колонка↔ячейка) — он обязан совпасть с `catalog.metrica_fields(source)` (что заказали); `write_partition` fail-loud ловит дрейф. Не переименовывать в p81 (это забота `parquet_store`), не держать `ym:s:*` в логике сверх передачи в `write_partition`.

10. **Граница дат: НЕ молчаливый clamp одиночного дня.** Для одного дня `date1=date2=date`. Голый `clamp_date_range(date,date)` при `date > вчера` **сдвинул бы день** на вчера (загрузили бы НЕ тот день — тихий баг). Поэтому в `load_day`/`ingest_day`: **валидировать** `parse_date(date) <= moscow_yesterday()` → иначе fail-loud `ValueError` (Logs API требует `date2 < today`; будущий день грузить нельзя). Диапазонный clamp (срезать хвост диапазона на «вчера») — забота 2.8 (`clamp_date_range` на границах диапазона). p81 формат дат — `format_date` (1.4), строго `YYYY-MM-DD`.

11. **Связь visits↔hits (AC #4) — факт данных, не код p81.** Оба источника грузятся **независимо** (свой цикл, своя партиция `data/raw/{source}/`, своя строка `load_state`). Связь `visits.watch_ids` ↔ `hits.watch_id` — свойство **каталога/модели** (поля хранятся, джойнятся в SQL агента через view 2.6), p81 **спецлогики связи не несёт**. AC #4 = «каждый грузится независимо; связь через `watch_ids↔watch_id`» → p81 просто умеет оба источника (тот же `load_day` с `source∈{visits,hits}`), связь зафиксируй в docstring/спеке.

12. **На любой fail между `mark_loading` и `mark_loaded` → `mark_failed` (best-effort) + raise (AC #2, опора реконсиляции 2.4).** Оберни цикл дня `try/except`: исключение (poll-fail/download-fail/`RowCountMismatchError`/прочее) → `mark_failed(conn,source,date)` (защитная отметка; реконсиляция 2.4 трактует `loading`/`failed` как незагруженный) + **re-raise** (наружу non-zero). `mark_failed` сам может упасть, если сбой — в `conn`/БД → оборачивай его `contextlib.suppress`/best-effort (вторичная ошибка не должна маскировать исходную); даже если `mark_failed` не прошёл, остаётся `loading` от `mark_loading` → реконсиляция всё равно перельёт (двойная страховка). **`RowCountMismatchError` (2.3) НЕ глушить** — это жёсткий fail целостности (наружу non-zero).

## Acceptance Criteria

1. **Given** дата + источник, **When** запускается p81, **Then** полный цикл: `create` → poll до `processed` → `download` частей → парсинг TSV → `parquet_store` (2.2) → сверка (2.3) → `load_state` (2.4) → `clean`.
2. **Given** статус `processing_failed`/`canceled`, **When** опрашивается запрос, **Then** fail с диагностикой, без молчаливого пропуска.
3. **Given** любая запись, **When** p81 пишет, **Then** под `.writer.lock` (2.5).
4. **Given** оба источника, **When** запускаются visits и hits, **Then** каждый грузится независимо; связь через `watch_ids`↔`watch_id` (FR-3).
5. **Given** poll, **When** ожидание затягивается, **Then** интервал ~30s, верхняя граница ожидания и лимит подряд-ошибок → fail (значения из конфига; rate-limit/retry — из клиента, не заново).
6. **Given** скачивание частей, **When** хотя бы одна часть не скачалась, **Then** день НЕ персистится (никаких частичных дней) — fail целиком, а не «собрали что есть». _[edge-case: частичные parts]_
7. **Given** точка коммита дня = запись `load_state` (после rename+сверки), **When** `clean` падает уже после коммита, **Then** это WARNING (квота-остаток), день остаётся загруженным — не откат и не fail. _[edge-case: clean после коммита / порядок операций]_

> **Примечание к AC #1 (формат TSV — подтверждается live-smoke).** Парсинг TSV в p81 фиксирует **простой** контракт: единый сплиттер строк с `row_check` (риск №2), разделитель ячеек `\t`, заголовок в каждой части. Экранирование встроенных `\t`/`\n` в значениях `yandex-docs` не подтверждает — **реальный wire-формат подтверждает live-smoke 2.7** (риск №4, defer 2.3 `deferred-work.md:35`); offline-фикстуры освежить из реального ответа.

> **Примечание к AC #3/#5 («из конфига», scope лока).** `config_manager` не тащим (NFR-6) → poll-параметры (интервал/таймаут/лимит ошибок) — модульные константы + переопределяемые kwargs (риск №7). Scope `.writer.lock` — **один захват на зацикливающем входе** (для 2.7 один день — `ingest_day`; для диапазона — 2.8), `writer_lock` не реентерабелен (риск №1).

### Порядок операций дня (протокол идемпотентного дня; architecture.md:379–381, риск №5)

```
mark_loading(conn, source, date)                 # 2.4 — день «в процессе» (реконсиляция: незагружен)
  → fields = catalog.metrica_fields(source)       # 1.5 — список из SSOT (FR-2)
  → resp = client.create_log_request(date,date,fields,source)  # 1.3
  → req_id = (resp.get("log_request") or resp)["request_id"]
  → info = poll до status=='processed'            # риск №5/№7; failed/canceled → RuntimeError (AC #2)
  → parts_bytes = download ВСЕХ частей в память    # AC #6 всё-или-ничего (риск №6)
  → columns, rows = parse(parts_bytes)             # единый split_tsv_rows (риск №2); 0 частей → catalog-поля,[] (риск №8)
  → expected = count_source_rows(parts_bytes)      # 2.3 — сырой счёт (заголовок на часть)
  → actual   = write_partition(source, date, columns, rows)  # 2.2 — temp→os.replace атомарно
  → verify_row_count(expected, actual, source=, date=)       # 2.3 — расхождение → RowCountMismatchError (НЕ глушить)
  → mark_loaded(conn, source, date, actual)        # 2.4 === ТОЧКА КОММИТА ДНЯ
  → clean_log_request(req_id)                       # 1.3 — ПОСЛЕ коммита; fail → WARNING, день остаётся (AC #7)
# на любом исключении до коммита: mark_failed(best-effort) + re-raise (риск №12)
```
