# Init и хранилище игры

Этот компонент отвечает за **разворачивание рабочего пространства одной игры** (`gdau-init`):
как из общего dev-репо родить отдельную папку игры, связать её с инструментами юнита и
подготовить к приёму данных. Документ описывает все части Epic 4: **симлинк-контракт и
preflight** (история 4.1), **шаблон хранилища и файл описания игры** (история 4.2) и
**полный разворот командой `gdau-init`** (история 4.3) — он собирает первые две вместе.

## Симлинк-контракт и preflight

### Что делает

При разворачивании игры этот блок **связывает её папку с инструментами юнита по явному списку**.
Список лежит в одном файле — `templates/paths-to-symlink.csv` (две колонки: `path` — что
связываем, `comment` — зачем). Связываемое — это код юнита, каталог схемы, справочники Logs API
и манифест проекта: они приходят в папку игры **ссылками** на dev-репо, а не копиями.

Перед тем как создавать ссылки, блок **проверяет, что система вообще умеет их создавать**. На
Windows для этого нужен включённый «Режим разработчика» (Developer Mode); если его нет — блок
**сразу останавливается с понятной инструкцией**, как его включить, и **ничего не разворачивает
наполовину**. На Linux ссылки создаются всегда, проверка там просто проходит.

Если всё в порядке, блок создаёт ссылки по списку. Уже существующая ссылка при повторном
развороте **обновляется** (а не вызывает ошибку). Если в списке указано то, чего в dev-репо нет,
— блок останавливается с понятной ошибкой. Если связать удалось только часть, а на чём-то
сорвалось — уже созданные ссылки **снимаются**, чтобы не осталось «полусвязанного» состояния.

### Зачем нужен

Чтобы **одно исправление инструмента видели сразу все игры**. Код, каталог схемы и справочники
живут в одном месте — dev-репо. Игры на них **ссылаются**, поэтому фикс бага или правка каталога
в dev-репо (после `git pull`) автоматически долетает до каждой игры — без копирования, без
миграций, без риска, что у разных игр разойдётся инструмент.

И чтобы **папку игры можно было перенести** на другую машину или другую ОС простым копированием.
Ссылки заданы **относительно** (не «жёстким» абсолютным путём вроде `G:\git\...`), поэтому пока
рядом переносят и dev-репо, и папку игры, ссылки не рвутся. Это и есть переносимость Windows↔Linux,
которую требует проект.

Наконец — **безопасность данных**. Если на месте будущей ссылки лежит реальный файл или папка
(а не ссылка), блок их **не удаляет**, а останавливается: данные игры не должны пострадать от
разворота.

### Контракт с другими

- **Список ссылок — единый источник истины.** Он правится в одном файле (`paths-to-symlink.csv`),
  а не разбросан по коду. Добавить новую связь — значит дописать строку в этот файл.
- **Что сейчас в списке.** Стабильные цели dev-репо: код юнита (`scripts`), каталог схемы
  (`development-docs`), справочники Logs API (`yandex-docs`), манифест проекта (`pyproject.toml`)
  с его `README.md` (на него ссылается поле `readme` манифеста — hatchling требует этот файл при
  editable-сборке в `uv sync`), файл блокировки зависимостей (`uv.lock` — нужен, чтобы окружение
  игры ставилось строго по локу) и конфиг MCP-сервера (`.mcp.json` — это канал чтения данных игры,
  даёт агенту инструмент запросов).
  Настройки агента разработки (`.claude/…`) в список **намеренно не добавлены**: они несут
  служебные навыки разработки самого юнита, а не нужны рабочему пространству игры.
- **Поведение на краях** (обещания блока): пустой/битый список или дубли → понятная ошибка;
  цель из списка отсутствует в dev-репо → понятная ошибка ещё **до** создания первой ссылки;
  существующая ссылка → обновляется, не падает; реальный файл/папка на месте ссылки → остановка,
  данные не удаляются; платформа не умеет ссылки → остановка с инструкцией Developer Mode; сбой
  посреди разворота → откат уже созданных ссылок.
- **Кто это вызывает.** Единственный потребитель — команда `gdau-init` (история 4.3). Она зовёт
  блок на шаге «ссылки», передавая ему **корень dev-репо и корень папки игры** (сам блок их не
  ищет — это делает 4.3). Блок ни в сеть не ходит, ни базу `gdau.duckdb` не открывает, ни данные
  не пишет — только работает с файлами: читает список, проверяет способность, создаёт ссылки.

### Границы (что делают другие истории Epic 4)

- **Копирование шаблона хранилища** (стартовая структура папок игры) — **файл описания игры**
  `gdd.md` и **каталог событий** `EVENTS.md` — история 4.2.
- **Полный разворот** `gdau-init` (проверка имени игры → копирование шаблона → ссылки (этот блок)
  → `.env` → установка окружения `uv sync` → создание `gdau.duckdb` и представлений → `git init`)
  и **откат всего хранилища** при сбое посреди разворота — история 4.3. Этот блок откатывает
  только **свои** ссылки; полный откат хранилища оборачивает его в 4.3.
- **Где что живёт.** Данные игры, её `.env` и база — в папке игры (хранилище, со своим `git`).
  Инструмент (код, каталог, справочники, манифест) — в dev-репо, в хранилище приходит ссылками.
  В dev-репо данные игр не пишутся.

## Шаблон хранилища и файл описания игры

### Что делает

Новая папка игры разворачивается из готового **шаблона**. В шаблоне — ровно те файлы, которые
игре нужны с первого дня: образец файла кредов (`.env.example`), правила игнорирования для
`git` хранилища (`.gitignore`), краткие инструкции агенту для работы в этом пространстве
(`CLAUDE.md`), **файл описания игры** (`gdd.md`) и **каталог событий аналитики** (`EVENTS.md`) —
оба заполняет владелец.

При развороте эти файлы **копируются** в папку игры. Если разворот делается **повторно** поверх
уже существующей папки, уже заполненные владельцем **описание игры (`gdd.md`) и каталог событий
(`EVENTS.md`) не затираются** — служебные файлы обновятся из шаблона, а то, что внёс владелец,
остаётся как есть. Если шаблона на месте нет или он неполный — разворот **сразу останавливается
с понятной ошибкой**, ничего не создавая (чтобы не родить пустую полу-готовую папку игры).

### Зачем нужен

Чтобы у **каждой игры с первого дня были все нужные файлы в одинаковой форме** — не приходилось
собирать папку руками и не было риска что-то забыть. И чтобы **агент знал, с какой игрой
работает**: он читает `gdd.md` (платформы, ключевые сценарии, что значат поля под эту игру),
`EVENTS.md` (цели и события Метрики и их смысл) и `CLAUDE.md` (правила рабочего пространства) —
без этого контекста выводы были бы поверхностными.

Ещё одна задача шаблона — **держать секреты и данные вне `git`**. Правила игнорирования в шаблоне
сразу настроены так, чтобы реальные креды (`.env`) и рабочие данные (`data/`, база) **не попадали
в историю** репозитория игры.

### Контракт с другими

- **Шаблон содержит ровно нужные игре файлы** — пять: образец кредов, правила игнорирования,
  инструкции агенту, описание игры и каталог событий аналитики. **Инструмент юнита** (код, каталог
  схемы, справочники) в шаблон **не входит**: он не копируется, а **подключается ссылками** из
  dev-репо (см. раздел про симлинки выше). Поэтому шаблон намеренно урезан под нашу задачу — без
  лишних надстроек.
- **Описание игры** (`gdd.md`) и **каталог событий** (`EVENTS.md`) заполняет владелец, и они
  **живут в папке игры** обычными файлами (не ссылками) — поэтому доступны агенту в рабочей сессии.
  Образец кредов копируется как `.env.example`; реальный `.env` со значениями появляется при
  развороте и в `git` не попадает.
- **Повторный разворот** обновляет служебные файлы из шаблона, но **бережёт заполненные владельцем
  `gdd.md` и `EVENTS.md`** — внесённый контекст игры не теряется.
- **Кто это вызывает.** Копирование шаблона — один из шагов команды `gdau-init` (история 4.3);
  она передаёт, **куда** копировать (корень папки игры). Сам шаг копирования в сеть не ходит, базу
  не открывает, данные не пишет и ссылки не создаёт — только проверяет шаблон и копирует файлы.

### Границы (что делают другие истории Epic 4)

- **Механизм ссылок** на инструмент dev-репо — история 4.1 (тот же файл, раздел выше).
- **Полный разворот** `gdau-init` (проверка имени игры → копирование шаблона (этот блок) → ссылки
  → реальный `.env` → установка окружения → база и представления → `git init`) и **откат всего
  хранилища** при сбое посреди разворота — история 4.3. Сам шаг копирования частичную копию **не
  откатывает**; полный откат хранилища оборачивает его в 4.3 (и при этом бережёт заполненные
  владельцем `gdd.md` и `EVENTS.md`).

## Полный разворот командой `gdau-init`

### Что делает

Одна команда `gdau-init {имя-игры}` поднимает рабочее пространство одной игры рядом с dev-репо
(папка `{имя-игры}-analytics`) **за один проход**: проверяет, что имя свободно и безопасно →
копирует стартовый шаблон (раздел про
шаблон выше) → связывает папку игры с инструментом юнита ссылками (раздел про симлинки выше) →
создаёт файл кредов `.env` с прописанным путём к хранилищу (токен и счётчик остаются пустыми) →
ставит окружение игры (`uv sync`) и **делает инструмент юнита видимым для команд этого окружения**
(чтобы `gdau-logs`/`gdau-init`, запущенные из папки игры, находили свой код, а не падали) →
создаёт пустую рабочую базу `gdau.duckdb` с представлениями `visits` и `hits` → заводит для папки
игры **отдельный git** и делает первый коммит (без `.env`).

### Зачем нужен

Чтобы **новая игра поднималась за минуты, без ручной возни** и без риска что-то забыть или собрать
непоследовательно. На выходе хранилище **готово к первой выгрузке** — владельцу остаётся только
вписать в `.env` токен Метрики и номер счётчика, после чего сразу работает приём
(`gdau-logs update`). Команда заменяет десяток ручных шагов одной и одинаково работает на Windows
и Linux.

### Контракт с другими

- **Имя игры проверяется строгим шаблоном** (буквы/цифры/`_`/`-`, без пробелов, разделителей пути,
  `..`, ведущих точек и зарезервированных Windows-имён). Непригодное имя → понятный отказ **до**
  любых действий.
- **Папка хранилища именуется `{имя-игры}-analytics`** — фиксированный суффикс `-analytics` отличает
  аналитический сторадж от прочих каталогов игры рядом с dev-репо. Суффикс — только в имени папки;
  само имя игры в сообщениях и в первом git-коммите остаётся без него.
- **Занятое имя → остановка без перезаписи.** Если папка с таким именем уже есть рядом с dev-репо,
  команда останавливается и ничего не трогает — чужие данные не пострадают.
- **Сбой на любом шаге → папка игры убирается целиком** (полный откат): имя снова свободно, повтор
  команды чистый. Откат снимает ссылки на инструмент, **не удаляя сам инструмент** в dev-репо. Если
  что-то помешало дочистить — команда честно скажет, что удалить вручную.
- **Секреты не попадают в git игры.** Реальный `.env` (с будущими токеном/счётчиком) в первый
  коммит не включается и игнорируется git хранилища.
- **Путь к хранилищу считается от dev-репо**, а не от текущего каталога запуска — папка игры всегда
  оказывается ровно соседом dev-репо, откуда бы команду ни запускали.
- **Кто это вызывает.** Команду запускает оператор/владелец при заведении новой игры. Прямых
  программных потребителей нет — это листовая команда. После неё хранилище читают/пишут приём
  (`gdau-logs`) и MCP-чтение, но это уже обычная работа, не разворот.

### Границы (что собирает, а что нет)

- `gdau-init` — **тонкий сборщик**: механику ссылок (4.1), копирования шаблона (4.2), типов
  представлений и работы с базой он **не переписывает**, а вызывает в правильном порядке. Своё у
  него — проверка имени, путь к хранилищу, генерация `.env`, установка окружения, `git init` и
  **полный откат хранилища** (которого примитивы намеренно не делают).
- Что связывается ссылками сейчас (финализировано в 4.3): к коду/каталогу/справочникам/манифесту
  добавлены `README.md` (его требует поле `readme` манифеста при editable-сборке в `uv sync`),
  `uv.lock` (для установки окружения строго по локу) и `.mcp.json` (канал чтения игры);
  `.claude/…` намеренно не связывается (служебные навыки разработки юнита, не рантайм игры).
- Команда **не ходит в сеть** (выгрузка Logs API — это уже `gdau-logs`, не разворот), **не пишет
  данные** игры (только пустую базу с представлениями) и **не трогает dev-репо** (всё создаётся в
  папке игры рядом).
