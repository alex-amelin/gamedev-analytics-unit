# Клиент Logs API

Компонент: разговор с Яндекс Метрикой по сети.
Реализация: `scripts/utils/metrica_client.py`.

## Что делает

Это единственная «телефонная трубка» к Метрике во всём юните. Через него
проходят все обращения к её Logs API — интерфейсу, который отдаёт сырые данные
посещений и событий выгрузками-файлами.

Клиент умеет вести полный жизненный цикл такой выгрузки:

- **заказать** выгрузку за период (`create_log_request`);
- **прикинуть заранее**, потянет ли API такой заказ (`evaluate_log_request`);
- **спросить статус** одного заказа или списка всех (`get_log_request`,
  `get_log_requests`) — выгрузка готовится не мгновенно, её надо опрашивать;
- **скачать** готовый результат по кускам (`download_log_request_part`) — большие
  выгрузки Метрика отдаёт частями;
- **убрать за собой** готовый заказ, освобождая квоту (`clean_log_request`).

Плюс несколько справочных запросов «не про данные, а про устройство аккаунта»:
информация о счётчике (`get_counter_info`), список доступных счётчиков
(`get_counters`) и список целей (`get_goals`). Они лёгкие и используются для
проверки доступа и подсказок.

## Зачем нужен

Метрика ограничивает, как часто к ней можно стучаться (не больше 30 запросов в
секунду и 5000 в сутки) и иногда отвечает временной ошибкой, когда перегружена.
Если бы каждый кусок кода ходил в сеть сам, эти правила пришлось бы повторять
везде — и где-нибудь обязательно ошиблись бы. Поэтому **вся сетевая дисциплина
живёт в одном месте, здесь**:

- клиент сам выдерживает паузы, чтобы не превысить лимит частоты, и считает
  суточный расход — при исчерпании останавливается с понятной ошибкой;
- при временных сбоях со стороны Метрики (коды 429/500/502/503) он сам делает
  несколько повторных попыток с нарастающими паузами, прежде чем сдаться;
- при «постоянных» отказах (нет доступа, неверный токен, не найдено — 401/403/404)
  он **не** тратит попытки впустую, а сразу падает с понятным сообщением, в
  котором виден код ответа.

Код перенесён («завендорен») из соседнего проекта directaiq — проверенный
временем, а не написанный заново. При переносе из него **вырезано всё лишнее**:
агрегированная отчётность, выгрузка офлайн-конверсий, привязка к Яндекс Директу и
тяжёлая библиотека обработки таблиц. Осталась только работа с Logs API на одной
лёгкой сетевой библиотеке.

> **Тонкость про повторы.** Запросы-«вопросы» (узнать статус, список, оценку)
> при временной ошибке повторяются автоматически. А три «действия» — заказать,
> скачать кусок, убрать заказ — намеренно **не** повторяются сами: решение о
> повторе принимает тот, кто ведёт весь цикл выгрузки (оркестратор приёма из
> Эпика 2), потому что вслепую повторять создание заказа или скачивание опасно.
> Это сохранённое поведение исходного клиента, а не недосмотр.

## Контракт с другими

- **Откуда берёт доступ.** Клиент **не добывает** токен и номер счётчика сам — их
  ему **передают готовыми** при создании. Поставляет их компонент кредов
  (см. [`creds.md`](creds.md)): `creds = read_metrica_credentials()` →
  `MetricaClient(token=creds.token, counter_id=creds.counter_id)`. Связывает эти
  две части CLI-команда создания заказа (история 1.6). Такое разделение
  («шов») держит клиент простым и легко проверяемым: в тестах ему дают
  фиктивный токен, без всякого окружения.
- **Что обещает потребителям.** Даёт оркестратору приёма (история 2.7) и CLI
  (1.6) надёжные «кирпичики» жизненного цикла выгрузки: заказать → дождаться →
  скачать части → убрать. Скачанные части отдаёт как есть — сырыми байтами
  (текст с разделителями-табуляциями), без разбора и преобразования: типизацией и
  раскладкой занимаются уже следующие компоненты приёма.
- **Как сообщает о проблемах.** Любой неустранимый сбой (исчерпаны повторы,
  обрыв сети, отказ доступа, ошибка в теле ответа) превращается в обычную ошибку
  с понятным текстом и ненулевым исходом — никаких «зависаний» или голых
  трейсбеков.
- **Что хранит про секрет.** Токен живёт **только** внутри сетевой сессии (в
  заголовке запроса) и нигде не оседает — ни в полях объекта, ни в логах, ни в
  тексте ошибок. Номер счётчика секретом не считается.

## Границы

Клиент **не** решает, за какие даты и какие поля заказывать выгрузку, **не**
сохраняет данные на диск и **не** ведёт учёт, что уже загружено — это работа
оркестратора приёма и хранилища (Эпик 2). Здесь только разговор с Метрикой по
сети и устойчивость к её ограничениям.
