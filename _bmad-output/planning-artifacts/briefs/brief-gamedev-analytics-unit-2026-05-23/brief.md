# Product Brief: Game Dev Analytics Unit

## Executive Summary

**Game Dev Analytics Unit** — это личный инструментарий, превращающий Claude Code в продуктового аналитика для моих HTML5-игр на Яндекс.Играх. Игры собирают поведение игроков в Яндекс Метрике. Ответить на продуктовые вопросы — как игроки себя ведут, как возвращаются, где отваливаются — через интерфейс Метрики можно, но долго и вручную, и это не масштабируется на параллельный ресёрч сразу по нескольким играм.

Юнит решает это так: даёт агенту готовый набор инструментов, чтобы вытащить сырые гранулярные данные через **Яндекс Метрика Logs API**, сложить их в локальное переносимое хранилище (Parquet + DuckDB), держать свежими и консистентными и понимать схему — что значит каждая колонка. Дальше я открываю рабочее пространство конкретной игры и **задаю вопрос или ставлю задачу на research**: данные, описание проекта и инструменты уже на руках у агента.

Цель первой версии — не методология и не «умный аналитик», а **фундамент**: минимально достаточный набор инструментов, чтобы данные жили в базе и по ним можно было сесть и спросить. Дальше дорабатываем итеративно.

## The Problem

Я делаю HTML5-игры на Яндекс.Играх. Метрика честно собирает поведение игроков, но:

- **Через UI Метрики — медленно и вручную.** Retention по дням, путь игрока внутри сессии, поведенческие воронки собрать можно, но каждый разбор кликается руками. На параллельный ресёрч сразу по нескольким играм такого темпа не хватает.
- **Сырые данные тяжёлые и не под рукой.** Logs API отдаёт гранулярное сырьё, но это асинхронная выгрузка (создать запрос → ждать → скачать частями), большие объёмы и формат, который надо куда-то аккуратно положить.
- **Без инструментов каждый вопрос начинается с нуля** — возня с API, файлами и SQL. Это убивает сам импульс «пойду проверю гипотезу».
- **Нет единого места знаний о данных.** Что значит конкретное поле Logs API, как устроена схема — держится в голове и быстро забывается.

## The Solution

Проект-инструментарий, который отдаёт Claude Code всё необходимое, чтобы данные игры были готовы к работе, а я — только спрашивал:

- **Приём данных.** Клиент Logs API + оркестрация полного цикла (создать запрос → опросить статус → скачать части → очистить запрос) за выбранный диапазон дат. Тянем не «всё подряд», а **заданный мной список полей**.
- **Хранение — два слоя.** Сырьё ложится в **Parquet, партиционированный по дням**, *как есть* (строками) — верно источнику, ничего не теряем на кривом TSV. Рабочий слой — **DuckDB** поверх файлов, где заданные поля **типизируются** (`TRY_CAST`: битая ячейка → `NULL` + лог, а не падение дня). Сильное сжатие, ноль сервера, переносится между Windows и Linux как папка.
- **Доступ агента к данным.** MCP-сервер `duckdb_query` (переиспользуем из directaiq с лёгкой доработкой) — даёт Claude Code схему, контекст и выполнение SQL.
- **Обновление.** Инкремент по дню; безопасный идемпотентный перезалив отдельного дня, не ломающий остальные; повторная подкачка свежего «горячего окна» для доезжающих данных.
- **Консистентность и «не сломать базу».** Сверка числа строк с источником (жёсткий fail, не warning), мета-таблица состояния загрузок (чекпойнт), защита от гонок и кривого TSV Метрики.
- **Понимание схемы.** Живой каталог: что значит каждая колонка Logs API — единый источник для DDL и для навигации агента.
- **Описание проекта.** Файл контекста конкретной игры, чтобы агент знал, с чем работает.
- **Инициализация рабочего пространства.** Одной командой разворачивается внешний per-game репозиторий, связанный симлинками с инструментами.

## Architecture & Approach

Двух-слойная схема, проверенная на родственном проекте `directaiq`:

- **Dev-репо (этот проект)** — инструменты, скрипты, миграции, документация. Развивается как сам продукт.
- **Внешнее per-game хранилище** — БД (Parquet+DuckDB), конфиги, рабочая папка. Специфично для каждой игры, живёт отдельно. Связано с инструментами **симлинками**, поднимается командой-инициализатором.

Так один набор инструментов обслуживает N игр, данные изолированы по проектам, а агент (**Claude Code**) — это и есть аналитик, у которого инструменты «на руках».

> `directaiq` — **архитектурный референс**, а не источник логики: он про traffic-аналитику B2B-рекламы (DuckDB, инкремент, миграции, MCP). Источники данных те же (Яндекс), но наш продукт — про поведение игроков, поэтому модель данных и процессы свои.

## Who This Serves

**Я — соло-геймдев**, делающий игры на Яндекс.Играх и ведущий по ним продуктовую аналитику сам. Мне нужно быстро превращать вопросы об игре в ответы, опираясь на сырые first-party данные, и уносить решения в разработку.

**Вторично:** каждая следующая моя игра — через тот же механизм внешних рабочих пространств.

## Success Criteria

Первая версия работает, если:

- **Главный сигнал.** Данные за последние ~30 дней одной игры лежат в хранилище; я открываю её рабочее пространство, задаю вопрос через Claude/SQL и получаю ответ в ту же сессию.
- **Безопасное обновление.** Повторный запуск обновления идемпотентен — базу не ломает; сверка строк с источником проходит.
- **Один шаг до новой игры.** Развернуть рабочее пространство под новую игру — одна команда.
- **Лёгкость и переносимость.** Объём на диске остаётся скромным (Parquet); папку с данными можно скопировать между Windows и Linux, и всё работает.

## Scope

**В первой версии — фундамент данных** (минимально достаточный): всё, что описано в разделе «The Solution» — приём через Logs API по заданному списку полей, два слоя хранения (Parquet + DuckDB с `TRY_CAST`), инкремент по дню и перезалив «горячего окна», идемпотентность и защита базы, сверка строк с жёстким fail и чекпойнт, собственный каталог схемы, MCP-сервер `duckdb_query`, файл контекста игры и команда инициализации внешнего хранилища.

**Явно вне первой версии:**

- Методологии, плейбуки, регламенты поведения агента (полагаемся на премису: есть данные + контекст + инструменты → дальше просто спросить).
- Сами аналитические ответы (retention, воронки, гипотезы) — это слой использования во внешнем репо.
- Бизнес-витрины, агрегаты, дашборды, модели атрибуции (типизация заданных полей в v1 — это корректные типы для запросов, а не витрины).
- Мобильная аналитика / AppMetrica (только web-Метрика Яндекс.Игр).
- Мульти-аккаунт, расписания/cron, тяжёлая инфраструктура очередей.

## Vision

Личный аналитический юнит, масштабирующийся на все мои игры. Слой инструментов обрастает переиспользуемыми research-плейбуками (v2+), и цикл «есть вопрос об игре → есть ответ» становится рутинным и быстрым. Claude Code — всегда доступный игровой аналитик, заземлённый на сырых first-party данных, а не на агрегатах чужого интерфейса.
