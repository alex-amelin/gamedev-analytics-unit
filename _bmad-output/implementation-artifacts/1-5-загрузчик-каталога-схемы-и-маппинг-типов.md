## Story

As a оператор юнита,
I want программный загрузчик каталога как единого источника полей и типов,
so that список полей и типы брались из одного места без рассинхрона выгрузки/типизации/MCP.

**Контекст эпика.** Пятая история Epic 1 «Каркас юнита и канал Logs API». Каркас (1.1), env-ридер (1.2), вендоренный `MetricaClient` (1.3) и примитив дат (1.4) уже `done`. Эта история кладёт **независимый stdlib-примитив** `scripts/utils/catalog.py` — единственную программную точку доступа к `development-docs/schema-catalog.csv` (SSOT). Покрывает **FR-16** (каталог = SSOT) и **FR-2** (заданный список полей берётся из каталога, не хардкодится). От него зависят: CLI `create` (1.6 — список `fields` для выгрузки), оркестратор p81 (2.7), `views.py` (2.6 — DDL view'ов по типам каталога), MCP-контекст (3.3 — семантика колонок из каталога). Рассинхрон поле↔тип↔семантика здесь = системный дефект ниже по течению.

**Это НЕ вендоринг.** В directaiq каталога-CSV не было (он держал родные `ym:s:*` прямо в таблицах ClickHouse — осознанно отвергнуто, см. architecture#Naming Patterns). Каталог — **наша собственная конструкция**. Пишем крошечный загрузчик на чистой stdlib `csv` (как 1.4 писал `dates.py` на stdlib `datetime`). Прецедента в directaiq нет; форму модуля берём у соседей по `scripts/utils/` (env_reader.py, dates.py): `from __future__ import annotations`, stdlib `logging`, русский docstring, `ValueError` fail-loud, `__all__`, инъектируемый шов (`path=`) для тестов.

**Каталог уже наполнен и сейчас валиден.** `development-docs/schema-catalog.csv` существует и заполнен (74 visits + 41 hits = 115 строк данных). Работа FR-16 сводится к **загрузчику + валидатору**, а НЕ к наполнению с нуля. Прогон валидаций (ниже) подтверждает: каталог сейчас проходит все инварианты этой истории. Загрузчик опирается на каталог, не правит его.

**Главные риски истории.**

1. **Имя колонки `type`, а не `working_type`.** Реальный заголовок CSV — `source,storage_name,metrica_field,type,description` (проверено). Часть `architecture.md` (строки 314/346/449/475) и addendum называют колонку `working_type` — это **устаревшая** формулировка; источник истины = реальный файл + epics AC #1 + project-context (строка 83). Загрузчик читает колонку **`type`**. Валидация заголовка на точный набор колонок ловит дрейф (в т.ч. случайный `working_type`) fail-loud.

2. **Набор DuckDB-типов шире таблицы маппинга AC #4.** Реальный каталог использует `SMALLINT` (×2) и `DOUBLE[]` (×2) — их НЕТ в таблице AC #4 (`UInt64/UInt32/Int32/UInt8/Date/DateTime/String/Array(T)`). Если набор «известных типов» взять буквально из AC #4 — реальный каталог не пройдёт валидацию (конфликт AC #5 «без потерь» ↔ AC #8 «неизвестный тип → fail»). **Решение:** расширить маппинг ClickHouse→DuckDB строками `Int16→SMALLINT` и `Float64→DOUBLE` (обоснованно — реальный каталог их содержит), а набор валидных DuckDB-типов вывести как **кодомен маппинга** + их `[]`-формы. Полный наблюдаемый набор типов (14): `VARCHAR, INTEGER, BIGINT, HUGEINT, SMALLINT, BOOLEAN, DATE, TIMESTAMP` + массивы `VARCHAR[], BIGINT[], HUGEINT[], SMALLINT[], TIMESTAMP[], DOUBLE[]` (scalar `DOUBLE` сам по себе в каталоге не встречается, но обязан быть в базовом наборе ради `DOUBLE[]`).

3. **Наивный `split(",")` порвёт описания.** 31 описание содержит запятые, 5 — точки-с-запятой (проверено в реальном файле; напр. `"Идентификатор визита, уникален в рамках одного года"`). Парсинг **только** через stdlib `csv` (RFC4180), не `str.split(",")` — иначе поля рвутся, тип/описание съезжают. _AC #6 — критичный, файл реально такой._

4. **Резолюция пути каталога под симлинками/wheel.** Каталог — артефакт **dev-репо** (`development-docs/`), приходит в per-game хранилище **симлинком** (не данные оператора → это НЕ кейс `GDAU_DATA_ROOT`/`.env`, см. [[dotenv-usecwd-gotcha]] — там обратное). Дефолтный путь резолвится от модуля: `Path(__file__).resolve().parents[2] / "development-docs" / "schema-catalog.csv"`. `.resolve()` проходит сквозь симлинк хранилища в dev-репо, где `scripts/` и `development-docs/` — реальные соседи. Путь **инъектируется** параметром (`path=`) — тесты всегда передают фикстуру явно. Битый симлинк/отсутствие файла → fail-loud (AC #8).

## Acceptance Criteria

1. **Given** `development-docs/schema-catalog.csv` (`source, storage_name, metrica_field, type, description`), **When** вызывается `catalog.py`, **Then** он парсит файл и отдаёт по источнику списки `storage_name`↔`metrica_field` и `type` (DuckDB) каждого поля.
2. **Given** каталог, **When** запрашивается список полей для выгрузки (FR-2), **Then** возвращается ровно набор `metrica_field` источника — единый источник, без хардкода «на всё».
3. **Given** строку без обязательной колонки или с `source` вне {visits,hits}, **When** загрузчик валидирует каталог, **Then** поднимается понятная ошибка (поле без записи = дефект).
4. **Given** потребность в маппинге справочника, **When** он применяется, **Then** есть таблица ClickHouse→DuckDB: UInt64→HUGEINT, UInt32/Int64→BIGINT, Int32→INTEGER, UInt8→BOOLEAN, Date→DATE, DateTime→TIMESTAMP, String→VARCHAR, `Array(T)`→DuckDB `T[]` (эквивалент `LIST<T>`) — типы не угадываются. _[edge-case: согласовать нотацию `T[]` каталога ↔ `LIST<T>`]_
5. **Given** текущий каталог (74 visits + 41 hits), **When** загрузчик прогоняется, **Then** все строки парсятся без потерь, **And** `visit_id`/`client_id`/`watch_id` имеют `type = HUGEINT`.
6. **Given** описания каталога содержат запятые/точки-с-запятой/кавычки (как в текущем файле), **When** загрузчик парсит CSV, **Then** используется RFC4180-парсер (`csv`-модуль), а не наивный `split(",")` — поля не рвутся. _[edge-case: критично, текущий файл реально содержит такие описания]_
7. **Given** две строки с одинаковым `storage_name` ИЛИ `metrica_field` в одном источнике, **When** каталог валидируется, **Then** понятная ошибка (дубль → коллизия DDL). _[edge-case: дубли]_
8. **Given** `type` вне известного набора DuckDB-типов (включая `T[]`-массивы) ИЛИ `metrica_field` не соответствует префиксу источника (`visits`→`ym:s:`, `hits`→`ym:pv:`) ИЛИ файл каталога отсутствует/симлинк битый, **When** загрузчик стартует, **Then** fail-loud с понятным сообщением. _[edge-case: неизвестный тип / префикс / отсутствие файла]_

### Публичный API (минимальный — NFR-6, не переусложнять)

```python
load_catalog(path: Path | None = None) -> Catalog      # парсинг + валидация, fail-loud
Catalog.fields_for(source) -> tuple[CatalogField, ...] # AC #1: поля источника
Catalog.metrica_fields(source) -> list[str]            # AC #2 / FR-2: список выгрузки
Catalog.duckdb_types(source) -> dict[str, str]         # AC #1: storage_name → тип (для views.py 2.6)
duckdb_type_for(clickhouse_type: str) -> str           # AC #4: справочник → DuckDB
CLICKHOUSE_TO_DUCKDB: dict[str, str]                   # AC #4: таблица маппинга
```

`CatalogField(source, storage_name, metrica_field, duckdb_type, description)` — frozen, slots. Атрибут `duckdb_type` (не `type` — не затенять builtin); маппится из CSV-колонки `type`.

### Маппинг типов — почему кодомен, а не отдельный список (AC #4, #8)

Каталог **уже** хранит DuckDB-типы (не ClickHouse). Поэтому роль `CLICKHOUSE_TO_DUCKDB` двоякая: (1) задокументировать происхождение типов («не угадано», AC #4) и дать прямой маппинг `duckdb_type_for` для будущего сидинга/расширения каталога; (2) его **кодомен** = единственный источник «какие DuckDB-типы легальны» для валидации (AC #8). Не дублировать набор типов вторым литералом — иначе он разъедется с маппингом. Массивы: валиден `T[]`, где `T` — скаляр из кодомена (один уровень; каталог глубже не идёт). Нотация каталога — `T[]`; `LIST<T>` — то же самое в прозе доков (DuckDB принимает обе формы), в CSV не пишем — это и есть «согласование» из edge-case AC #4.

**Осознанное расширение таблицы AC #4:** добавлены `Int16→SMALLINT` и `Float64→DOUBLE`. Буквальная таблица AC #4 их не содержит, но реальный каталог использует `SMALLINT`/`DOUBLE[]` → без расширения AC #5 («все строки без потерь») и AC #8 («неизвестный тип → fail») противоречат друг другу. Расширение — единственный способ удовлетворить оба. _Вынесено в финальные вопросы Шефу; по умолчанию применяю расширение (надёжный вариант) — [[feedback-decide-and-apply]]._

### Парсинг — только stdlib `csv` (AC #6)

`csv.DictReader` (на базе `csv.reader`) — RFC4180: корректно разбирает поля в кавычках с запятыми/`;`/экранированными кавычками. **Никогда** `str.split(",")` — 31 описание с запятой порвёт строку, тип/описание съедут. Открывать с `newline=""` (требование `csv` к корректной обработке встроенных переводов строк) и `encoding="utf-8"` (кириллица + «—» в описаниях). Заголовок валидируется на точный `CATALOG_COLUMNS` → дрейф колонок (вкл. `working_type`) ловится сразу.

### Резолюция пути каталога (риск #4 — НЕ как `.env`)

Каталог — артефакт **dev-репо**, путешествует с кодом (симлинк в хранилище), а НЕ данные оператора. Поэтому резолюция **от модуля** корректна (в отличие от `.env`, где [[dotenv-usecwd-gotcha]] требует cwd-relative): `Path(__file__).resolve().parents[2] / "development-docs" / "schema-catalog.csv"`. `.resolve()` следует за симлинком хранилища в dev-репо, где `scripts/` и `development-docs/` — реальные соседи. `parents[2]`: `scripts/utils/catalog.py` → `utils` → `scripts` → корень. Путь инъектируется (`path=`) — тесты передают фикстуру; интеграционный AC #5-тест дёргает дефолт (прод-ветка). Отсутствие файла / битый симлинк → fail-loud (AC #8).

### Тестируемость — инъекция пути (как `today_msk` в 1.4)

Единственный шов — параметр `path`. Юнит-тесты пишут мини-CSV-фикстуры в `tmp_path`/`tests/fixtures/` и передают путь явно — детерминично, без зависимости от реального каталога. Один интеграционный тест (AC #5) грузит **реальный** `development-docs/schema-catalog.csv` дефолтным путём — закрывает прод-ветку резолюции и «без потерь». Анти-зависимость — через `ast` (не подстроку: docstring/комментарии содержат слова `pandas` и пр. → ложный красный).

### Latest Tech Information

- **Python 3.13 stdlib `csv`:** `csv.DictReader`/`csv.reader` — RFC4180-совместимы (кавычки, встроенные запятые/переводы строк). Открывать файл с `newline=""` (иначе `csv` неверно обработает встроенные `\n`). `encoding="utf-8"` обязателен (кириллица/«—»). Никаких сторонних парсеров (NFR-6).
