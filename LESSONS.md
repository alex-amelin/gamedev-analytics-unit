# LESSONS — сложности с реальным live-smoke против Logs API

> Заготовка к **ретроспективе Epic 2**. Зафиксированы трудности, вскрытые при первом полноценном
> end-to-end прогоне оркестратора p81 (story 2.7) против РЕАЛЬНОГО Logs API (счётчик `108178404`,
> 2026-05-25). Цель — не отчёт об успехе, а разбор того, что было больно/рискованно и что менять.
>
> **Краткий исход:** после сессии оба источника доезжают (live **hits PASSED 65s** [79 472×41],
> **visits PASSED 80s** [1893×73]); offline 296 / mypy strict 19. Но путь к этому вскрыл несколько
> системных проблем — они ниже.

---

## Главный вывод

**Live-smoke сделал ровно то, ради чего существует, и поймал то, что offline-моки увидеть не могли.**
Три класса дефектов прошли мимо зелёных offline-гейтов (pytest 292, mypy strict) и всплыли только на
реальном API:
1. рассинхрон каталога с реальным контрактом Logs API (имена полей);
2. баг масштабирования записи (производительность на реальном объёме);
3. поля, недоступные на тарифе конкретного счётчика.

Если бы мы доверились «offline зелёный + один раз глазами посмотрели live» — выкатили бы юнит,
который **физически не грузит ни visits, ни hits** на реальных данных.

---

## Сложность 1. «Live-smoke сделан» в DoD ≠ live-smoke реально зелёный

**Что было.** Story 2.7 при сдаче имела DoD #12 «live ... фикстуры освежены» отмеченным `[x]`. По факту:
первый «реальный прогон» — visits упал на `create` (поле тарифа), hits **оборвали вручную** на фазе
скачивания. **Ни один источник не прошёл полный цикл**, фикстуры из реального ответа не освежались
(остались рукописными). DoD-чекбокс и Completion Notes противоречили друг другу.

**Почему больно.** «Прогнали и посмотрели» легко выдать за «зелёно». Полу-прогон создаёт ложную
уверенность, а реальная проверка контракта (правая половина протокола: download→parse→write→verify→
commit→clean) так и осталась неподтверждённой.

**К обсуждению.** Чёткий критерий: **live-DoD закрывается только зелёным end-to-end прогоном**
(оба источника), а не фактом запуска. Освежение фикстур — часть этого критерия, не «по возможности».

## Сложность 2. Каталог разошёлся с реальным контрактом Logs API (имена полей)

**Что было.** Реальный `create_log_request("visits")` отклонил **19** полей с `Unknown field in the
request` (НЕ тариф!). Весь блок маркетинга/атрибуции (`TrafficSource`, `AdvEngine`, `Direct*`,
`SocialNetwork*`, `RecommendationSystem`, `Messenger`, …) был записан без **обязательного префикса
модели атрибуции**: Logs API требует `ym:s:<attribution>TrafficSource` (`first`/`last`/`lastsign`/…).
При этом **hits-часть каталога была корректна** (`ym:pv:lastTrafficSource`) — то есть SSOT был
**сам себе противоречив**: один источник прав, другой нет.

**Почему больно.** Каталог — единственный источник истины (правило проекта: «поле без записи = дефект»,
«не угадывать»). Но его наполняли **без сверки с официальным справочником Logs API**, а сам справочник
**в репо отсутствует** (`yandex-docs/metrika-api/` = только `.gitkeep`; реальные доки лежат в соседнем
`G:\git\directaiq\yandex-docs\metrika-api\`). Offline-тесты ловят только покрытие/типы по маппингу — имя
поля, которого нет в API, мок никогда не отвергнет.

**К обсуждению.**
- Завести в репо официальный справочник полей Logs API (visits/hits) и **тест/процедуру сверки**
  каталога с ним (имена + параметризация `<attribution>`), не только маппинг типов.
- Единая дисциплина атрибуции по источникам (мы выбрали `last`, зеркаля hits) — зафиксировать как
  решение, а не выводить эмпирически на каждом поле.

## Сложность 3. Баг производительности невидим для моков (масштаб реального дня)

**Что было.** `parquet_store.write_partition` (story 2.2, уже `done`) грузил строки построчным
`conn.executemany(INSERT … VALUES)` в колоночный DuckDB. На 3-строчной фикстуре — мгновенно; на
**реальном дне hits (~79 472 строки / ~54 МБ)** — **зависание 25+ минут** (диагностировано по «лог замер
на download, `.tmp` не создан»). Исправлено bulk-загрузкой через транзиентный CSV + нативный `read_csv`.

**Почему больно.** «Корректно на 3 строках» ≠ «работает на реальном дне». Латентный критичный дефект
дожил в `done`-истории до интеграции. Диагностика «почему висит» потребовала ручного расследования
(mtime лога, состояние каталога партиции, чтение кода вставки) — сигнала «я застрял в executemany» не было.

**К обсуждению.**
- Хотя бы один тест/бенч на **реалистичном объёме** (десятки-сотни тыс. строк), не только happy-path
  на мини-фикстуре. Примитивы, которые пишут данные, обязаны иметь проверку масштаба.
- Стори, строящие примитивы в изоляции без live, могут таить баги контракта/масштаба до интеграционной
  истории — учитывать это в оценке «готовности» примитива.

## Сложность 4. Наблюдаемость live-прогона почти нулевая

**Что было.** Вывод pytest буферизовался (через `| tail` и без `-s`), статус опроса/причина падения не
видны в реальном времени. Зависание выглядело как «просто долго грузится» — пришлось лезть в mtime лога и
код. Помог только `-s --log-cli-level=INFO`.

**Почему больно.** Реальный цикл идёт минуты (poll ~30s, download десятки МБ). Без живого лога невозможно
отличить «нормально работает» от «зависло».

**К обсуждению.** Зафиксировать обязательный рецепт live-прогона с живым логом; прод-поверхность
`gdau-logs update` (2.9) должна печатать прогресс по фазам (create/poll/download/write).

## Сложность 5. Live-тест не диагностируется из собственного вывода

**Что было.** Когда прогон убивали на середине, pytest не успевал напечатать секцию FAILURES — **имя
падающего поля терялось**. Чтобы найти все запрещённые/невалидные поля, пришлось писать отдельный
скрипт-разведчик (итеративный `create`, парс ошибки).

**Почему больно.** Полный pytest-цикл — дорогой инструмент для отладки контракта полей (минуты, реальные
запросы). Точечная диагностика быстрее и дешевле.

**К обсуждению.** Лёгкий preflight/`evaluate` (у Logs API есть метод `evaluate`) для проверки списка
полей против API **до** дорогого полного цикла — кандидат в инструментарий (2.9 / отдельная утилита).

## Сложность 6. Реальный API имеет побочные эффекты и оставляет «хвосты»

**Что было.** Прерванные и разведочные прогоны оставили на Метрике несколько неубранных log-запросов
(`55243779`, `55244653`, `55245517`) — `clean` выполняется только на happy-path, а убитые/пробные
прогоны их не чистят. Плюс тратится дневная квота (≤5000 запросов/день) и пишется в реальное хранилище.

**Почему больно.** Live-smoke — не бесплатный и не безсайд-эффектный. Накопление «висящих» запросов и
расход квоты при отладке реальны.

**К обсуждению.** Дисциплина уборки (ручная чистка осиротевших запросов; помнить про квоту); явно
проговорить, что live-smoke пишет в реальное хранилище под `.writer.lock`.

## Сложность 7. Трение инструментария на Windows при live-итерациях

**Что было.** `taskkill /F` ломался под Git Bash (MSYS превращал `/F` в путь `F:/`) — пришлось гасить
зависший процесс через PowerShell `Stop-Process`. `GDAU_DATA_ROOT` не персистентный — передаём инлайн в
каждой команде.

**Почему больно.** Мелочи, но на каждой live-итерации они тормозят и отвлекают от сути.

**К обсуждению.** Зафиксировать рабочие рецепты (запуск live, гашение процесса, передача `GDAU_DATA_ROOT`)
в README/спеке смоука, чтобы не переоткрывать каждый раз.

## Сложность 8. Освежение фикстур из реального ответа легко проскочить (defer 2.3(b))

**Что было.** Реальный wire-формат TSV/массивов подтверждён обоими зелёными live-прогонами **косвенно**
(данные распарсились, колонки ⊆ каталога), но `tests/fixtures/logs_hits_sample.tsv` так и остался
**рукописным** «по образцу» — освежить из реального ответа отложено (остаётся единственным открытым
follow-up).

**Почему больно.** «Освежить фикстуры» — последний шаг, который проще всего пропустить, а именно он
держит offline-моки в согласии с реальностью (иначе контракт снова разойдётся незаметно).

**К обсуждению.** Встроить «сохранить N реальных строк в фикстуру» прямо в процедуру live-прогона, а не
оставлять отдельным благим намерением.

## Что обсудить на ретроспективе (сводка)

1. **Критерий live-DoD:** «зелёный end-to-end оба источника + фикстуры освежены», не «запустили».
2. **Каталог ↔ API:** справочник Logs API в репо + автоматическая/процедурная сверка имён и параметризации.
3. **Масштаб:** тест на реалистичном объёме для всех примитивов записи; пересмотр «готовности» изолированных примитивов.
4. **Наблюдаемость:** обязательный живой лог для live; прогресс по фазам в прод-CLI (2.9).
5. **Диагностика контракта:** лёгкий preflight/`evaluate` до дорогого полного цикла.
6. **Side-effects реального API:** уборка осиротевших запросов, учёт квоты, запись в реальное хранилище.
7. **Windows-рецепты live:** запуск/гашение/окружение — задокументировать.
