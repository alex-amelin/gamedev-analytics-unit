# PRD: Game Dev Analytics Unit
*Рабочее название — подтвердить.*

## 0. Назначение документа

Этот PRD — для меня как владельца и автора, и для двух нижестоящих воркфлоу: архитектуры (`bmad-create-architecture`) и разбивки на эпики/истории (`bmad-create-epics-and-stories`). Он определяет **возможности** v1 «фундамента данных», а не реализацию: тех-механики, дельты против `directaiq` и rejected-alternatives вынесены в `addendum.md` рядом. Документ опирается на готовый продуктовый бриф (`_bmad-output/planning-artifacts/briefs/brief-gamedev-analytics-unit-2026-05-23/`) и его аддендум — он их не дублирует, а доводит до требований. Структура: словарь (§3) задаёт точную лексику; фичи (§4) сгруппированы, функциональные требования (FR) пронумерованы глобально и стабильно; допущения помечены инлайн `[ДОПУЩЕНИЕ]` и собраны в §9.

## 1. Vision

**Game Dev Analytics Unit** — личный инструментарий, который превращает Claude Code в продуктового аналитика моих HTML5-игр на Яндекс.Играх. Игры собирают поведение игроков в Яндекс Метрике; ответы на продуктовые вопросы — как игроки возвращаются, где отваливаются, как проходят сессию — там есть, но через UI Метрики добываются медленно и вручную. **Узкое место — не «нельзя ответить», а темп:** ручной клик-разбор не тянет параллельный поток гипотез сразу по нескольким играм. Именно эта параллельность — центральная боль, которую снимает юнит.

Юнит даёт агенту готовый набор инструментов: вытащить сырые гранулярные данные через **Яндекс Метрика Logs API**, сложить их в локальное переносимое хранилище (Parquet + DuckDB), держать свежими и консистентными и понимать схему — что значит каждая колонка. Logs API отдаёт сырьё асинхронно, большими объёмами и в формате, который надо аккуратно положить, — эту возню юнит берёт на себя, чтобы данные были «под рукой», а не за барьером выгрузки. Дальше я открываю рабочее пространство конкретной игры и просто **задаю вопрос или ставлю задачу на research**: данные, описание проекта и инструменты уже на руках у агента.

Цель первой версии — **фундамент**, а не методология и не «умный аналитик»: минимально достаточный набор инструментов, чтобы данные жили в базе и по ним можно было сесть и спросить. Один набор инструментов обслуживает N игр; данные изолированы по проектам внешними рабочими пространствами. Конечная картина — Claude Code как всегда доступный игровой аналитик, **заземлённый на сырых first-party данных, а не на агрегатах чужого интерфейса**; слой инструментов со временем обрастает переиспользуемыми research-плейбуками (v2+), и цикл «есть вопрос об игре → есть ответ» становится рутинным. Дальше дорабатываем итеративно.

## 2. Целевой пользователь

### 2.1 Primary Persona

**Я — соло-геймдев**, делающий HTML5-игры на Яндекс.Играх и ведущий по ним продуктовую аналитику сам. Технически грамотен (работаю в Claude Code, читаю SQL, держу два связанных репозитория). Времени на ручной клик-разбор в UI Метрики нет; ценю быстрый цикл «возникла гипотеза → проверил → унёс в разработку». Уже прошёл этот путь на родственном проекте `directaiq` (B2B-трафик-аналитика) и переношу проверенную двух-репо-архитектуру на геймдев.

### 2.2 Jobs To Be Done

- **Функциональный:** превратить продуктовый вопрос об игре в ответ на сырых first-party данных — в ту же рабочую сессию, не клик-за-кликом в чужом UI.
- **Функциональный:** держать данные нескольких игр свежими и консистентными малыми усилиями (обновил — не сломал).
- **Контекстный:** вести параллельный ресёрч сразу по нескольким играм, не начиная каждый раз с возни с API/файлами/SQL.
- **Эмоциональный:** возня с API/файлами/SQL на каждый вопрос убивает сам импульс «пойду проверю гипотезу» — инструмент должен этот импульс сохранять, а не гасить.
- **Социальный (вторично):** иметь переносимый, аккуратный фундамент, который не стыдно развивать как продукт.

### 2.3 Non-Users (v1)

- Не для команд/мульти-аккаунта — один оператор, один токен Метрики на игру.
- Не для мобильной аналитики/AppMetrica — только web-Метрика Яндекс.Игр.
- Не для конечных игроков и не для внешних стейкхолдеров — это личный внутренний инструмент.

### 2.4 Key User Journeys

*Один оператор, поэтому журналы лёгкие — это JTBD, доведённые до сцены. FR ссылаются на них по ID.*

- **UJ-1. Развернуть рабочее пространство под новую игру.** Я завёл новую игру на Яндекс.Играх, у меня есть её counter_id и токен Метрики. Запускаю одну команду-инициализатор → получаю внешнее per-game хранилище: структура папок, симлинки на инструменты, `.env`, окружение, пустая БД со схемой, git-репо. Готово к первой выгрузке за минуты, без ручной возни.

- **UJ-2. Обновить данные игры, не сломав базу.** У игры уже есть хранилище. Запускаю обновление за диапазон дат → подтягиваются только отсутствующие дни плюс перезаливается «горячее окно» последних N дней (доезжающие данные). Сверка числа строк с источником проходит; повторный запуск идемпотентен — база не дублируется и не ломается.

- **UJ-3. Спросить и получить ответ в ту же сессию.** Открываю рабочее пространство игры в Claude Code. У агента на руках: данные (Parquet+DuckDB), каталог схемы (что значит каждая колонка), описание проекта и MCP-инструмент для SQL. Задаю вопрос («как retention по дням за последние 30 дней?») или ставлю задачу на research → агент пишет и выполняет SQL через MCP, возвращает ответ. Я уношу решение в разработку.

## 3. Глоссарий

*Нижестоящие воркфлоу и читатели используют термины ровно так. FR, UJ и SM используют их дословно; синоним где-либо в PRD — нарушение дисциплины.*

- **Юнит (Game Dev Analytics Unit)** — весь продукт: dev-репо инструментов + механизм внешних per-game хранилищ.
- **Dev-репо** — этот репозиторий: инструменты, скрипты, миграции, документация, каталог схемы. Развивается как сам продукт. Один на все игры.
- **Внешнее хранилище** (per-game storage) — отдельная папка под конкретную игру: БД (Parquet+DuckDB), конфиги, `.env`, рабочая папка. Связана с dev-репо симлинками. По одному на игру.
- **Logs API** — Яндекс Метрика Logs API: асинхронная выгрузка сырых гранулярных данных (создать запрос → опросить статус → скачать части → очистить запрос).
- **Источник** (source) — тип данных Logs API: `visits` (визиты) или `hits` (хиты/просмотры внутри сессии). Связь visits↔hits — через `watchIDs`/`watchID`.
- **watchIDs / watchID** — поле связи источников: `visits.watchIDs` (массив id хитов визита) ↔ `hits.watchID` (id отдельного хита). Через него хиты группируются в визиты.
- **Заданный список полей** — curated-набор полей Logs API (visits/hits), который называет владелец. Тянем только его, не «всё подряд».
- **Сырьё** (raw layer) — данные как есть из Logs API, строками, верно источнику. Лежат в Parquet.
- **Parquet-партиция** — файл сырья за один день. Партиционирование по дню — единица загрузки, перезалива и идемпотентности.
- **Рабочий слой** (working layer) — DuckDB поверх Parquet, где заданные поля типизированы. То, к чему обращается SQL агента.
- **Типизация** — приведение заданных полей к типам в рабочем слое через `TRY_CAST` (битая ячейка → `NULL` + лог, не падение).
- **Инкремент** — догрузка только отсутствующих дней.
- **Hot-window** — окно последних N дней, перезаливаемое при каждом обновлении ради доезжающих данных.
- **Перезалив дня** — идемпотентная перезапись Parquet-партиции одного дня; остальные дни не трогаются.
- **Сверка строк** — сравнение числа строк (источник vs БД) за день. Расхождение → жёсткий fail.
- **Мета-таблица состояния** — чекпойнт загрузок: какие дни загружены, когда, сколько строк. Источник истины для инкремента и crash-recovery.
- **Каталог схемы** — собственный справочник: что значит каждая колонка заданного списка. Единый источник для DDL и навигации агента.
- **Описание проекта** (project context) — файл контекста конкретной игры, чтобы агент знал, с чем работает.
- **MCP `duckdb_query`** — MCP-сервер, дающий Claude Code схему, контекст БД и выполнение SQL. Переиспользуется из `directaiq` с доработкой.
- **Init-команда** — команда-инициализатор, разворачивающая внешнее хранилище и связывающая его симлинками с dev-репо.
- **`directaiq`** — родственный репозиторий-референс (B2B-трафик-аналитика). Источник вендоримых примитивов и архитектурного паттерна, НЕ источник продуктовой логики.

## 4. Фичи

*Каждая подсекция — связная фича: описание поведения, затем вложенные FR. FR пронумерованы глобально (FR-1…FR-N) для стабильных ссылок из нижестоящих артефактов. Тех-механики и дельты против `directaiq` — в `addendum.md`.*

### 4.1 Приём данных через Logs API

**Description:** Полный цикл выгрузки за заданный диапазон дат: создать запрос → опросить статус (poll до `processed`) → скачать части → очистить запрос (освободить квоту). Тянем **только заданный список полей** для выбранного источника (`visits`/`hits`), не curated-подмножество `directaiq` и не «всё». Низкоуровневый клиент Logs API вендорим из `directaiq` как примитив; оркестрацию под Parquet/DuckDB пишем свою. Реализует UJ-2.

**Functional Requirements:**

#### FR-1: Оркестрация полного цикла Logs API

Юнит может за указанный диапазон дат и источник провести полный цикл выгрузки: создать лог-запрос, дождаться готовности, скачать все части, очистить запрос.

**Consequences (testable):**
- Команда с `--date1`, `--date2`, `--source {visits|hits}` доводит данные диапазона до сырьевого слоя или завершается ненулевым кодом с понятной причиной.
- Статус опрашивается до `processed`; при `processing_failed`/`canceled` — fail с диагностикой, без молчаливого пропуска.
- После успешной персистенции лог-запрос на стороне Метрики очищается (`clean`), чтобы не копить квоту.
- Соблюдаются ограничения Logs API: rate limit (≤30 req/s, ≤5000 req/day) и retry с backoff на 429/500/502/503 — поведение наследуется от вендоренного клиента; проверяется тем, что выгрузка многодневного диапазона не упирается в rate-limit-ошибки.

#### FR-2: Тянуть только заданный список полей

Юнит выгружает ровно тот набор полей, который задан владельцем для источника, — параметризуемый, не захардкоженный «на всё».

**Consequences (testable):**
- Список полей visits и список полей hits заданы в одном месте (конфиг/каталог) и используются и при выгрузке, и при типизации, и в каталоге схемы — без рассинхрона.
- Добавление/удаление поля в заданном списке не требует правок в нескольких местах кода.

**Notes:** `[NOTE FOR PM]` Точный список полей visits/hits ещё не зафиксирован — см. Open Question #1.

#### FR-3: Поддержка источников visits и hits

Юнит выгружает оба источника Logs API — `visits` (визиты/сессии) и `hits` (хиты внутри сессии). Решение владельца (2026-05-23): оба в v1 — `visits` дают готовые сессионные/retention-поля (`isNewUser`, `visitDuration`, `bounce`, `pageViews`, UTM), `hits` дают путь игрока внутри сессии (последовательность URL).

**Consequences (testable):**
- Каждый источник грузится и хранится независимо (свои партиции, своя типизация).
- Связь visits↔hits доступна через `visits.watchIDs` ↔ `hits.watchID` (нужна для группировки хитов в визиты).

#### FR-4: Креды и окружение

Юнит читает токен и счётчик Метрики из окружения внешнего хранилища через тонкий env-ридер.

**Consequences (testable):**
- Читаются `YANDEX_METRICA_TOKEN` и `YANDEX_METRICA_COUNTER_ID`; отсутствие любого → понятная ошибка до сетевых вызовов.
- Нет fallback на `YANDEX_DIRECT_TOKEN` (в отличие от `directaiq` — у нас только Метрика).
- `.env` живёт во внешнем per-game хранилище, не в dev-репо.

#### FR-5: Безопасная граница диапазона дат

Юнит не запрашивает у Logs API данные за сегодня/будущее (Logs API требует `date2 < today`).

**Consequences (testable):**
- `date2` зажимается на «вчера по МСК»; запрос за более позднюю дату не падает, а clamp'ится с записью в лог.

### 4.2 Хранение — два слоя

**Description:** Сырьё ложится в **Parquet, партиционированный по дню**, как есть (строками) — верно источнику, ничего не теряем на кривом TSV Метрики. Поверх — **рабочий слой DuckDB**, где заданные поля типизируются. Сильное сжатие, ноль сервера, переносится между Windows и Linux как папка/файл. Реализует UJ-2, UJ-3.

**Functional Requirements:**

#### FR-6: Сырьевой слой — Parquet по дням, как есть

Юнит сохраняет сырьё Logs API в Parquet, партиционированный по дню, строками, без приведения типов.

**Consequences (testable):**
- Одна Parquet-партиция = один день одного источника.
- Значения хранятся как пришли от Метрики (строки) — на этом слое нет CAST и нет молчаливого усечения.
- Партиция самодостаточна: её можно перезаписать, не затрагивая остальные дни.
- Изменение заданного списка полей (FR-2) не мигрирует уже записанные партиции на лету: новые поля появляются только в партициях, записанных после изменения. Приведение исторического сырья к новому списку — через явный перезалив затронутых дней (FR-10), а не через инкрементальную эволюцию существующих Parquet-файлов (см. §5 Non-Goals).

#### FR-7: Рабочий слой — DuckDB с типизацией через TRY_CAST

Юнит даёт рабочий слой DuckDB поверх Parquet, где заданные поля приведены к типам через `TRY_CAST`.

**Consequences (testable):**
- Битая/непарсимая ячейка → `NULL` + запись в лог; загрузка дня не падает из-за одной ячейки.
- Поля для арифметики дат и числовых сравнений (retention/поведение) доступны в корректных типах, а не как строки.
- ID-поля (`visitID`, `clientID`, `watchID`) типизируются как `HUGEINT`: значения Metrica ID превышают 2^63 и не помещаются в `BIGINT` (см. addendum §C).
- Способ материализации рабочего слоя (DuckDB-view'ы поверх Parquet против материализованных таблиц) — решение фазы архитектуры (Open Question #3), не фиксируется в PRD.

#### FR-8: Переносимость Windows ↔ Linux

Хранилище переносится копированием папки между Windows и Linux и продолжает работать.

**Consequences (testable):**
- Ноль серверных процессов: только файлы (Parquet) + встроенный движок (DuckDB).
- Объём сырья за ~30 дней замеряется на первой реальной выгрузке и фиксируется как базовый ориентир (ожидание — единицы–десятки МБ на игру за месяц благодаря колоночному сжатию Parquet). Конкретный «порог тревоги по объёму» (триггер пересмотра DuckDB→ClickHouse) задаётся после первого замера (Open Question #5).

### 4.3 Обновление и инкремент

**Description:** Догрузка по дню; идемпотентный перезалив отдельного дня, не ломающий остальные; перезалив «горячего окна» последних N дней для доезжающих данных. Реализует UJ-2.

**Functional Requirements:**

#### FR-9: Инкрементальная догрузка по дню

Юнит при обновлении грузит только отсутствующие дни диапазона.

**Consequences (testable):**
- Уже загруженные дни (по мета-таблице состояния, FR-12) пропускаются.
- Источник истины «что загружено» — мета-таблица + фактическое наличие партиции, не только `SELECT DISTINCT date` (осознанное отличие от `directaiq`).

#### FR-10: Идемпотентный перезалив одного дня

Юнит может перезалить отдельный день, перезаписав только его Parquet-партицию, не трогая другие дни.

**Consequences (testable):**
- Перезалив дня = перезапись одной партиции; данные других дней не дублируются и не теряются.
- Повторный запуск того же дня даёт тот же результат (идемпотентность), без `DROP` всей таблицы (осознанное отличие от `directaiq`, где `--force` дропал таблицу целиком).

#### FR-11: Hot-window — перезалив последних N дней

Юнит при каждом обновлении перезаливает последние N дней, чтобы подтянуть доезжающие данные.

**Consequences (testable):**
- N — конфигурируемый параметр; перезалив окна опирается на per-day идемпотентность (FR-10).
- N = 3 дня по умолчанию (решение владельца 2026-05-23). Пересмотреть, если реальный доезд данных Метрики на первой выгрузке покажет иное. (`directaiq` держит 21-дневное окно для атрибуции Метрики — но это доезжающая атрибуция, не сырые визиты/хиты.)

### 4.4 Консистентность и защита базы

**Description:** «Не сломать базу» — это требование первого класса. Жёсткая сверка строк, атомарная запись дня, чекпойнт состояния, дисциплина одного писателя. Реализует UJ-2.

**Functional Requirements:**

#### FR-12: Мета-таблица состояния (чекпойнт)

Юнит ведёт мета-таблицу состояния загрузок: какие дни/источники загружены, когда, сколько строк.

**Consequences (testable):**
- На старте обновления состояние реконсилируется против фактического `count()`/наличия партиций (а не доверяется слепо).
- День считается загруженным только после успешной персистенции и сверки (FR-13).
- **Политика разрешения конфликта мета×факт:** источник истины — фактическая Parquet-партиция, не запись мета-таблицы. День засчитывается загруженным только при выполнении всех трёх условий: партиция существует, сверка строк (FR-13) сошлась, запись мета-таблицы согласуется с фактом. При любом расхождении (партиции нет / сверка не сошлась / мета расходится с фактом) день помечается незагруженным и переливается (FR-10), а мета-запись приводится к факту.

#### FR-13: Сверка строк — жёсткий fail

Юнит сверяет число строк за день (источник Logs API vs то, что легло в хранилище), и при расхождении завершается с ошибкой.

**Consequences (testable):**
- Расхождение → исключение/ненулевой код, а НЕ warning (осознанное отличие от `directaiq`, где это warning с `exit_code=0`).
- День с несошедшейся сверкой не помечается загруженным в мета-таблице (FR-12).

#### FR-14: Атомарная запись дня (нет частичных дней)

Юнит пишет день атомарно: сначала во временный файл, затем атомарный rename в финальную партицию.

**Consequences (testable):**
- Крэш посреди записи дня не оставляет частичную партицию в хранилище → нет «частично загруженных» дней, которые при перезапуске считались бы готовыми (осознанное отличие от `directaiq`).
- После сбоя повторный запуск корректно дозагружает прерванный день.

#### FR-15: Дисциплина одного писателя

Юнит гарантирует единственного писателя через process-level лок: второй конкурентный запуск на запись завершается fail-fast, а не пишет в ту же БД.

**Consequences (testable):**
- Операция записи берёт эксклюзивный лок (файловый, на уровне внешнего хранилища); запуск второго писателя при удержанном локе немедленно завершается понятной ошибкой — без ожидания и без записи.
- Чтение (SQL агента через MCP, FR-17) лока не требует и им не блокируется.
- Запись Parquet по дням дополнительно снижает контеншн на DuckDB-локе; одиночный писатель — естественное состояние для одного оператора.
- `[NON-GOAL for MVP]` Тяжёлая FIFO-очередь/воркер/disk-guard уровня `directaiq` — вне v1; достаточно fail-fast лока. Опциональная лёгкая проверка свободного места допустима.

### 4.5 Каталог схемы

**Description:** Живой собственный справочник: что значит каждая колонка заданного списка Logs API. Без него знание о данных — что значит конкретное поле, как устроена схема — держится в голове и быстро забывается; каталог делает его явным и переносимым. Единый источник и для DDL рабочего слоя, и для навигации агента. Не вендорим чужой CSV. Реализует UJ-3.

**Functional Requirements:**

#### FR-16: Собственный каталог полей

Юнит ведёт каталог в машиночитаемом структурированном формате: для каждой колонки заданного списка — имя, источник (visits/hits), тип в рабочем слое, человекочитаемое значение.

**Consequences (testable):**
- Каталог хранится в структурированном, программно парсимом формате (напр. YAML/CSV/JSON), а не как свободная проза. Это машинный единый источник: из него генерируются типы рабочего слоя (FR-7), DDL и семантика колонок, отдаваемая агенту через MCP (FR-18).
- Поле в заданном списке (FR-2) без записи в каталоге — это дефект (каталог покрывает весь список).
- Человекочитаемая документация-проза (по образцу `data-architecture.md` из `directaiq`) при желании генерируется из каталога, а не ведётся параллельно — единый источник истины, без рассинхрона.

### 4.6 Доступ агента к данным (MCP)

**Description:** MCP-сервер `duckdb_query` даёт Claude Code схему, контекст БД и выполнение SQL. Переиспользуем из `directaiq` с лёгкой доработкой под нашу схему. Реализует UJ-3.

**Functional Requirements:**

#### FR-17: Выполнение SQL через MCP

Агент может выполнять SQL-запросы к рабочему слою через MCP `duckdb_query` и получать результат.

**Consequences (testable):**
- Доступен единый инструмент `duckdb_query(query, format, limit)` с сервисными командами `--tables`, `--schema [TABLE]`, `--context`, `--sample TABLE [N]`, `--export` (наследуется от инструмента `directaiq`).
- Произвольный SQL к рабочему слою выполняется и возвращает результат в запрошенном формате (json/markdown/csv).
- Большие результаты (свыше порога, по умолчанию 500 строк) авто-экспортируются в файл вместо переполнения ответа — поведение наследуется; проверяется запросом, отдающим больше порога.

#### FR-18: Контекст и схема для агента

MCP отдаёт агенту контекст БД и схему так, чтобы агент мог ориентироваться без ручных подсказок.

**Consequences (testable):**
- Контекст включает таблицы/view'ы, типы, row counts и диапазоны дат.
- Семантика колонок согласована с каталогом схемы (FR-16); специфичная для `directaiq` семантика (НДС/деньги Директа) убрана/заменена под нашу схему.
- Доработка MCP в v1 — **лёгкая** (решение владельца 2026-05-23): обновить таблицу семантик колонок (`_COST_COLUMN_SEMANTICS`) под нашу схему + убрать Direct/НДС/goal-плейсхолдеры; интерфейс единого инструмента `duckdb_query(query, format, limit)` сохраняется. Гейм-специфичный контекст/плейсхолдеры (напр. окна retention) — кандидаты на v2.

### 4.7 Двух-репо и инициализация рабочего пространства

**Description:** Архитектура из двух частей: dev-репо (инструменты) ↔ внешнее per-game хранилище (БД+конфиги+рабочая папка), связь симлинками, разворачивание одной командой. Плюс файл описания проекта, чтобы агент знал игру. Реализует UJ-1, UJ-3.

**Functional Requirements:**

#### FR-19: Init-команда разворачивает внешнее хранилище

Юнит одной командой создаёт новое внешнее per-game хранилище из шаблона и связывает его с dev-репо.

**Consequences (testable):**
- Команда выполняет по шагам: проверка имени → копирование шаблона хранилища → симлинки по контракту → генерация `.env` → подготовка окружения/зависимостей → создание БД + схема → `git init`.
- На выходе хранилище готово к первой выгрузке (FR-1) без ручных правок, кроме внесения токена/счётчика в `.env`.
- Имя занято/папка существует → fail-loud, без перезаписи чужих данных.

#### FR-20: Контракт симлинков (dev-репо → хранилище)

Внешнее хранилище ссылается на инструменты dev-репо симлинками по явному контракту, так что обновления инструментов видны всем играм.

**Consequences (testable):**
- Набор симлинкуемых путей задан декларативно (контракт), а не разбросан по коду init.
- Обновление инструмента/команды/доки в dev-репо автоматически доступно во всех хранилищах без копирования.

#### FR-21: Описание проекта (project context)

Каждое хранилище содержит файл описания конкретной игры, доступный агенту.

**Consequences (testable):**
- Файл создаётся из шаблона при init (FR-19) и заполняется владельцем.
- Агент имеет доступ к описанию проекта в рабочей сессии (UJ-3).

## 5. Non-Goals (явно)

- **Не методология и не «умный аналитик».** Никаких плейбуков, регламентов поведения агента, гайдов по research в v1. Премиса: есть данные + контекст + инструменты → дальше просто спросить.
- **Не сами аналитические ответы.** Retention, воронки, проверка гипотез — это слой использования во внешнем репо, не часть юнита.
- **Не бизнес-витрины.** Агрегаты, дашборды, модели атрибуции, материализованные витрины — вне v1. Типизация заданных полей — это корректные типы для запросов, не витрина.
- **Не мобильная аналитика.** AppMetrica и нативные приложения — вне; только web-Метрика Яндекс.Игр.
- **Не мульти-аккаунт и не тяжёлая инфраструктура.** Один оператор, один токен на игру; без расписаний/cron, без очередей/воркеров уровня `directaiq`.
- **Юнит не становится `directaiq`.** `directaiq` — референс и источник примитивов; мы не тащим его продуктовую логику (трафик-аналитика, НДС/деньги Директа, атрибуционные таблицы).
- **Не инкрементальная эволюция схемы Parquet.** Смена заданного списка полей не переписывает исторические партиции автоматически; приведение истории к новому списку — это явный перезалив затронутых дней (FR-6, FR-10), а не миграция существующих Parquet-файлов на лету.

## 6. MVP Scope

### 6.1 In Scope

- Приём через Logs API по заданному списку полей, источники visits + hits (FR-1…FR-5).
- Два слоя хранения: Parquet-сырьё по дням + рабочий слой DuckDB с `TRY_CAST` (FR-6…FR-8).
- Инкремент по дню, идемпотентный перезалив дня, hot-window (FR-9…FR-11).
- Консистентность: мета-таблица состояния, жёсткая сверка строк, атомарная запись, дисциплина одного писателя (FR-12…FR-15).
- Собственный каталог схемы (FR-16).
- MCP `duckdb_query` с доработкой под нашу схему (FR-17, FR-18).
- Двух-репо, init-команда, контракт симлинков, описание проекта (FR-19…FR-21).

### 6.2 Out of Scope for MVP

- Аналитические плейбуки и регламенты агента → v2+. `[NOTE FOR PM]` Эмоционально нагруженный пункт vision'а — революцировать, когда фундамент стабилен.
- Бизнес-витрины/агрегаты/дашборды/атрибуция — отдельный слой, не фундамент.
- AppMetrica/мобилка — другой источник данных.
- Мульти-аккаунт, cron/расписания, тяжёлая очередь/воркер — избыточно для одного оператора.
- Миграция DuckDB → ClickHouse — не дверь в один конец; запускается только если конкретная игра «взорвётся» по объёму (см. Open Question #5).

## 7. Success Metrics

*Личный инструмент — метрики калиброваны как сигналы готовности v1, а не как продуктовые KPI.*

**Primary**
- **SM-1 (Главный сигнал):** Данные за последние ~30 дней одной игры лежат в хранилище; я открываю её рабочее пространство, задаю вопрос через Claude/SQL и получаю ответ в ту же сессию. Validates FR-1, FR-6, FR-7, FR-17, FR-18, FR-21.

**Secondary**
- **SM-2 (Безопасное обновление):** Повторный запуск обновления идемпотентен — базу не ломает; сверка строк с источником проходит. Validates FR-9, FR-10, FR-11, FR-13, FR-14.
- **SM-3 (Один шаг до новой игры):** Развернуть рабочее пространство под новую игру — одна команда. Validates FR-19, FR-20.
- **SM-4 (Лёгкость и переносимость):** Объём сырья за ~30 дней замерен и зафиксирован как базовый ориентир; папку хранилища можно скопировать между Windows и Linux, и она продолжает работать (та же выгрузка читается, MCP-запросы выполняются) без правок путей и без перезагрузки данных. Validates FR-6, FR-8.

**Counter-metrics (не оптимизировать)**
- **SM-C1 (Полнота полей):** НЕ тянуть «всё подряд» ради охвата. Заданный список полей остаётся curated; рост числа полей без продуктовой нужды — антисигнал. Counterbalances SM-1.
- **SM-C2 (Соблазн витрин):** НЕ строить агрегаты/витрины ради «красивых метрик». Юнит остаётся фундаментом данных; появление витрин в v1 — выход за scope. Counterbalances SM-1.
