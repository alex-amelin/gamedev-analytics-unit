# Поверхность gdau-logs

Компонент: набор команд, которыми оператор вручную водит Logs API.
Реализация: `scripts/tools/logs_api_cli.py` (console-команда `gdau-logs`).

## Что делает

Это «пульт» для работы с выгрузками Метрики из командной строки. Каждая кнопка на
пульте — отдельная подкоманда, и все они неинтерактивные (ничего не спрашивают по
ходу, всё задаётся аргументами сразу):

- **`create`** — заказать выгрузку за период по источнику (`visits` или `hits`);
- **`evaluate`** — заранее прикинуть, потянет ли Метрика такой заказ, не тратя квоту
  на сам заказ;
- **`status`** — узнать, готов ли уже конкретный заказ и из скольких частей состоит;
- **`download`** — скачать готовые части заказа в файлы;
- **`clean`** — убрать за собой готовый заказ, освободив квоту;
- **`list`** — показать список всех заказов;
- **`info`** — справочно показать данные счётчика (быстрая проверка, что доступ есть).

На вход — подкоманда и её аргументы (даты, источник, номер запроса, путь сохранения).
На выход — результат **обычным человеческим текстом**: подписи вида «Поле: значение»
для одного объекта и простая выровненная таблица для списка. Машинных форматов
(json/csv) здесь нет намеренно: текст одинаково понятен и агенту-помощнику, и
человеку, а лишний формат-движок только усложнил бы простой инструмент.

## Зачем нужен

Это **канал действий** юнита, рассчитанный в первую очередь на агента. Возможности
не прячутся в один большой «волшебный» запуск, а выставлены отдельными
командами-кирпичиками — так агент (или человек) может собрать из них любой сценарий
работы с API под конкретную задачу: заказать → подождать → скачать → прибраться.

Сами по себе команды тонкие: они не повторяют логику работы с сетью, а опираются на
уже готовые части юнита и просто аккуратно их сшивают. За счёт этого три важных
правила выполняются автоматически, без участия оператора:

- **какие поля выгружать — решает каталог**, а не оператор руками. Список полей
  всегда берётся из единого справочника (см. [`catalog.md`](catalog.md)), поэтому
  заказ не разъезжается с тем, что мы умеем раскладывать.
- **дата конца сама зажимается на «вчера по МСК»** — Метрика не отдаёт сегодняшний
  день, и команда не даст случайно его заказать (а на бессмысленном диапазоне сразу
  понятно падает, ещё до обращения к сети).
- **доступ берётся из окружения** — токен и номер счётчика читаются из `.env`
  (см. [`creds.md`](creds.md)); отдельного флага для счётчика нет, чтобы не плодить
  второй путь с секретами.

## Контракт с другими

- **Что принимает.** Подкоманду и аргументы к ней. Поля выгрузки и их типы сам не
  придумывает — берёт из каталога. Даты принимает строками строго вида `YYYY-MM-DD`.
- **На что опирается.** На каталог полей (что заказывать), на примитив дат (зажать
  конец периода и проверить его), на чтение кредов (доступ) и на клиента Logs API
  (единственная «телефонная трубка» к сети). Устойчивость к лимитам и повторные
  попытки — целиком забота клиента; CLI их не дублирует.
- **Что обещает.** Любая команда либо делает своё дело и завершается **успехом с
  кодом 0**, напечатав результат понятным текстом, либо **останавливается с понятным
  сообщением и ненулевым кодом**. Полупустой или ранний результат за успех не
  выдаётся: скачать можно только готовый заказ, а уже существующий файл команда молча
  не перезатирает. Голый вызов без подкоманды или неверный источник — это сразу
  подсказка по использованию, а не невнятная ошибка. Секреты в вывод и логи не
  попадают.

## Границы

Здесь живут только **примитивы** жизненного цикла — тонкие обёртки над клиентом
Метрики. Полноценный приём за диапазон одной командой (`update`/`load`), складывание
данных в хранилище с проверкой целостности и под «замком», раскладка скачанного по
таблицам с типами — это **отдельная работа Эпика 2**, не этот компонент. Команда
`download` сохраняет сырые файлы туда, куда указал оператор (по умолчанию — в текущий
каталог), и в рабочее хранилище игры ничего сама не пишет.
