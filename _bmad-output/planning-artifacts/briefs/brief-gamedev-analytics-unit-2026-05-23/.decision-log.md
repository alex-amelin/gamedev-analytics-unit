# Decision Log — Game Dev Analytics Unit (brief)

Канонический журнал решений и контекста сессии брифа. 2026-05-23.

## Намерение
- **Create** — новый продуктовый бриф. Артефактов ранее не было (`docs/`, `_bmad-output/` пусты).

## Зафиксированные решения и контекст
1. **Что за продукт.** Game Dev Analytics Unit — личный инструментарий + рабочие процессы, чтобы **Claude Code** делал продуктовую аналитику HTML5-игр на **Яндекс.Играх** по данным **Яндекс Метрика Logs API**. Пользователь приходит с вопросом/задачей → агент с инструментами на руках лезет в данные, проводит research, отвечает → пользователь уносит решение в разработку.
2. **Прошлый техсрез выброшен.** Ранее планировался срез Метрика→ClickHouse→MCP — делаем заново с чистого листа, к нему не привязываемся.
3. **Архитектура — двух-репозиторная (как в `G:\git\directaiq`).** Dev-репо = инструменты/скрипты/доки (развиваются как сам проект). Внешнее per-project хранилище = БД + конфиги + рабочая папка (специфично для каждой игры). Связь — симлинки, инициализация командой-инициализатором. `directaiq` — **архитектурный референс**, НО он про traffic-аналитику B2B-рекламы; наш продукт — продуктовая аналитика поведения игроков (внутренний продукт: как играют, как возвращаются, retention).
4. **Агент — Claude Code** (не отдельный кастомный агент).
5. **v1 = минимальный фундамент данных.** Набор минимально необходимых инструментов: получить данные → обновить → проверить консистентность → не сломать базу → описать схему (что значит каждая колонка) + описание проекта. БЕЗ методологий и регламентов поведения агента. Премиса: если у агента есть хранилище, описание проекта и инструменты работы с данными/источниками — дальше остаётся только спросить или дать задачу на research. Следующие версии = улучшение этого процесса.
6. **Аналитические ответы — вне v1.** Сами ответы на вопросы («почему просел retention») — слой использования во внешнем репо, не входит в v1.
7. **Сигнал готовности v1.** Данные за последние ~30 дней лежат в базе; можно сесть и через Claude/SQL задать по ним вопрос и получить ответ.
8. **Стиль работы.** Личный проект, цель — быстро завести первую рабочую версию и итеративно дорабатывать. Быстрый путь (черновик с `[ДОПУЩЕНИЕ]`).

9. **Хранилище — DuckDB + Parquet** (выбор пользователя, 2026-05-23). Сырьё → партиционированный по дням Parquet (сжатие); DuckDB — движок запросов поверх. Обоснование: покрывает требования (сырые гранулярные данные, мало места, переносимость Win↔Linux, ноль сервера, быстрый старт, проверено в directaiq); объём личной игры на Яндекс.Играх — в зоне комфорта DuckDB (ГБ–десятки ГБ, не ТБ). Не дверь в один конец: тот же Parquet при необходимости заливается в ClickHouse. Отвергнут ClickHouse (сервер/WSL-Docker трение под Windows, хуже переносимость) — оправдан только при реально массивных объёмах. Честный неизвестный: фактический вес данных; порог пересмотра — если конкретная игра взорвётся.

10. **MCP `duckdb_query` — в v1** (решение 2026-05-23). Переиспользуем из directaiq с лёгкой доработкой; даёт Claude Code схему/контекст/SQL. (Отменяет прежнее допущение «MCP в v2».)
11. **Поля — только заданный список** (решение 2026-05-23). Тянем не «всё подряд», а curated-набор, который называет владелец. Конкретный список — деталь реализации, фиксируется при сборке и в каталоге.
12. **Типизация — два слоя** (рекомендация Mary, принята владельцем 2026-05-23). Сырой Parquet — как есть, строками (верно источнику, защита от молчаливого усечения на кривом TSV). Рабочий слой DuckDB — типизация заданных полей через `TRY_CAST` (битая ячейка → NULL + лог, не валит загрузку дня). Обоснование: retention/поведение = арифметика дат + числовые сравнения; «всё String» была защитой от неизвестных полей, а у нас поля известны. Двухслойка снимает оба минуса (эргономика vs молчаливая потеря).
13. **Собственный каталог полей — в v1** (подтверждено 2026-05-23). Описываем что значит каждая колонка; единый источник для DDL и навигации агента (не вендорим чужой CSV).
14. **Уточнение боли** (2026-05-23). НЕ «через Метрику невозможно». Можно, но долго и вручную; ключевая боль — это не масштабируется на **параллельный поток ресёрча по нескольким играм**. Усиливает мульти-проектную линию (внешние per-game пространства, init-команда).
15. **Addendum заведён** (2026-05-23). Техническая глубина вынесена в `addendum.md` как вход для PRD: заземлено на directaiq + наши осознанные отличия. Сверка с кодом directaiq подтвердила:
    - идемпотентность directaiq = skip-or-full-drop (нет per-day перезалива) → у нас per-day через Parquet-партиции;
    - сверка строк в directaiq = **warning** (`p81_load_logs.py:783`, память верна) → у нас **жёсткий fail**;
    - типизация directaiq = строгий CAST роняет день → у нас сырой Parquet + `TRY_CAST`;
    - hot-window в directaiq нет (только clamp date2) → у нас явное окно N дней;
    - crash-recovery directaiq оставляет частичный день → у нас атомарный temp→rename.
