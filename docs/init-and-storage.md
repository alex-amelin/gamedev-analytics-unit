# Init и хранилище игры

Этот компонент отвечает за **разворачивание рабочего пространства одной игры** (`gdau-init`):
как из общего dev-репо родить отдельную папку игры, связать её с инструментами юнита и
подготовить к приёму данных. Документ пополняется по ходу Epic 4: сейчас описан **симлинк-контракт
и preflight** (история 4.1); шаблон хранилища и файл описания игры добавит 4.2, полный
разворот командой `gdau-init` — 4.3.

## Симлинк-контракт и preflight

### Что делает

При разворачивании игры этот блок **связывает её папку с инструментами юнита по явному списку**.
Список лежит в одном файле — `templates/paths-to-symlink.csv` (две колонки: `path` — что
связываем, `comment` — зачем). Связываемое — это код юнита, каталог схемы, справочники Logs API
и манифест проекта: они приходят в папку игры **ссылками** на dev-репо, а не копиями.

Перед тем как создавать ссылки, блок **проверяет, что система вообще умеет их создавать**. На
Windows для этого нужен включённый «Режим разработчика» (Developer Mode); если его нет — блок
**сразу останавливается с понятной инструкцией**, как его включить, и **ничего не разворачивает
наполовину**. На Linux ссылки создаются всегда, проверка там просто проходит.

Если всё в порядке, блок создаёт ссылки по списку. Уже существующая ссылка при повторном
развороте **обновляется** (а не вызывает ошибку). Если в списке указано то, чего в dev-репо нет,
— блок останавливается с понятной ошибкой. Если связать удалось только часть, а на чём-то
сорвалось — уже созданные ссылки **снимаются**, чтобы не осталось «полусвязанного» состояния.

### Зачем нужен

Чтобы **одно исправление инструмента видели сразу все игры**. Код, каталог схемы и справочники
живут в одном месте — dev-репо. Игры на них **ссылаются**, поэтому фикс бага или правка каталога
в dev-репо (после `git pull`) автоматически долетает до каждой игры — без копирования, без
миграций, без риска, что у разных игр разойдётся инструмент.

И чтобы **папку игры можно было перенести** на другую машину или другую ОС простым копированием.
Ссылки заданы **относительно** (не «жёстким» абсолютным путём вроде `G:\git\...`), поэтому пока
рядом переносят и dev-репо, и папку игры, ссылки не рвутся. Это и есть переносимость Windows↔Linux,
которую требует проект.

Наконец — **безопасность данных**. Если на месте будущей ссылки лежит реальный файл или папка
(а не ссылка), блок их **не удаляет**, а останавливается: данные игры не должны пострадать от
разворота.

### Контракт с другими

- **Список ссылок — единый источник истины.** Он правится в одном файле (`paths-to-symlink.csv`),
  а не разбросан по коду. Добавить новую связь — значит дописать строку в этот файл.
- **Что сейчас в списке.** Существующие стабильные цели dev-репо: код юнита (`scripts`), каталог
  схемы (`development-docs`), справочники Logs API (`yandex-docs`), манифест проекта
  (`pyproject.toml`). Конфиг MCP (`.mcp.json`) и настройки агента (`.claude/…`) добавятся в список
  позже, когда эти артефакты появятся и состав разворота финализируется (история 4.3) — раньше
  времени их в списке нет, иначе разворот падал бы на отсутствующей цели.
- **Поведение на краях** (обещания блока): пустой/битый список или дубли → понятная ошибка;
  цель из списка отсутствует в dev-репо → понятная ошибка ещё **до** создания первой ссылки;
  существующая ссылка → обновляется, не падает; реальный файл/папка на месте ссылки → остановка,
  данные не удаляются; платформа не умеет ссылки → остановка с инструкцией Developer Mode; сбой
  посреди разворота → откат уже созданных ссылок.
- **Кто это вызывает.** Единственный потребитель — команда `gdau-init` (история 4.3). Она зовёт
  блок на шаге «ссылки», передавая ему **корень dev-репо и корень папки игры** (сам блок их не
  ищет — это делает 4.3). Блок ни в сеть не ходит, ни базу `gdau.duckdb` не открывает, ни данные
  не пишет — только работает с файлами: читает список, проверяет способность, создаёт ссылки.

### Границы (что делают другие истории Epic 4)

- **Копирование шаблона хранилища** (стартовая структура папок игры) и **файл описания игры**
  `PROJECT.md` — история 4.2.
- **Полный разворот** `gdau-init` (проверка имени игры → копирование шаблона → ссылки (этот блок)
  → `.env` → установка окружения `uv sync` → создание `gdau.duckdb` и представлений → `git init`)
  и **откат всего хранилища** при сбое посреди разворота — история 4.3. Этот блок откатывает
  только **свои** ссылки; полный откат хранилища оборачивает его в 4.3.
- **Где что живёт.** Данные игры, её `.env` и база — в папке игры (хранилище, со своим `git`).
  Инструмент (код, каталог, справочники, манифест) — в dev-репо, в хранилище приходит ссылками.
  В dev-репо данные игр не пишутся.
