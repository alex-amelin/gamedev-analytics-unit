## PRD Analysis

### Functional Requirements

PRD нумерует FR глобально и стабильно (FR-1…FR-21), сгруппированы по 7 фичам (§4). Каждый FR имеет блок testable Consequences.

**§4.1 Приём данных через Logs API**
- **FR-1: Оркестрация полного цикла Logs API.** За указанный диапазон дат и источник провести полный цикл: создать лог-запрос → дождаться `processed` → скачать части → очистить запрос. Соблюдает rate limit (≤30 req/s, ≤5000 req/day) и retry/backoff на 429/500/502/503.
- **FR-2: Тянуть только заданный список полей.** Выгружает ровно curated-набор полей источника, параметризуемый, не захардкоженный. Единый источник списка для выгрузки + типизации + каталога.
- **FR-3: Поддержка источников visits и hits.** Оба источника в v1; каждый грузится/хранится независимо; связь `visits.watchIDs` ↔ `hits.watchID`.
- **FR-4: Креды и окружение.** Читает `YANDEX_METRICA_TOKEN` и `YANDEX_METRICA_COUNTER_ID` из env внешнего хранилища; отсутствие → ошибка до сетевых вызовов; нет fallback на Direct-токен.
- **FR-5: Безопасная граница диапазона дат.** Не запрашивает данные за сегодня/будущее; `date2` зажимается на «вчера по МСК» с логом.

**§4.2 Хранение — два слоя**
- **FR-6: Сырьевой слой — Parquet по дням, как есть.** Партиция = один день одного источника; значения строками без CAST; партиция самодостаточна (перезаписываема); смена списка полей не мигрирует историю на лету.
- **FR-7: Рабочий слой — DuckDB с типизацией через `TRY_CAST`.** Битая ячейка → `NULL` + лог, не падение; ID-поля (`visitID`, `clientID`, `watchID`) → `HUGEINT`; способ материализации (view vs таблицы) отдан архитектуре (OQ#3).
- **FR-8: Переносимость Windows ↔ Linux.** Ноль серверных процессов (только Parquet + встроенный DuckDB); объём за ~30 дней замеряется на первой выгрузке (OQ#5).

**§4.3 Обновление и инкремент**
- **FR-9: Инкрементальная догрузка по дню.** Грузит только отсутствующие дни; источник истины — мета-таблица + фактическое наличие партиции.
- **FR-10: Идемпотентный перезалив одного дня.** Перезапись только одной партиции; повторный запуск идемпотентен; без `DROP` всей таблицы.
- **FR-11: Hot-window — перезалив последних N дней.** N конфигурируем, по умолчанию N=3; опирается на per-day идемпотентность (FR-10).

**§4.4 Консистентность и защита базы**
- **FR-12: Мета-таблица состояния (чекпойнт).** Какие дни/источники загружены, когда, сколько строк; реконсиляция против фактического `count()`/партиций; источник истины при конфликте — фактическая партиция.
- **FR-13: Сверка строк — жёсткий fail.** Расхождение источник vs хранилище → исключение/ненулевой код (НЕ warning); несошедшийся день не помечается загруженным.
- **FR-14: Атомарная запись дня.** Запись во временный файл → атомарный rename; крэш не оставляет частичную партицию.
- **FR-15: Дисциплина одного писателя.** Process-level эксклюзивный лок; второй писатель → fail-fast; чтение через MCP лока не требует.

**§4.5 Каталог схемы**
- **FR-16: Собственный каталог полей.** Машиночитаемый формат (YAML/CSV/JSON): имя, источник, тип рабочего слоя, человекочитаемое значение для каждой колонки; единый источник для DDL/типов/семантики MCP; поле без записи в каталоге — дефект.

**§4.6 Доступ агента к данным (MCP)**
- **FR-17: Выполнение SQL через MCP.** Единый инструмент `duckdb_query(query, format, limit)` + сервисные команды (`--tables`, `--schema`, `--context`, `--sample`, `--export`); большие результаты (>500 строк) авто-экспорт в файл.
- **FR-18: Контекст и схема для агента.** Контекст: таблицы/view, типы, row counts, диапазоны дат; семантика согласована с каталогом (FR-16); снятие Direct/НДС-специфики; доработка лёгкая.

**§4.7 Двух-репо и инициализация рабочего пространства**
- **FR-19: Init-команда разворачивает внешнее хранилище.** Одна команда: проверка имени → копирование шаблона → симлинки → `.env` → окружение → БД+схема → `git init`; имя занято → fail-loud.
- **FR-20: Контракт симлинков (dev-репо → хранилище).** Набор симлинкуемых путей задан декларативно; обновление инструментов dev-репо видно всем играм без копирования.
- **FR-21: Описание проекта (project context).** Файл-контекст игры создаётся из шаблона при init; доступен агенту в сессии.

**Total FRs: 21** (FR-1 … FR-21, без пропусков в нумерации).

### Non-Functional Requirements

PRD не выделяет отдельную пронумерованную секцию NFR — нефункциональные требования встроены в Consequences отдельных FR и в §5/§7. Извлечены как производные NFR:

- **NFR-1 (Производительность/Rate-limiting):** соблюдение лимитов Logs API ≤30 req/s, ≤5000 req/day; retry с backoff на 429/500/502/503 (из FR-1).
- **NFR-2 (Надёжность/Целостность данных):** жёсткий fail при расхождении строк (FR-13); атомарная запись без частичных дней (FR-14); crash-recovery через мета-таблицу (FR-12, FR-14).
- **NFR-3 (Конкурентность):** дисциплина одного писателя через process-level лок, fail-fast (FR-15); чтение не блокируется.
- **NFR-4 (Отказоустойчивость типизации):** битая ячейка → `NULL` + лог, не падение загрузки (FR-7).
- **NFR-5 (Переносимость):** копирование папки между Windows и Linux без правок путей/перезагрузки; ноль серверных процессов (FR-8, SM-4).
- **NFR-6 (Масштабируемость/объём):** ожидание единицы–десятки МБ на игру/месяц; порог пересмотра DuckDB→ClickHouse фиксируется после первого замера (FR-8, OQ#5).
- **NFR-7 (Безопасность/секреты):** креды только из env внешнего хранилища; `.env` не в dev-репо; нет fallback на Direct-токен (FR-4).
- **NFR-8 (Идемпотентность):** повторный запуск обновления не дублирует/не ломает базу (FR-9, FR-10, FR-11).
- **NFR-9 (Usability AI-native):** fail-loud понятные ошибки; авто-экспорт больших результатов (>500 строк) вместо переполнения ответа (FR-17, FR-19).
