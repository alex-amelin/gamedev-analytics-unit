# MCP-чтение: инструмент `duckdb_query`

Канал, через который агент (оператор юнита) задаёт вопросы данным игры прямо из сессии
Claude Code и получает ответ в той же сессии. Это **компонент только для чтения** — он ничего
не записывает и не меняет.

## Что делает

Агент вызывает один инструмент — `duckdb_query` — и передаёт три вещи:

- **запрос** (`query`) — SQL к данным игры;
- **формат** (`format`) — как вернуть ответ: `json` (для дальнейшей обработки), `markdown`
  (таблицей для человека) или `csv` (выгрузка). По умолчанию `json`;
- **лимит** (`limit`) — сколько строк максимум вернуть. По умолчанию 100, потолок 10 000;
  ноль/отрицательное → 100.

В ответ приходит результат запроса в выбранном формате. Данные доступны через два «окна»
(view'ы): **`visits`** — визиты, **`hits`** — просмотры/события внутри визитов. Имена колонок —
в стиле `snake_case` (например, `visit_id`, `client_id`, `page_views`); родных имён Яндекс
Метрики (`ym:s:*`) здесь нет.

Примеры запросов:

- `SELECT count(*) FROM visits` — сколько визитов загружено;
- `SELECT date, count(*) FROM visits GROUP BY 1 ORDER BY 1` — визиты по дням;
- `DESCRIBE hits` — какие колонки и типы есть у `hits`.

### Сервисные команды (навигация и выгрузка)

Кроме обычного SQL, в поле `query` можно передать короткую служебную команду — чтобы
осмотреться в данных, не вспоминая имена таблиц и колонок, или сохранить большую выборку в файл:

- **`--context`** — авто-обзор всего рабочего слоя одним вызовом: какие есть объекты
  (`visits`, `hits`, журнал загрузок `load_state`), какие у каждого колонки и типы, сколько в
  них строк и за какой диапазон дат данные, а рядом с каждой колонкой — **что она означает**
  (человекочитаемое описание из каталога схемы). Удобно как первый шаг: агент сразу понимает,
  с чем работает, без ручных подсказок;
- **`--tables`** — список доступных таблиц и «окон» (`visits`, `hits` и др.);
- **`--schema`** — схема всех объектов: какие у них колонки и типы;
- **`--schema visits`** — колонки и типы одной таблицы, **плюс колонка `semantics`** — что
  означает каждое поле (описание из каталога схемы);
- **`--sample visits 10`** — несколько строк-примеров из таблицы (по умолчанию 5; если указать
  ноль или меньше — вернётся хотя бы одна строка);
- **`--export "SELECT … FROM visits" итог.csv`** — сохранить результат запроса в файл
  (форматы `.csv`, `.parquet`, `.json`).

`--context` всегда возвращает обзор в виде размеченного текста (markdown) — параметр `format`
для неё не действует. Семантика колонок (что означает поле) берётся из **каталога схемы** —
единого источника истины о полях, поэтому она не расходится с тем, как поля типизированы.

### Большие результаты сохраняются в файл

Если результат **больше 500 строк**, инструмент не переполняет ответ — он сам сохраняет всё
в файл и возвращает короткое сообщение вида «Результат велик (N строк). Экспортирован в …».
Так агент получает указатель на файл, а не простыню из тысяч строк. Меньшие результаты, как
и прежде, ограничиваются лимитом строк (с флагом `has_more`, если что-то осталось за пределами).

## Зачем нужен

Чтобы замкнуть путь «спросил → ответ в ту же сессию» (SM-1/UJ-3): агенту не нужно выгружать
файлы и открывать сторонние инструменты — он спрашивает данные на месте. Сервисные команды
помогают сориентироваться без ручного перебора имён, а экспорт больших результатов в файл не
даёт переполнить ответ. Приём и запись данных в саму базу — это отдельный канал (команда
`gdau-logs`); здесь — только анализ уже загруженного и выгрузка результатов в отдельные файлы.

## Контракт с другими

- **Только чтение, гарантированно.** Соединение с базой открывается в режиме read-only
  (`DatabaseManager`, рабочий слой 2.1). Файл-замок писателя (`.writer.lock`, 2.5) **не
  берётся** — чтение не мешает приёму данных и не блокируется им. Любая попытка записи
  (`INSERT`/`UPDATE`/`DELETE`/`CREATE`/`DROP`/`ALTER`/`COPY … TO`/`PRAGMA`/`SET`/`ATTACH`/
  `INSTALL`/`LOAD`) **отклоняется** — даже если бы движок её технически пропустил. Разрешён
  ровно один запрос за вызов (несколько команд через `;` отклоняются).
- **Данные читаются через типизированные view'ы** `visits`/`hits` (рабочий слой 2.6) поверх
  базы `gdau.duckdb` хранилища игры. Их наполняет приём (`gdau-logs`) и разворачивание
  хранилища (init); MCP их только читает.
- **Семантика колонок согласована с каталогом схемы.** Что означает поле (`--context` и колонка
  `semantics` в `--schema TABLE`) берётся из каталога `development-docs/schema-catalog.csv` —
  единого источника истины о полях. Описания не хранятся в коде сервера, поэтому семантика не
  расходится с типами и именами колонок. Если поля нет в каталоге (например, служебные колонки
  журнала `load_state`) — семантика просто не показывается, ошибки нет. Если сам каталог
  недоступен или повреждён — `--context`/`--schema` вернут понятную ошибку (сервер не падает),
  а не половинчатый обзор без пояснений.
- **Экспорт пишет только в `data/results/`, безопасно.** Файлы `--export` и авто-экспорта
  больших результатов кладутся **строго** в каталог `data/results/` хранилища игры (создаётся
  на месте при первой записи). Защита от ошибок и злоупотреблений: путь за пределы каталога
  (абсолютный или с `..`) — **отказ**; неизвестное расширение (не `.csv`/`.parquet`/`.json`) —
  **отказ** (без молчаливого до-приписывания); запись поверх существующего файла — **отказ**
  (предложит другое имя), чтобы не затереть прежнюю выгрузку. Сам запрос внутри `--export`
  обязан быть запросом чтения — попытка записи (`DROP`/`INSERT`/…) отклоняется и до файла не
  доходит. База `gdau.duckdb` при экспорте не меняется — пишется отдельный файл-результат,
  замок писателя по-прежнему не берётся.
- **Каждый вызов журналируется.** Обращение к инструменту записывается в журнал (audit) —
  отдельный JSON-файл в `data/mcp_output/` (что вызвали, когда, с какими параметрами и каким
  результатом). Если журнал записать не удалось (нет места, нет каталога) — это не ломает сам
  запрос: ответ агенту всё равно возвращается, а сбой журналирования отмечается предупреждением.
- **До первой выгрузки данных нет.** Если база ещё не создана, инструмент вернёт понятную
  подсказку «запусти `gdau-logs update`», а не техническую ошибку.
- **Защита от тяжёлых запросов.** У запроса есть верхняя граница времени; «убегающий» запрос
  прерывается с понятным сообщением. Большие результаты ограничиваются лимитом строк (агент
  видит в `json`-ответе флаг `has_more`, что результат усечён).
- **Сервер запускается** через `.mcp.json` командой `uv run python -m scripts.mcp.gdau_mcp_server`;
  для резолюции пути к базе ему нужна только переменная `GDAU_DATA_ROOT` (берётся из `.env`
  хранилища). Креды Яндекс Метрики каналу чтения не нужны.

## Границы (что НЕ здесь)

Канал чтения собран полностью (Epic 3 закрыт): произвольный SQL (3.1), сервисные команды
навигации, экспорт больших результатов и журнал обращений (3.2), авто-контекст рабочего слоя и
семантика колонок из каталога схемы (3.3) — всё поверх одной точки входа `duckdb_query`.

Чего канал **не** делает: не пишет в базу `gdau.duckdb` (только читает) и не ходит во внешний
Яндекс Метрика API — это задача приёма (`gdau-logs`). Семантику колонок канал не придумывает: она
живёт в каталоге схемы, а здесь только показывается. Никакой рекламной (Яндекс Директ) специфики —
денежных/НДС-колонок, целей-плейсхолдеров, отдельного конфига целей — в канале нет: продукт про
веб-аналитику игры, а не про рекламу.

## Как проверить вручную (smoke, опционально)

1. Указать на готовое хранилище с данными: `GDAU_DATA_ROOT=G:/gdau-smoke`.
2. Поднять сервер: `uv run python -m scripts.mcp.gdau_mcp_server` (или подключить через
   `.mcp.json` в Claude Code).
3. Проверить чтение и навигацию:
   - `duckdb_query("SELECT count(*) FROM visits")` — число загруженных визитов;
   - `duckdb_query("--context")` — обзор рабочего слоя: есть `visits`/`hits` с числом строк и
     диапазоном дат, у колонок — описания из каталога;
   - `duckdb_query("--tables")` — список таблиц/view;
   - `duckdb_query("--schema visits")` — колонки, типы и колонка `semantics` (описания каталога);
   - `duckdb_query("--sample visits 3")` — три строки-примера.
4. Проверить выгрузку: `duckdb_query("--export \"SELECT * FROM visits LIMIT 10\" v.csv")` —
   файл `v.csv` появится в `data/results/`.
5. Заглянуть в `data/mcp_output/` — на каждый вызов там лежит JSON-файл журнала обращений.
