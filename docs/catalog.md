# Каталог схемы

Компонент: единый справочник полей, которые юнит выгружает из Метрики.
Реализация: `scripts/utils/catalog.py`; сам справочник — `development-docs/schema-catalog.csv`.

## Что делает

Каталог — это табличка-список всех полей, которые нас интересуют в данных Метрики.
Каждая строка описывает одно поле сразу с трёх сторон:

- **родное имя в Метрике** (например `ym:s:visitID`) — как поле называется в её API;
- **наше короткое имя** (`visit_id`) — как оно будет называться у нас в таблицах;
- **тип данных** (число, дата, текст, список) — в терминах нашего рабочего движка;
- плюс **человеческое описание**, что это за поле.

Компонент `catalog.py` читает эту табличку и отдаёт её содержимое в удобном виде:
список полей для заказа выгрузки, соответствие «наше имя → тип», полный набор
полей по каждому источнику. Источников два — `visits` (визиты) и `hits` (просмотры).

При чтении он же **проверяет табличку на ошибки** и, если что-то не так, сразу
останавливается с понятным сообщением (а не молча отдаёт кривые данные). Что считается
ошибкой: пустая обязательная ячейка, незнакомый источник, имя поля без правильной
приставки источника, неизвестный тип, два поля с одинаковым именем внутри одного
источника, сбитый заголовок таблички.

## Зачем нужен

Одни и те же поля участвуют в трёх местах: их **заказывают** у Метрики, по ним
**раскладывают** скачанные данные с правильными типами и по ним же агенту
**объясняют**, что лежит в каждой колонке. Если хранить этот список в трёх местах,
он рано или поздно разъедется: где-то поле добавили, где-то забыли, типы перестали
совпадать — и данные тихо поедут.

Поэтому список живёт **в одном месте** — в табличке-каталоге, и это единственный
источник истины. Любое поле, которого нет в каталоге, считается недосмотром, а не
данными. Типы при этом не выдумываются на глаз: для каждого типа из справочника
Метрики заранее зафиксировано, во что он превращается у нас (например, огромные
идентификаторы Метрики не влезают в обычное целое — для них отведён специальный
большой целочисленный тип).

## Контракт с другими

- **Что принимает на вход.** Табличку-каталог из пяти колонок
  (`source, storage_name, metrica_field, type, description`). Путь к ней по умолчанию
  известен, но его можно подменить — этим пользуются тесты, подсовывая маленькие
  примеры. Табличка — часть кода юнита (живёт рядом с ним), а не данные конкретной
  игры; в рабочее пространство игры она приходит ссылкой на общий файл.
- **Что обещает потребителям.** Тем, кто заказывает выгрузку (CLI и оркестратор
  приёма), — готовый список полей источника ровно в том порядке, как в табличке.
  Тем, кто раскладывает данные по таблицам, — соответствие «наше имя → тип». Тем,
  кто объясняет данные агенту, — имена и описания полей. Всё строго по источнику:
  поля визитов не перемешиваются с полями просмотров.
- **Как сообщает о проблемах.** Любая нестыковка в табличке — это сразу понятная
  ошибка с номером проблемной строки и её содержимым, а не тихий пропуск. Лучше
  громко упасть на старте, чем выгрузить или разложить данные неправильно.

## Границы

Каталог только **читает и проверяет** свою табличку и переводит типы. Он **не**
ходит в сеть, **не** создаёт таблиц и не раскладывает данные, **не** знает про пути
рабочего пространства игры и **не** правит саму табличку. Разбор скачанных данных,
создание представлений с типами и реальная типизация — работа компонентов приёма
(Эпик 2). Здесь — только единый список полей и их типов.
