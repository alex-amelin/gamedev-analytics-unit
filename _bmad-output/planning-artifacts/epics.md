# gamedev-analytics-unit - Epic Breakdown

## Overview

Этот документ — полная разбивка на эпики и истории для **Game Dev Analytics Unit**: декомпозиция требований PRD и архитектурных решений в реализуемые истории. UX-спеки нет (оператор юнита — агент Claude Code через каналы CLI + MCP, человеческого UI нет). Источник истины по поведению — PRD (FR-1…FR-21); источник истины по реализации — Architecture Decision Document (решения, паттерны, структура, Implementation Sequence).

## Requirements Inventory

### Functional Requirements

**Группа 4.1 — Приём данных через Logs API**
- **FR-1:** Оркестрация полного цикла Logs API за диапазон дат и источник (create → poll до `processed` → download parts → clean); fail с диагностикой при `processing_failed`/`canceled`; соблюдение rate-limit и retry/backoff (наследуется от вендоренного клиента).
- **FR-2:** Тянуть только заданный список полей (параметризуемый, не захардкоженный «на всё»); единый список для visits и hits, используемый при выгрузке, типизации и в каталоге схемы — без рассинхрона.
- **FR-3:** Поддержка обоих источников — `visits` (сессионные/retention-поля) и `hits` (путь внутри сессии); грузятся/хранятся независимо; связь visits↔hits через `visits.watchIDs` ↔ `hits.watchID`.
- **FR-4:** Креды и окружение — тонкий env-ридер читает `YANDEX_METRICA_TOKEN` + `YANDEX_METRICA_COUNTER_ID`; отсутствие любого → понятная ошибка до сетевых вызовов; без Direct-fallback; `.env` живёт во внешнем хранилище.
- **FR-5:** Безопасная граница диапазона дат — `date2` зажимается на «вчера по МСК» с записью в лог (Logs API требует `date2 < today`).

**Группа 4.2 — Хранение, два слоя**
- **FR-6:** Сырьевой слой — Parquet, партиции по дню, строками, без CAST; одна партиция = один день одного источника; партиция самодостаточна (перезапись не трогает другие дни); смена списка полей не мигрирует исторические партиции на лету.
- **FR-7:** Рабочий слой — DuckDB поверх Parquet с типизацией через `TRY_CAST` (битая ячейка → `NULL` + лог, не падение); ID-поля (`visit_id`/`client_id`/`watch_id`) → HUGEINT (Metrica ID > 2^63).
- **FR-8:** Переносимость Windows ↔ Linux копированием папки; ноль серверных процессов (только Parquet-файлы + встроенный DuckDB); объём сырья за ~30 дней замеряется на первой выгрузке.

**Группа 4.3 — Обновление и инкремент**
- **FR-9:** Инкрементальная догрузка по дню — грузятся только отсутствующие дни (по мета-таблице + факту партиции, не только `SELECT DISTINCT date`).
- **FR-10:** Идемпотентный перезалив одного дня — перезапись только его Parquet-партиции, без `DROP` всей таблицы; повторный запуск даёт тот же результат.
- **FR-11:** Hot-window — перезалив последних N дней при каждом обновлении (доезжающие данные); N конфигурируем, по умолчанию N=3; опирается на per-day идемпотентность (FR-10).

**Группа 4.4 — Консистентность и защита базы**
- **FR-12:** Мета-таблица состояния `load_state` (source, date, row_count, loaded_at, status) — чекпойнт; на старте реконсилируется против факта партиций (источник истины — факт партиции); день засчитан только при: партиция есть + сверка сошлась + мета согласована.
- **FR-13:** Сверка строк — жёсткий fail: расхождение число строк (источник vs БД) → исключение/ненулевой код (НЕ warning); день с несошедшейся сверкой не помечается загруженным.
- **FR-14:** Атомарная запись дня — сначала в `.tmp`, затем атомарный rename в финальную партицию; крэш не оставляет частичных дней; после сбоя повторный запуск дозагружает прерванный день.
- **FR-15:** Дисциплина одного писателя — process-level эксклюзивный `.writer.lock`; второй конкурентный писатель → fail-fast (без ожидания/записи); чтение (MCP) лока не требует.

**Группа 4.5 — Каталог схемы**
- **FR-16:** Собственный каталог полей в машиночитаемом формате — для каждой колонки заданного списка: имя, источник (visits/hits), тип рабочего слоя, человекочитаемое значение; единый источник (SSOT) для DDL рабочего слоя (FR-7) и семантики, отдаваемой агенту через MCP (FR-18); поле без записи в каталоге = дефект.

**Группа 4.6 — Доступ агента к данным (MCP)**
- **FR-17:** Выполнение SQL через MCP `duckdb_query(query, format, limit)` с сервисными командами (`--tables`, `--schema [TABLE]`, `--context`, `--sample TABLE [N]`, `--export`); произвольный SQL → результат (json/markdown/csv); большие результаты (>500 строк) авто-экспортируются в файл.
- **FR-18:** Контекст и схема для агента — MCP отдаёт таблицы/view'ы, типы, row counts, диапазоны дат; семантика колонок согласована с каталогом (FR-16); Direct/НДС/goal-плейсхолдеры убраны/заменены под нашу схему; интерфейс единого инструмента сохраняется (лёгкая доработка).

**Группа 4.7 — Двух-репо и инициализация рабочего пространства**
- **FR-19:** Init-команда разворачивает внешнее per-game хранилище из шаблона одной командой (проверка имени → копирование шаблона → симлинки → генерация `.env` → подготовка окружения/зависимостей → создание БД + схема → `git init`); на выходе готово к первой выгрузке без ручных правок кроме токена/счётчика; имя занято → fail-loud.
- **FR-20:** Контракт симлинков (dev-репо → хранилище) — набор симлинкуемых путей задан декларативно (а не разбросан по коду init); обновление инструмента в dev-репо автоматически видно всем хранилищам.
- **FR-21:** Описание проекта (project context) — каждое хранилище содержит файл описания игры (из шаблона при init, заполняет владелец); агент имеет к нему доступ в рабочей сессии.

### NonFunctional Requirements

*PRD не нумерует NFR отдельно; формализованы из раздела «Non-Functional Requirements» архитектуры и сквозных требований целостности.*

- **NFR-1 (Целостность данных — доминирующий NFR, «не сломать базу»):** идемпотентность (FR-10), атомарность записи (FR-14), жёсткая сверка строк (FR-13), единственный писатель (FR-15), crash-recovery и реконсиляция мета×факт (FR-12). Требование первого класса.
- **NFR-2 (Переносимость + ноль серверов):** хранилище переносится копированием папки между Windows и Linux и продолжает работать; ноль серверных процессов (только файлы Parquet + встроенный DuckDB); окружение через `uv run` (без bash/nushell-обвязки).
- **NFR-3 (Устойчивость к лимитам Logs API):** rate-limit ≤30 req/s, ≤5000 req/day; retry с backoff [30,60,120] на 429/500/502/503 — наследуется от вендоренного клиента, в оркестраторе заново не реализуется.
- **NFR-4 (Корректность типов):** HUGEINT для `visit_id`/`client_id`/`watch_id` (UInt64 > 2^63, не влезает в BIGINT); типы сидятся из официального справочника Logs API через маппинг ClickHouse→DuckDB, не угадываются.
- **NFR-5 (Безопасность кредов):** `.env` живёт во внешнем per-game хранилище, не в dev-репо и не коммитится; отсутствие токена/счётчика → fail-loud до сетевых вызовов.
- **NFR-6 (Простота-первой — направляющий инвариант):** не тащить тяжёлую инфраструктуру directaiq (queue/worker, disk-guard, cron, мульти-аккаунт, `BaseScript`, `config_manager`); усложнять только по реальной потребности.
- **NFR-7 (Качество кода / CI):** type hints обязательны (mypy); тесты `pytest` зеркалят `scripts/`; CI на GitHub Actions (uv + pytest + mypy); правило каталога (все поля списка покрыты) проверяется.

- **Швы вендоринга (требуют развязки при переносе):** (1) `MetricaClient.__init__` зовёт `AuthManager` → инжектировать готовые креды от env-ридера (условие FR-4); (2) обрезать reporting/Stat-методы клиента и зависимость на `polars` → клиент чисто на `requests`; (3) MCP завязан на `config_manager` (goal-плейсхолдеры) и хардкод НДС/денег в `_COST_COLUMN_SEMANTICS` + regex-fallback `(cost|.*_revenue)` → заменить семантикой из каталога, снять плейсхолдеры (условие FR-17/18).
- **Каталог схемы — формат и сидинг:** CSV `development-docs/schema-catalog.csv`, колонки `source, storage_name, metrica_field, working_type, description`; строка на поле; SSOT для DDL view и семантики MCP. `working_type` сидится из официального справочника Logs API (`yandex-docs/metrika-api/...logs_fields_visits.md` и `_hits.md`) через явный маппинг ClickHouse→DuckDB (UInt64→HUGEINT, UInt8→BOOLEAN, Array(T)→LIST<T> и т.д.). Человекочитаемая `data-architecture.md` генерируется из каталога.
- **Реальные symlinks + preflight:** контракт `templates/paths-to-symlink.csv`; init делает preflight-проверку способности создавать symlink (на Windows — Developer Mode), fail-loud с инструкцией.
- **Раскладка хранения (OQ#3 — решено):** партиции `data/raw/{source}/{YYYY-MM-DD}.parquet`; рабочий слой — DuckDB-**view'ы** поверх Parquet (не материализованные таблицы); view→таблицы — обратимый escape hatch при росте объёма (не one-way).
- **Структура dev-репо повторяет каркас directaiq** (тренированная навигация владельца): `scripts/utils/*`, `scripts/8x_metrica_logs_api/p81_load_logs.py` (оркестратор), `scripts/tools/logs_api_cli.py` (CLI), `scripts/mcp/gdau_mcp_server.py` + `tools/core.py`, `scripts/init/init_project.py`. Entry points: `gdau-logs`, `gdau-init`; MCP — через `.mcp.json`.
- **✅ OQ#1 ЗАКРЫТ (список полей зафиксирован) — дата-эпики разблокированы:** заданный список полей = содержимое `development-docs/schema-catalog.csv` уже целиком (76 полей `visits` + 41 поле `hits`; `working_type` уже в DuckDB-форме, `metrica_field` проставлены). Базово тянем **весь этот датасет** в две отдельные таблицы/view — `visits` и `hits` (решение владельца 2026-05-23). Каталог уже существует и наполнен → он SSOT; работа FR-16 сводится к загрузчику каталога (`catalog.py`) и опоре DDL/MCP на него, а не к наполнению с нуля. Каталог включает Direct-атрибуцию и ecommerce-поля (`purchase_*`/`products_*`/`events_product_*`/`direct_*`) — это осознанное решение владельца по скоупу полей, каталог авторитетен.

### Epic 2: Приём данных и безопасное обновление хранилища

После эпика данные игры за период лежат в базе и читаются SQL; повторное обновление идемпотентно и не ломает базу (SM-1, SM-2, SM-4). Ядро NFR-1 «не сломать базу».
**FRs covered:** FR-1, FR-3, FR-6, FR-7, FR-8, FR-9, FR-10, FR-11, FR-12, FR-13, FR-14, FR-15

### Epic 3: Доступ агента к данным через MCP

После эпика замыкается цикл «спросил → ответ в ту же сессию» (SM-1 целиком, UJ-3).
**FRs covered:** FR-17, FR-18

### Epic 4: Развёртывание рабочего пространства игры (init)
Одной командой `gdau-init {game}` развернуть per-game хранилище: проверка свободного имени → копирование шаблона хранилища → симлинки по декларативному контракту (+ preflight-проверка Dev Mode, fail-loud) → генерация `.env` → `uv sync` → создание `gdau.duckdb` + view'ы из каталога → `git init`; плюс файл описания проекта `PROJECT.md` для агента. После эпика новая игра поднимается одной командой без ручной возни (SM-3, UJ-1).
**FRs covered:** FR-19, FR-20, FR-21

### Story 1.1: `uv`-каркас, раскладка проекта и CI

As a оператор юнита,
I want воспроизводимый `uv`-каркас dev-репо с раскладкой `scripts/`, entry points и CI,
So that все остальные инструменты ложатся на стабильный фундамент со стеком, закреплённым локом.

**Acceptance Criteria:**

- **Given** инициализируемый репозиторий, **When** выполняется `uv init --package` (src-layout), **Then** создаются `pyproject.toml` + `uv.lock` + `.python-version` (пол `>=3.13`), **And** `uv sync` отрабатывает без ошибок.
- **Given** pyproject, **When** смотрим зависимости, **Then** запинены ровно `duckdb` (1.5.x), `requests`, `mcp` (≥1.2), `python-dotenv`, `PyYAML`, **And** аналитический стек (pandas/numpy/scipy/numba/prophet/polars) и `tapi-yandex-*` отсутствуют.
- **Given** раскладку, **When** смотрим дерево, **Then** существуют `scripts/{utils,8x_metrica_logs_api,tools,mcp,init}/`, `tests/`, `templates/`, `development-docs/`, `yandex-docs/metrika-api/` (по карте соответствия архитектуры).

## Epic 2: Приём данных и безопасное обновление хранилища

Полный путь записи как один связный компонент (упорядоченные истории 2.1→2.9): соединение/пути → атомарная Parquet-партиция → сверка строк → мета-состояние → лок писателя → view'ы из каталога → оркестратор полного цикла → инкремент/hot-window → команда `update`. После эпика данные игры за период лежат в базе и читаются SQL; повторное обновление идемпотентно и не ломает базу (SM-1, SM-2, SM-4). Покрывает FR-1, FR-3, FR-6, FR-7, FR-8, FR-9, FR-10, FR-11, FR-12, FR-13, FR-14, FR-15; ядро NFR-1. _AC усилены прогоном edge-case hunter._

**Acceptance Criteria:**

- **Given** `gdau-init {game}`, **When** выполняется, **Then** по шагам: проверка свободного имени → копирование шаблона (4.2) → симлинки по контракту (4.1, + preflight) → генерация `.env` → `uv sync` → создание `gdau.duckdb` + view'ы из каталога (2.6) → `git init` + initial commit.
- **Given** имя занято / папка `../{game}` существует, **When** выполняется init, **Then** fail-loud без перезаписи чужих данных.
- **Given** успешный init, **When** завершён, **Then** хранилище готово к первой выгрузке (FR-1) без ручных правок, кроме токена/счётчика в `.env`.
- **Given** initial commit, **When** он делается, **Then** `.env` НЕ коммитится.
- **Given** init на Python, **When** запускается на Windows и Linux, **Then** работает кросс-платформенно (без bash/nushell).
- **Given** сбой посреди init (любой шаг), **When** он происходит, **Then** уже созданное хранилище/симлинки очищаются (откат), чтобы имя не оставалось «занятым» мусором и повтор был возможен. _[edge-case: частичный init → мусор]_
- **Given** имя игры со спецсимволами/path-separator/пробелами/зарезервированными Windows-именами (`CON`,`NUL`,…)/ведущими точками/слишком длинное, **When** проверяется имя, **Then** валидация по строгому шаблону с понятным отказом. _[edge-case: опасное имя]_
- **Given** dev-репо ещё не git-репо ИЛИ `{game}` оказывается вложенным в существующий git-репо, **When** делается `git init`, **Then** репо инициализируется только в `{game}`, вложенность/уже-репо распознаётся, без путаницы. _[edge-case: git-вложенность]_
- **Given** `uv sync` падает / нет сети ИЛИ `git` не установлен, **When** это случилось, **Then** для `git` — preflight с инструкцией; для `uv sync` — отчёт + откат частичного хранилища. _[edge-case: провал зависимостей/git]_
- **Given** повторный запуск init после частичного сбоя, **When** он стартует, **Then** поведение определено: чистый повтор (после очистки) либо явная ошибка с указанием очистить остаток. _[edge-case: resume/идемпотентность init]_
- **Given** неоднозначность пути `../{game}` от cwd, **When** резолвится цель, **Then** она вычисляется относительно корня dev-репо (`dev_repo_root.parent / game`), а не от текущего каталога вызова. _[edge-case: резолюция пути от cwd]_
- **Given** диск полон / нет прав на запись при копировании шаблона, **When** копирование падает, **Then** `OSError` ловится → очистка + понятная ошибка. _[edge-case: диск/права при копировании]_
- **Given** initial commit (где `.env` исключён), **When** он делается, **Then** в индексе есть файлы шаблона (коммит не пустой) — нет ошибки «nothing to commit». _[edge-case: пустой initial commit]_
- **Given** создание view'ов на свежем хранилище без данных, **When** init строит view'ы из каталога, **Then** они создаются, толерантно к нулю партиций (см. 2.6 пустой источник). _[edge-case: view'ы на пустом хранилище]_
