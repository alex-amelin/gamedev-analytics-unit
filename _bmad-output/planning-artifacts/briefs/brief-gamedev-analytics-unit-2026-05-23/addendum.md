# Addendum — техническая глубина и референс-заметки

Не часть лёгкого брифа. Это вход для будущих PRD и архитектуры. Здесь зафиксировано, **как работа с данными устроена в `G:\git\directaiq`** (образец, хорошо выстроенный под MVP), и **какие осознанные отличия** мы вносим под нужды геймдев-аналитики. Все расхождения между directaiq и нашими требованиями отмечены явно.

> Формат каждого пункта: **directaiq (как сейчас)** → **наше решение**. Ссылки на файлы — для PRD; номера строк могут сдвинуться.

---

## 1. Жизненный цикл приёма (Logs API)

**directaiq.** Клиент `scripts/utils/metrica_client.py` (чистый `requests`): `create_request → get_status (poll) → get_parts → download_part → clean_request`. Оркестрация — `scripts/8x_metrica_logs_api/p81_load_logs.py` (класс-загрузчик). CLI-обёртка — `scripts/tools/logs_api_cli.py` (`create/status/download/load/clean`).

**Наше решение.** Тот же цикл и тот же клиент как примитив (вендорим). Оркестрацию пишем свою под Parquet/DuckDB. Тянем **только заданный список полей** — не curated-подмножество directaiq и не «всё».

## 2. Хранение и партиционирование

**directaiq.** DuckDB single-file (`{external_storage}/data/duckdb/...`), загрузка через `read_csv(...)` напрямую в таблицы. Промежуточные TSV в temp.

**Наше решение.** Два слоя. Сырьё → **Parquet, партиции по дню**, как есть (строками, верно источнику). Рабочий слой → **DuckDB поверх Parquet** с типизацией заданных полей. Преимущества: мало места (сжатие), переносимость Win↔Linux (папка/файл, ноль сервера), дешёвый перезалив дня.

## 3. Идемпотентность и перезалив дня ⚠️ ОТЛИЧИЕ

**directaiq.** PRIMARY KEY на `t81_metrica_logs_*` нет. При старте `_get_loaded_dates_duckdb()` берёт `SELECT DISTINCT date` → если дата уже есть, день **пропускается**. `--force` **дропает таблицу целиком** и заливает заново. Перезалить *один* день нельзя — только всю таблицу.

**Наше решение.** Настоящая per-day идемпотентность: перезалив дня = **перезапись одного Parquet-файла этого дня**, остальные дни не трогаются. Естественно ложится на партиции по дню. (Совпадает с идеей `DROP PARTITION + INSERT`, но без сервера.)

## 4. Hot-window / доезжающие данные ⚠️ ОТЛИЧИЕ

**directaiq.** Автоперезалива последних дней **нет**. `get_moscow_safe_end_date()` лишь clamp'ит `date2` на «вчера по МСК» (Logs API требует `date2 < today`). Перекрытие диапазона ради late-данных — задача пользователя (вручную задать меньший `--date1`).

**Наше решение.** Явное **окно перезалива последних N дней** при каждом обновлении (N — параметр; уточнить в PRD), чтобы доезжающие данные подтягивались автоматически. Опирается на per-day идемпотентность (п.3).

## 5. Консистентность / сверка ⚠️ ОТЛИЧИЕ

**directaiq.** Сверка числа строк есть, но это **WARNING, не fail** (`p81_load_logs.py:783`: `if imported != csv_rows: logger.warning(...)`), скрипт продолжает с `exit_code=0`. Проверок на битый TSV, число частей (кроме info-лога), дубликаты — нет.

**Наше решение.** Сверка строк (источник vs БД) — **жёсткий fail** (исключение/ненулевой код), а не warning. День не считается загруженным, пока сверка не сошлась.

## 6. Типизация при загрузке ⚠️ ОТЛИЧИЕ

**directaiq.** Явный строгий `CAST` в SQL (`CAST("ym:s:visitID" AS HUGEINT)`, `read_csv(..., types={...})`). Битое значение → ошибка CAST → **падает весь день** (exception пробрасывается).

**Наше решение.** Сырой Parquet — без кастов (строками, верно источнику). Типизация заданных полей — в слое DuckDB через **`TRY_CAST`**: битая ячейка → `NULL` + запись в лог, без падения загрузки дня. Снимает и риск молчаливой потери данных на входе, и риск падения всего дня из-за одной кривой ячейки.

## 7. Чекпойнт / состояние и crash-recovery

**directaiq.** `table_metadata` (DuckDB: `table_name, last_updated, updated_by_script, row_count, operation_type, comment`), пишется через `BaseScript._log_table_operation()`. Состояние «что загружено» определяется прямым `SELECT DISTINCT date`. Слабое место: крэш посреди дня → день залит частично, но при перезапуске **считается загруженным и пропускается** (нужен `--force`).

**Наше решение.** Атомарная запись дня: пишем во временный файл → атомарный rename в финальный Parquet. Недозаписанный день в хранилище не появляется → нет «частично загруженных» дней. Плюс мета-таблица состояния загрузок (по образцу `table_metadata`) и реконсиляция против фактического `count()` на старте.

## 8. Параллелизм / очередь

**directaiq.** Загрузки p81/p88 — только через `queue_cli` (нельзя напрямую). Причина: single-writer lock DuckDB. `queue_worker` берёт `flock` на `.worker.lock` (один воркер, FIFO) + disk-guard (`MIN_FREE_GB = 10`, при нехватке — пауза, sentinel `.low_disk_space`).

**Наше решение.** Для одиночного пользователя тяжёлая очередь — **вне v1**. Дисциплина «один писатель» сохраняется естественно (последовательные запуски). Запись Parquet по дням снижает контеншн на DuckDB-локе. Disk-guard — опционально, как лёгкая проверка.

## 9. Каталог схемы и доступ агента

**directaiq.** Единый источник — `development-docs/data-architecture.md` (поля, типы, семантика). Агенту схема/контекст отдаются через MCP `duckdb_query --context` (таблицы + схемы + row counts + диапазоны дат) и `--schema [TABLE]` (DDL одной таблицы).

**Наше решение.** **Собственный каталог** полей (не вендорим чужой CSV): значение каждой колонки — единый источник для DDL и навигации агента. MCP `duckdb_query` переиспользуем из directaiq с лёгкой доработкой.

## 10. Двух-репо: init и симлинк-контракт

**directaiq.** `scripts/nushell/init_project.nu` по шагам: проверка имени → копирование `templates/external_storage/` → запись `.migrations_applied` → симлинки по `templates/paths-to-symlink.csv` (`.claude/*`, `activate.sh`, `scripts`, `pyproject.toml`, `toolkit.nu`, …) → генерация `.env` (`*_DATA_ROOT`) → venv + deps → создание БД + схема → `git init`.

**Наше решение.** Тот же паттерн: dev-репо (инструменты) ↔ внешнее per-game хранилище (БД + конфиги + рабочая папка), связь симлинками, init одной командой. Конкретный контракт путей — в PRD.

## 11. Креды и окружение

**directaiq.** `AuthManager.get_metrica_credentials()` читает `YANDEX_METRICA_TOKEN` (fallback `YANDEX_DIRECT_TOKEN`) + `YANDEX_METRICA_COUNTER_ID` (обязателен). `.env` лежит во внешнем хранилище и грузится в `activate.sh`.

**Наше решение.** `YANDEX_METRICA_TOKEN` + `YANDEX_METRICA_COUNTER_ID` (без Direct-fallback — у нас только Метрика). `.env` во внешнем per-game хранилище; тонкий env-ридер вместо завязанного на Direct `AuthManager`.

## Открытые технические вопросы для PRD

- **Точный список полей** Logs API (visits/hits), которые тянем — назвать и занести в каталог.
- **N в hot-window** — сколько последних дней перезаливать.
- **Раскладка и именование** Parquet-партиций — материализуем ли DuckDB-таблицы или запрашиваем Parquet напрямую.
- **Объём доработки MCP** `duckdb_query` в v1 под нашу схему.
- **Фактический вес данных** — порог пересмотра DuckDB→ClickHouse; замерить на первой реальной выгрузке.
