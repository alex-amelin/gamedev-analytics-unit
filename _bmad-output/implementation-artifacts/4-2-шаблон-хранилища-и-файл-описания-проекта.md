## Story

As a оператор юнита,
I want шаблон внешнего per-game хранилища с файлом описания игры и примитив его копирования,
so that новое хранилище разворачивалось из готового набора файлов, агент знал, с какой игрой работает, а повторный init не затирал заполненное владельцем описание.

**Контекст эпика.** ВТОРАЯ история **Epic 4 «Развёртывание рабочего пространства игры (init)»** (FR-19/20/21). Эпик разворачивает per-game хранилище одной командой `gdau-init {game}`: проверка имени → **копирование шаблона (4.2, эта история)** → симлинки по контракту (4.1) + preflight → `.env` → `uv sync` → `gdau.duckdb` + view'ы из каталога (2.6) → `git init` (4.3). Истории Epic 4 — упорядоченные «снизу вверх»: 4.1 — декларативный симлинк-контракт + механизм (примитив, **сделана до 4.2**), **4.2 — статический шаблон хранилища + `PROJECT.md` + примитив копирования** (эта), 4.3 — оркестратор `gdau-init`, склеивающий 4.1+4.2 в полный init. 4.2 покрывает **FR-21** (каждое хранилище содержит файл описания игры из шаблона при init; агент имеет к нему доступ в рабочей сессии) и поставляет **сам шаблон**, который копирует 4.3 (FR-19 «копирование шаблона»). Обслуживает **NFR-5** (`.env` в хранилище, не коммитится — через `.gitignore` шаблона), границу двух-репо и **NFR-6** (шаблон урезан под геймдев, без инфры/маркетинга directaiq).

**Что именно даёт 4.2 (две части — параллель с 4.1).** Как 4.1 поставила декларативный артефакт (`paths-to-symlink.csv`) **+** примитив-потребитель (`symlinks.py`), 4.2 поставляет:
1. **Статический шаблон** `templates/external_storage/` — набор файлов, которые физически копируются в новое хранилище (AC #1, #3, #4): `.env.example`, `.gitignore`, `CLAUDE.md`, `PROJECT.md`.
2. **Примитив копирования** `scripts/init/scaffold.py` (новый, инъектируемый, как `symlinks.py` 4.1): валидирует наличие шаблона **до любых мутаций** (AC #5), копирует файлы в хранилище, **не затирает уже заполненный `PROJECT.md`** при повторном init (AC #6). 4.3 вызовет его на шаге «копирование шаблона», обернёт в полный откат хранилища при сбое.

**Зачем шаблон, а не «создать файлы кодом» (корень требования).** Содержимое хранилища, специфичное для игры, — это **редактируемые человеком файлы** (`PROJECT.md` заполняет владелец, `.env` — креды, `CLAUDE.md` — инструкции агенту в рабочем пространстве). Их естественно держать **версионируемым шаблоном** в dev-репо (`templates/external_storage/`), а не генерировать строками в коде init: правка формулировки в `CLAUDE.md`/`PROJECT.md` — это правка файла-шаблона (git diff читаемый), а не правка Python-литерала. Это форма directaiq (`templates/external_storage/` существует там), но **наполнение — своё под геймдев**, урезанное (AC #4).

**Зачем `PROJECT.md` (FR-21).** Агент Claude Code — единственный оператор юнита. Чтобы он понимал предметную область конкретной игры (платформа, ключевые события/цели, что значат поля под эту игру, особенности атрибуции), хранилище несёт **файл описания игры** `PROJECT.md`: владелец заполняет его при заведении игры, агент читает в рабочей сессии. У directaiq аналога нет (там `config/client_context.md` под рекламный аккаунт) — **`PROJECT.md` это новый артефакт gdau** под продуктовую/игровую семантику.

**Кто это потребляет (проектируй API под них).**
- **`gdau-init` / `init_project.py` (4.3)** — единственный прямой потребитель примитива копирования. Сейчас `scripts/init/init_project.py` — **стаб** (печатает «not yet implemented», exit 0; заведён в 1.1). 4.3 вызовет `copy_storage_template(storage_root=…, template_root=…)` на шаге «копирование шаблона» — **до** симлинков (4.1) и до генерации `.env`/`uv sync`/БД. Проектируй модуль **переиспользуемым и инъектируемым** (корни хранилища и шаблона приходят параметрами — тесты на `tmp_path`, прод даёт реальные пути), как швы 2.x и `symlinks.py` 4.1.
- **4.3 (откат частичного init)** — AC 4.3 (epics.md:482, 488) требует отката всего хранилища при сбое посреди init и ловли `OSError` при копировании. 4.2 примитив копирования **сам отката не делает** (граница, см. решение D6): полный откат хранилища (скопированный шаблон + симлинки + `.env` + БД) — забота 4.3. 4.2 лишь fail-loud при отсутствии шаблона (AC #5) и идемпотентное сохранение `PROJECT.md` (AC #6).
- **Агент в рабочей сессии (UJ-3)** — потребитель `PROJECT.md` и `CLAUDE.md` хранилища: читает их для контекста игры и правил рабочего пространства (AC #3).
- **НЕ потребители:** приём (Epic 2), MCP-чтение (Epic 3), симлинки (4.1 — отдельный примитив). 4.2 — чисто файловая операция (копирование) + статические артефакты; в сеть не ходит, БД не открывает, `.writer.lock` не берёт, симлинки не создаёт.

**Раскладка артефактов истории (4 файла шаблона + модуль + спека + тест):**
- `templates/external_storage/.env.example` — **новый**: образец `.env` с `YANDEX_METRICA_TOKEN=`/`YANDEX_METRICA_COUNTER_ID=` (AC #1).
- `templates/external_storage/.gitignore` — **новый**: игнор `.env`, `data/`, `.writer.lock` + симлинкуемые пути инфры (AC #1; решение D3).
- `templates/external_storage/CLAUDE.md` — **новый**: инструкции агенту в рабочем пространстве игры, написаны заново под gdau (НЕ вендоринг маркетингового directaiq; AC #4, решение D4).
- `templates/external_storage/PROJECT.md` — **новый**: шаблон описания игры с подсказками-плейсхолдерами для владельца (FR-21; AC #2, #3; решение D5).
- `scripts/init/scaffold.py` — **новый** примитив копирования шаблона (валидация fail-loud + копирование + сохранение `PROJECT.md`); инъектируемые корни. (`tests/test_init_scaffold.py` — зеркало по имени.)
- `docs/init-and-storage.md` — **дополнение** новым разделом «шаблон хранилища и `PROJECT.md`» (файл заводит 4.1; если 4.1 ещё не слита — создать файл, см. Task 3).
- `tests/test_init_scaffold.py` — offline-набор (контент шаблона + примитив копирования).

### Главные риски / решения (читать до кода)

> **Делегирование.** Шеф делегирует решения этой истории по принципу [[feedback-decide-and-apply]] («реши сам, главное чтобы работало надёжно»). Уточняющие (не блокирующие) вопросы — в конце файла.

1. **Скоуп 4.2 = статический шаблон + примитив копирования в `scripts/init/scaffold.py` (решение D1).** Параллель с 4.1 (артефакт + примитив-потребитель; 4.3 оркеструет). Поведение «копировать шаблон / не затереть `PROJECT.md` / fail-loud при отсутствии шаблона» (AC #2/#5/#6) — нетривиально и заслуживает **протестированного дома** уже сейчас, чтобы 4.3 остался тонким оркестратором (имя → copy → симлинки → `.env` → `uv sync` → БД → git + полный откат). Модуль живёт в `scripts/init/` (рядом с `init_project.py` и `symlinks.py` 4.1), а не в `utils/` (там — примитивы пути данных). Зависит **только** от stdlib (`shutil`, `os`, `pathlib`, `logging`); **НЕ** импортирует `paths.py`/`database_manager`/`duckdb`/`metrica_client`/`symlinks.py` — работает с **инъектируемыми** `storage_root`/`template_root` (резолюцию корня хранилища при init делает 4.3 до вызова 4.2). `from __future__ import annotations` первой строкой; русский модульный docstring (роль + границы); идентификаторы английские, docstrings русские; type hints везде (mypy strict). Имя теста — `tests/test_init_scaffold.py` (зеркало модуля; в architecture.md тест 4.2 не назван — выбрано по конвенции `test_init_*`).

2. **Шаблон МИНИМАЛЬНЫЙ и только-про-игру; `.claude/` в шаблон НЕ входит (AC #4; решение D2).** directaiq-шаблон перегружен: `.claude/` с маркетинговыми скилами (`marketing-intelligence`, `attribution`, `landing-pages`, Direct/Wordstat/Prophet/AppMetrica CLI), `config/client_context.md`, папки `data/todo`/`data/uploads`. Для gdau **инфраструктура агента (`.claude/`, `scripts`, конфиги) приходит СИМЛИНКОМ из dev-репо** (контракт 4.1), а НЕ копируется шаблоном. Поэтому gdau-шаблон содержит **ровно 4 файла** (`.env.example`, `.gitignore`, `CLAUDE.md`, `PROJECT.md`) — никаких `.claude/`, никаких Direct/marketing-скилов, никаких directaiq-конфигов. Это прямо закрывает AC #4 («шаблон урезан под геймдев»). **Тест-страж (AC #4):** в `templates/external_storage/` нет `.claude/`, нет файлов с маркером directaiq/Direct/Wordstat/Prophet/AppMetrica/marketing (см. Task 5).

3. **`.gitignore` шаблона игнорит секреты + данные + Python/uv-артефакты + симлинкуемую инфру (AC #1; решение D3).** Минимум по AC #1: `.env`, `data/`, `.writer.lock`. Сверх того (project-context инварианты + 4.1 Project Structure Notes:241 «симлинки в хранилище игнорируются `.gitignore` шаблона»):
   - `*.duckdb`/`*.duckdb.wal`/`*.parquet` (бинарь БД/сырья);
   - **Python/uv-артефакты** — `.venv/`/`venv/`/`__pycache__/`/`*.py[cod]`/`.mypy_cache/`/`.pytest_cache/`/`.ruff_cache/`: 4.3 делает `uv sync` **в хранилище** → там появится локальный `.venv/` (и кэши при прогоне) → без игнора `git` хранилища их подхватит. Зеркало Python-секции dev-репо `.gitignore`;
   - **симлинкуемые пути инфры** (`scripts`, `development-docs`, `yandex-docs`, `pyproject.toml`, `.mcp.json`, `.claude`) — чтобы `git` хранилища игры (свой `git init` в 4.3) не коммитил инфра-симлинки на dev-репо (они — не данные игры; относительные ссылки коммитить в data-репо бессмысленно и мешает разделению двух-репо).

   Сохранить `!.env.example` (образец коммитится). Комментарий симлинк-секции ссылается на `templates/paths-to-symlink.csv` (как directaiq держит синхронность). **Соответствие dev-репо `.gitignore`:** dev-репо игнорит Python-секцию + `.env`/`data/`/`*.writer.lock`/`*.duckdb`/`*.parquet` — шаблонный `.gitignore` это **зеркалит для хранилища**, плюс добавляет симлинк-пути (их в dev-репо нет — там это реальные каталоги).

4. **`CLAUDE.md` хранилища написан ЗАНОВО под gdau, НЕ вендоринг directaiq (AC #4; решение D4).** directaiq `CLAUDE.md` (203 строки) — про рекламный аккаунт: `direct_cli`/`wordstat_cli`/`prophet_cli`/`appmetrica_cli`, `client_context.md`, папки `todo`/`uploads`, атрибуция Директа. Ничего из этого к gdau не применимо. Пишем **свой короткий** `CLAUDE.md` рабочего пространства игры: (а) ты работаешь в per-game хранилище конкретной игры — контекст игры в `PROJECT.md`; (б) данные (`data/`, `gdau.duckdb`, `.env`) живут здесь; инструмент юнита (`scripts/`, каталог, справочники) подключён **симлинками** из dev-репо — правки инструмента не здесь, а в dev-репо; (в) **приём данных** — CLI `gdau-logs update --date1 --date2 --source` (канал действий/записи); **чтение/анализ** — MCP `duckdb_query` (read-only, канал чтения; view'ы `visits`/`hits`); (г) креды Метрики — в `.env` (не коммитятся); (д) принцип «делай только то, что просят; сначала существующие CLI/MCP, не плодить скрипты» (сохраняем дух directaiq-правила, без маркетинга). Без жаргона, человеческим языком.

5. **`PROJECT.md` — шаблон-вопросник для владельца (FR-21; AC #2/#3; решение D5).** Файл-болванка с подсказками-плейсхолдерами, чтобы владелец дал агенту контекст игры: название игры; платформа(ы) (web/iOS/Android/PC/консоль); счётчик Метрики (для справки, не креды); ключевые пользовательские сценарии/воронки; важные цели/события Метрики и что они значат; особенности атрибуции/трафика; глоссарий терминов игры; что агенту полезно знать при анализе. Плейсхолдеры в форме `<!-- заполни: … -->` / `_(опиши …)_` — чтобы незаполненный файл был очевидно-болванкой. Заголовок поясняет: «заполняет владелец после `gdau-init`; агент читает для контекста». **Не** содержит кредов/секретов (они в `.env`).

6. **Примитив копирования: валидация ДО мутаций (AC #5), сохранение `PROJECT.md` (AC #6), БЕЗ отката (граница 4.3; решение D6).**
   - **Валидация fail-loud ДО любого копирования (AC #5):** `template_root` не каталог ИЛИ не хватает любого из обязательных файлов (`.env.example`/`.gitignore`/`CLAUDE.md`/`PROJECT.md`) → `StorageTemplateError` **до** создания/копирования чего-либо в `storage_root` (чтобы не родить полу-битое хранилище). Проверка — полная, до первой мутации.
   - **Сохранение заполненного `PROJECT.md` (AC #6):** при копировании, если в `storage_root` уже есть `PROJECT.md` (владелец заполнил) → **пропустить** его (не перезаписывать), залогировать INFO; остальные файлы копируются (обновляются из шаблона). Имена-исключения вынесены в константу `PRESERVE_ON_REPEAT = ("PROJECT.md",)`.
   - **AC #6 — это гарантия ПРИМИТИВА, не следствие текущего 4.3-потока (важно для Acceptance Auditor).** В штатном init 4.3 **падает fail-loud, если `../{game}` уже существует** (epics.md:478) → ветка «повторный init поверх заполненного `PROJECT.md`» в текущем прод-потоке 4.3 **не триггерится**. AC #6 защищает примитив **независимо** от этого: на случай будущего refresh/re-sync-потока или resume-после-частичного-сбоя (epics.md:486), где copy может быть вызван на уже существующем хранилище с заполненным владельцем `PROJECT.md`. Это **намеренная защитная гарантия** («не уничтожить описание игры владельца»), а не мёртвый код и не противоречие 4.3 — тестируется на `tmp_path` (Task 4). **Кросс-стори флаг для 4.3:** полный откат хранилища 4.3 (epics.md:482) при resume **не должен** удалять заполненный владельцем `PROJECT.md` (иначе AC #6 примитива обесценится на уровне оркестратора) — зафиксировать при сборке 4.3.
   - **БЕЗ отката (D6, граница 4.1↔4.3):** примитив 4.2 не откатывает частично скопированное — полный откат хранилища (шаблон + симлинки + `.env` + БД) делает 4.3 (epics.md:482/488), не дублировать здесь. 4.2 только: валидировать (fail-loud до мутаций) + копировать + сохранить `PROJECT.md`. Сырой `OSError` от `shutil` оборачивать в `StorageTemplateError` с путём (никогда «голый» наружу — паттерн ревью 2.1), но именно **ловлю/очистку диск-полон/прав** при копировании оркеструет 4.3 (epics.md:488).

7. **Тесты: контент статического шаблона + примитив копирования на `tmp_path`; live НЕ нужен (решение D7).** CI ubuntu + windows. Стратегия:
   - **контент шаблона (всегда идут):** все 4 файла существуют в `templates/external_storage/`; `.env.example` содержит обе переменные `YANDEX_METRICA_TOKEN`/`YANDEX_METRICA_COUNTER_ID` (и НЕ содержит значений-секретов после `=`); `.gitignore` игнорит `.env`, `data/`, `.writer.lock` + симлинк-пути и хранит `!.env.example`; `PROJECT.md` непустой и содержит маркеры-плейсхолдеры; **AC #4-страж**: в каталоге нет `.claude/` и нет файлов с маркерами directaiq/marketing (`direct_cli`/`wordstat`/`prophet`/`appmetrica`/`client_context`);
   - **примитив копирования на `tmp_path`:** на `tmp_path` собрать минимальный валидный `template_root` (4 файла) → `copy_storage_template(storage_root=…, template_root=…)` → все 4 файла появились в `storage_root`, содержимое идентично; **отсутствие/битость шаблона** (нет каталога / не хватает файла) → `pytest.raises(StorageTemplateError)`, и в `storage_root` **ничего не создано** (валидация до мутаций, AC #5); **повторный init** — подложить заполненный `PROJECT.md` в `storage_root` → после `copy_storage_template` его содержимое **не изменилось** (AC #6), остальные файлы обновлены; копирование в **несуществующий** `storage_root` создаёт его; копирование на **реальном** `DEFAULT_TEMPLATE_ROOT` (smoke) проходит;
   - **анти-зависимость (`ast`):** `scripts/init/scaffold.py` не импортирует `pandas`/`polars`/`numpy`/`pyarrow`/`duckdb`/`scripts.utils.paths`/`database_manager`/`metrica_client`/directaiq-инфру — только stdlib (паттерн `tests/test_parquet_store.py`);
   - **live-тест НЕ нужен** (нет внешнего API; правило opt-in live — только для Logs API, [[realapi-smoke-tests]]) — зафиксировать в Dev Agent Record, как 2.1–2.6 и 4.1.

## Acceptance Criteria

1. **Given** `templates/external_storage/`, **When** смотрим шаблон, **Then** он содержит `.env.example` (`YANDEX_METRICA_TOKEN=`/`YANDEX_METRICA_COUNTER_ID=`), `.gitignore` (игнор `.env`, `data/`, `.writer.lock`), `CLAUDE.md`, `PROJECT.md`.
2. **Given** init создаёт хранилище (4.3), **When** копируется шаблон, **Then** `PROJECT.md` создаётся из шаблона для заполнения владельцем (FR-21).
3. **Given** рабочую сессию (UJ-3), **When** агент работает в хранилище, **Then** `PROJECT.md` ему доступен.
4. **Given** шаблон урезан под геймдев, **When** смотрим содержимое, **Then** нет Direct/marketing-скилов и конфигов directaiq.
5. **Given** `templates/external_storage/` отсутствует/битый, **When** init обращается к шаблону, **Then** fail-loud ДО любых мутаций (не создаётся пустое битое хранилище). _[edge-case: отсутствующий шаблон]_
6. **Given** `PROJECT.md` уже заполнен владельцем (повторный init), **When** копируется шаблон, **Then** существующий `PROJECT.md` не затирается. _[edge-case: затирание описания игры]_

```python
    # --- валидация ДО любых мутаций (AC #5) ---
    if not root.is_dir():
        raise StorageTemplateError(f"Шаблон хранилища не найден/не каталог: {root}")
    for name in REQUIRED_TEMPLATE_FILES:
        if not (root / name).is_file():
            raise StorageTemplateError(f"В шаблоне отсутствует обязательный файл: {name}")
    # --- мутации ---
    try:
        storage_root.mkdir(parents=True, exist_ok=True)
        copied: list[Path] = []
        for entry in sorted(root.iterdir()):
            dest = storage_root / entry.name
            if entry.name in PRESERVE_ON_REPEAT and dest.exists():
                logger.info("Сохраняю существующий %s (повторный init)", entry.name)
                continue
            if entry.is_dir():
                shutil.copytree(entry, dest, dirs_exist_ok=True)
            else:
                shutil.copy2(entry, dest)
            copied.append(dest)
    except OSError as exc:  # диск полон/права — оборачиваем; откат хранилища у 4.3 (D6)
        raise StorageTemplateError(f"Сбой копирования шаблона в {storage_root}: {exc}") from exc
    return copied
```

### Latest Tech Information

- **`shutil.copy2(src, dst)`** — копирует файл с метаданными (mtime/права); подходит для шаблонных файлов. **`shutil.copytree(src, dst, dirs_exist_ok=True)`** (Python ≥3.8) — рекурсивное копирование каталога без падения, если назначение существует; нужно только если шаблон обзаведётся подкаталогами (сейчас плоский — 4 файла). Оба бросают `OSError` (диск полон/права) — оборачиваем в `StorageTemplateError`.
- **`Path.mkdir(parents=True, exist_ok=True)`** — создать корень хранилища (и родителей) идемпотентно; на повторном init `exist_ok=True` не падает.
- **`Path.iterdir()` + `sorted(...)`** — детерминированный обход top-level записей шаблона (порядок ФС не гарантирован; `sorted` фиксирует для воспроизводимости `copied` и тестов).
- **`.env.example` vs `.env`:** dev-репо `.gitignore` уже игнорит `.env`/`.env.*` с `!.env.example` (строки 12–14) — шаблонный `.env.example` коммитится, любой реальный `.env` — нет. Шаблон хранилища несёт ИМЕННО `.env.example`; реальный `.env` появляется в хранилище игры (генерит 4.3) и игнорится `.gitignore` хранилища.
- **Web-ресёрч не требуется:** stdlib стабилен и зафиксирован локом; внешнего сетевого контракта в истории нет (live-smoke неприменим, как 2.1–2.6 / 4.1).
